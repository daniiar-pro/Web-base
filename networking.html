Chapter 1

The Big Picture

Before diving into layers and protocols, let's understand what networking actually is and why it matters.

What Is a Computer Network?

A computer network is simply two or more devices connected together so they can share data. That's it. Whether it's two laptops connected by a cable, or billions of devices connected across the globe (the Internet) — the fundamental idea is the same: get data from point A to point B reliably.

📬 Real-World Analogy

Think of networking like the postal system. You write a letter (data), put it in an envelope (packet), write the address (IP), drop it in a mailbox (router), and the postal service figures out the best route to deliver it. The recipient opens the envelope and reads your letter. Networking works almost the same way — just billions of times faster.

Why Do We Need Models?

Networking is incredibly complex. To manage this complexity, engineers break the communication process into layers. Each layer handles one specific job and talks only to the layers directly above and below it. This is called the layered architecture, and it's the foundation of the OSI and TCP/IP models.

Benefits: Each layer can be developed independently. You can swap WiFi for Ethernet without changing your web browser. A new app can run on the same network infrastructure. This separation of concerns is what makes the Internet so flexible and powerful.

The OSI Model — 7 Layers

The Open Systems Interconnection model is the universal framework for understanding network communication. Click each layer to learn more.

🏗️ Think of it like building a house

Layer 1 is the foundation (physical cables). Layer 2 is the framing (local addressing). Layer 3 is the plumbing/electrical (routing). Layer 4 is the doors/hallways (reliable delivery). Layers 5–7 are the furniture, paint, and decor (the stuff you actually interact with). Each layer depends on the ones below it.

Layer 7 — Application Layer

This is the only layer that directly interacts with the user. It provides network services to applications. When you type a URL in your browser, the browser uses HTTP (Application Layer protocol) to request that web page.

Key Protocols: HTTP/HTTPS (web), SMTP/IMAP/POP3 (email), FTP/SFTP (file transfer), DNS (domain name resolution), SSH (secure remote access), SNMP (network management).

Data Unit: Data / Message

6
Presentation
Translates, encrypts, and compresses data between formats
SSL/TLSJPEGASCIIMPEG

Layer 6 — Presentation Layer

The translator of the network. It ensures that data sent by one system can be understood by another. It handles three main functions: Translation (converting character encoding like ASCII to EBCDIC), Encryption/Decryption (SSL/TLS encryption happens here), and Compression (reducing data size for faster transmission).

Real example: When you visit an HTTPS site, TLS encryption (Presentation Layer) scrambles your data so no one in between can read it.

Data Unit: Data

5
Session
Opens, manages, and closes communication sessions
NetBIOSRPCPPTP

Layer 5 — Session Layer

The session layer is the "conversation manager." It establishes, maintains, and terminates sessions between two devices. Think of it like a phone call: someone dials (opens session), you talk (maintain session), and someone hangs up (closes session).

It also handles checkpointing: if you're transferring a 500 MB file and the connection drops at 400 MB, the session layer can resume from 400 MB instead of starting over.

Modes: Simplex (one-way), Half-duplex (take turns), Full-duplex (both ways simultaneously)

Data Unit: Data

4
Transport
End-to-end delivery, reliability, flow control, error recovery
TCPUDPQUIC

Layer 4 — Transport Layer

This is the "reliability manager." It takes data from the upper layers, breaks it into segments, and ensures they arrive correctly and in order at the destination. The two stars of this layer are TCP and UDP.

TCP (Transmission Control Protocol): Reliable, ordered, error-checked. Uses a 3-way handshake (SYN → SYN-ACK → ACK) to establish connection. Used for web, email, file transfer.

UDP (User Datagram Protocol): Fast, no handshake, no guarantee. Used for video streaming, gaming, DNS lookups, VoIP.

Port Numbers: This layer uses port numbers (0–65535) to direct data to the right application. HTTP = 80, HTTPS = 443, SSH = 22, DNS = 53.

Data Unit: Segment (TCP) / Datagram (UDP)

3
Network
Logical addressing (IP) and routing between different networks
IPICMPARPOSPFBGP

Layer 3 — Network Layer

The "GPS navigator" of networking. This layer handles logical addressing (IP addresses) and routing — figuring out the best path for data to travel from source to destination across multiple networks.

IP Addressing: Every device on the Internet has an IP address. IPv4 (e.g., 192.168.1.1) uses 32 bits, giving ~4.3 billion addresses. IPv6 (e.g., 2001:db8::1) uses 128 bits for an astronomical number of addresses.

Routing Protocols: OSPF and RIP (within an organization), BGP (between organizations / ISPs — this is what literally holds the Internet together).

Key Devices: Routers operate at this layer.

Data Unit: Packet

2
Data Link

EthernetWiFiPPPARP

Layer 2 — Data Link Layer

This layer handles communication between devices on the same local network. It uses MAC addresses (Media Access Control) — a unique 48-bit hardware address burned into every network interface card (e.g., AA:BB:CC:DD:EE:FF).

Two Sub-layers:

• LLC (Logical Link Control): Identifies which Network layer protocol (IPv4, IPv6, etc.) the frame carries.

• MAC (Media Access Control): Controls how devices gain access to the medium and transmit data.

Error Detection: Uses CRC (Cyclic Redundancy Check) in the frame trailer to detect corrupted data.

Key Devices: Switches, Bridges operate at this layer.

Data Unit: Frame

1
Physical
Raw bit transmission — cables, signals, voltages, radio waves
EthernetUSBFiberWiFi Radio

Layer 1 — Physical Layer

The absolute bottom layer. It deals with the raw transmission of bits (1s and 0s) over a physical medium. It doesn't know or care what the bits mean — it just moves them.

Transmission Media:

• Copper cables (Ethernet): Use electrical voltage changes. Cat5e, Cat6, Cat6a cables.

• Fiber optic: Use light pulses. Much faster, longer range, immune to electromagnetic interference.

• Wireless (WiFi): Use radio frequency waves. Standards: 802.11a/b/g/n/ac/ax (WiFi 6).

Concepts: Bit rate, bandwidth, latency, signal encoding, modulation.

Key Devices: Hubs, Repeaters, Cables, Antennas.

Data Unit: Bit

Chapter 4

Physical Layer — The Wires

Everything in networking ultimately comes down to moving bits through a physical medium.

🔌

Copper (Ethernet)

Uses electrical signals over twisted-pair cables. Cat5e supports 1 Gbps, Cat6a supports 10 Gbps. Max distance ~100m. Susceptible to electromagnetic interference (EMI).

💡

Fiber Optic

Uses pulses of light through glass/plastic fibers. Single-mode fiber can span 100+ km. Multi-mode is shorter range but cheaper. Immune to EMI. Backbone of the Internet.

📡

Wireless (Radio)

WiFi uses 2.4 GHz and 5 GHz frequencies. WiFi 6 (802.11ax) supports up to 9.6 Gbps theoretical. Signal degrades with distance, walls, and interference.

🛰️

Satellite

Communication via satellites in orbit. GEO satellites have ~600ms latency. LEO constellations (like Starlink) reduce latency to ~20-40ms. Used in remote areas.

Key Physical Layer Concepts

Bandwidth: The maximum rate of data transfer — measured in bits per second (bps). Your "100 Mbps" Internet connection means 100 million bits per second maximum throughput.

Latency: The time it takes for a bit to travel from source to destination. Measured in milliseconds (ms). Light in fiber travels at ~200,000 km/s (about 2/3 the speed of light in vacuum).

Throughput: The actual data rate achieved in practice — always less than bandwidth due to overhead, congestion, and errors.

Chapter 5

Data Link Layer — Local Delivery

How data gets from one device to another on the same local network.

MAC Addresses

Every network interface has a unique MAC address — a 48-bit address written as six pairs of hex digits: AA:BB:CC:DD:EE:FF

The first 3 bytes identify the manufacturer (OUI), and the last 3 bytes are unique to the device. MAC addresses work only on the local network — they don't survive past a router.

FCS (Frame Check Sequence): A CRC checksum to detect errors. If the checksum doesn't match, the frame is discarded.

Switches — The Traffic Directors of LANs

A switch is the key device at Layer 2. It maintains a MAC address table that maps each MAC address to a physical port. When a frame arrives, the switch looks up the destination MAC and forwards the frame only to the correct port — instead of flooding it everywhere (like a hub would).

How the table is built: When a device sends a frame, the switch records the source MAC address and the port it arrived on. Over time, it learns where every device is.

ARP — Bridging Layer 2 and Layer 3

ARP (Address Resolution Protocol) is the glue between IP addresses and MAC addresses. When your computer knows the IP of a device on the same network but not its MAC address, it broadcasts an ARP request: "Who has 192.168.1.5? Tell me your MAC!" The device with that IP replies with its MAC address. Your computer caches this mapping in its ARP table.

Chapter 6

Network Layer — IP & Routing

The layer responsible for getting data across different networks — from your home to a server on the other side of the world.

IP Addresses Explained

An IP address is a logical address assigned to a device. Unlike MAC addresses (burned into hardware), IP addresses can change and are assigned by network configuration (DHCP or manually).

IPv4

32-bit address. Written as 4 octets: 192.168.1.100

~4.3 billion addresses total. We ran out! That's why NAT and IPv6 exist.

IPv6

128-bit address: 2001:0db8::8a2e:0370:7334

340 undecillion addresses. Enough to give every atom on Earth an IP address (not really, but close).

Subnetting — Dividing Networks

Subnetting divides a large network into smaller, more manageable pieces. A subnet mask tells the device which part of the IP is the network portion and which is the host portion.

📞 vs 📢 Analogy

TCP is like a phone call: You dial, wait for the other person to pick up, talk back and forth, confirm you understood each other, and hang up properly. UDP is like shouting into a megaphone: You blast your message out. Maybe they hear it, maybe they don't. But it's fast!

Chapter 8

Upper Layers — Where Users Live

Session, Presentation, and Application layers — where data becomes useful.

🌐

HTTP / HTTPS

The foundation of the web. HTTP sends requests (GET, POST, PUT, DELETE) and receives responses. HTTPS adds TLS encryption. Uses port 80 (HTTP) and 443 (HTTPS).

📧

SMTP / IMAP / POP3

Email protocols. SMTP sends mail (port 25/587). IMAP (port 143/993) syncs mail across devices. POP3 (port 110/995) downloads and typically deletes from server.

📁

FTP / SFTP

File Transfer Protocol transfers files between systems. FTP is unencrypted (ports 20-21). SFTP runs over SSH (port 22) for secure transfers.

🔑

SSH

Secure Shell provides encrypted remote access to servers. Replaced insecure Telnet. Uses port 22. Supports key-based authentication for passwordless login.

🔒

TLS / SSL

Transport Layer Security encrypts data in transit. Powers HTTPS. Uses certificates, key exchange (Diffie-Hellman), and symmetric encryption (AES). SSL is the deprecated predecessor.

🔄

DHCP

Dynamic Host Configuration Protocol auto-assigns IP addresses. Your router's DHCP server gives each device an IP, subnet mask, gateway, and DNS server when they join the network.

Chapter 9

DNS — The Internet's Phonebook

How "google.com" becomes "142.250.185.78"

Why DNS Exists

Humans remember names. Computers use numbers. DNS (Domain Name System) translates human-friendly domain names into IP addresses that computers can route to. Without DNS, you'd have to memorize IP addresses for every website.

DNS Resolution — Step by Step

① Browser Cache Check

Your browser first checks its own cache. "Have I looked up google.com recently?" If yes, use the cached IP. If no, move on.

② OS Cache / hosts file

The operating system checks its DNS cache and the local hosts file (/etc/hosts on Linux/Mac, C:\Windows\System32\drivers\etc\hosts on Windows).

③ Recursive Resolver (ISP / 8.8.8.8)

The query goes to a recursive DNS resolver (usually your ISP's, or public ones like Google's 8.8.8.8 or Cloudflare's 1.1.1.1). This resolver does the heavy lifting.

④ Root Name Server

The resolver asks a root server: "Where do I find .com domains?" The root server responds with the address of the .com TLD server. There are 13 root server clusters worldwide.

⑤ TLD Name Server (.com)

The resolver asks the .com TLD server: "Where's google.com?" The TLD server responds with the address of Google's authoritative name server.

⑥ Authoritative Name Server

The resolver asks Google's name server: "What's the IP for google.com?" Answer: "142.250.185.78". The resolver caches this answer and returns it to your browser.

⑦ Connection Established!

Your browser now has the IP address and can establish a TCP connection to the server. All of this happens in milliseconds.

Chapter 10

The Full Journey — Visiting a Website

Let's trace every single step that happens when you type "https://www.example.com" and press Enter.

Step 1: URL Parsing

Browser parses the URL: protocol (https), domain (www.example.com), path (/). It determines it needs to make an HTTPS request.

Step 2: DNS Resolution

Browser resolves "www.example.com" to an IP address (e.g., 93.184.216.34) through the DNS hierarchy we covered above.

Step 3: TCP Handshake

Browser initiates a TCP 3-way handshake with the server at 93.184.216.34 on port 443 (HTTPS). SYN → SYN-ACK → ACK. Connection established.

Step 4: TLS Handshake

Since it's HTTPS, a TLS handshake occurs: Client Hello → Server Hello + Certificate → Key Exchange → Finished. Both sides now share a symmetric encryption key.

Step 5: HTTP Request

Browser sends an encrypted HTTP GET request for "/". This data is encrypted (TLS), put in a TCP segment with port numbers, wrapped in an IP packet, framed with MAC addresses, and sent as bits on the wire.

Step 6: Routing Across the Internet

The packet travels from your router → ISP → multiple backbone routers (each one looking at the destination IP and making forwarding decisions) → the server's network. It might cross oceans through undersea fiber cables!

Step 7: Server Processes Request

The server receives the packet, de-encapsulates through all layers, decrypts the TLS, reads the HTTP request, finds the requested resource, and prepares a response.

Step 8: Response Travels Back

The server sends back an HTTP 200 OK response with the HTML content, following the same encapsulation process in reverse. The response is broken into multiple TCP segments if it's large.

Step 9: Browser Renders the Page

Browser receives the HTML, parses it, discovers it needs CSS, JavaScript, images — and makes additional HTTP requests for each. The DOM is built, styles are applied, JavaScript executes, and you see the page!

⏱️ Speed Context

All of these steps — DNS lookup, TCP handshake, TLS handshake, HTTP request/response, and rendering — typically happen in under 1 second. A DNS lookup takes ~20-120ms. A TCP handshake takes one round-trip (~10-100ms depending on distance). The TLS handshake adds another 1-2 round-trips. The rest depends on server speed and page complexity.

Chapter 11

Network Security Essentials

Understanding threats and how networks are protected.

🔥

Firewalls

Filter traffic based on rules (IP, port, protocol). Stateful firewalls track connection states. Next-gen firewalls (NGFW) inspect application-layer content.

🔒

VPN

Virtual Private Network creates an encrypted tunnel over the public Internet. Protocols: IPsec, OpenVPN, WireGuard. Used for remote access and privacy.

🛡️

IDS / IPS

Intrusion Detection/Prevention Systems monitor network traffic for suspicious patterns. IDS alerts; IPS actively blocks threats.

🔐

TLS/SSL Encryption

Encrypts data in transit using asymmetric (key exchange) and symmetric (data encryption) cryptography. Certificates verify server identity.

🎭

Common Attacks

DDoS (overwhelming with traffic), Man-in-the-Middle (intercepting communications), DNS Spoofing (redirecting to malicious sites), ARP Poisoning (local network attacks).

📋

Best Practices

Use HTTPS everywhere. Keep firmware updated. Segment networks. Use strong authentication. Monitor logs. Principle of least privilege.
