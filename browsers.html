02 DNS & Networking: From Domain to IP

2.1 What is DNS?

The Domain Name System (DNS) is the internet's phonebook. Humans use domain names like www.google.com, but computers communicate using IP addresses like 142.250.185.46. DNS translates one to the other.

DNS is a globally distributed, hierarchical database. There is no single computer that knows all mappings — instead, the work is split across millions of servers organized in a tree structure.

2.2 The DNS Resolution Process — Step by Step

2.3 The Detailed DNS Journey

Step 1 — Browser Cache Check

The browser first checks its own internal DNS cache. Chrome, for example, keeps recently resolved domains in memory. If the domain was visited recently and the cached record hasn't expired (based on its TTL — Time To Live), the browser skips all network queries and uses the cached IP immediately.

Step 2 — OS Resolver (Stub Resolver)

If the browser cache misses, it calls the operating system's resolver. On Linux/macOS, this goes through getaddrinfo(). The OS checks its own cache and also the /etc/hosts file (a local mapping file where you can manually define domain→IP pairs, e.g. 127.0.0.1 localhost).

Step 3 — Router Cache

If the OS doesn't have the answer, the query goes to your home/office router, which maintains its own small DNS cache for all devices on the network.

Step 4 — Recursive Resolver (ISP or Public)

This is the workhorse. Your ISP runs recursive resolvers, or you might use public ones like Google's 8.8.8.8 or Cloudflare's 1.1.1.1. The recursive resolver is the one that does the heavy lifting — it talks to multiple servers on your behalf.

Step 5 — Root Nameservers

If the recursive resolver doesn't have the answer cached, it starts at the very top of the DNS hierarchy: the root nameservers. There are 13 root server clusters (named a.root-servers.net through m.root-servers.net), operated by organizations like ICANN, Verisign, and NASA. Each cluster runs hundreds of actual machines globally using anycast routing. The root doesn't know the final IP — it only knows which servers handle top-level domains (TLDs).

Query: "Where can I find .com?"
Answer: "Try the .com TLD servers at 192.5.6.30."

Step 6 — TLD Nameservers

The TLD servers (managed by registries like Verisign for .com) don't know the final IP either. They know which authoritative nameservers are responsible for each registered domain.

Query: "Where can I find example.com?"
Answer: "The authoritative nameservers for example.com are ns1.example.com and ns2.example.com."

Step 7 — Authoritative Nameserver

This is the server that actually has the definitive DNS records for the domain. It's usually run by the domain owner's hosting provider or DNS service (like Cloudflare, Route53, etc.).

Query: "What is the A record for example.com?"
Answer: "93.184.216.34, TTL: 3600 seconds."

Step 8 — Response Propagates Back

The IP address travels back through the chain: Authoritative → Recursive Resolver (which caches it for the TTL duration) → OS → Browser. Each layer caches the result to speed up future queries. The browser now has the IP and can initiate a connection.

DNS over HTTPS (DoH) and DNS over TLS (DoT) Traditional DNS queries are sent in plaintext UDP — anyone on the network can see what domains you're resolving. Modern browsers increasingly use DNS over HTTPS (DoH), which encrypts DNS queries inside HTTPS requests, typically to resolvers like Cloudflare (1.1.1.1) or Google (8.8.8.8). This prevents ISPs and network operators from snooping on your browsing. Firefox, Chrome, and Edge all support DoH.

2.4 DNS Performance & Optimization

DNS lookups are blocking — nothing else can happen until the browser knows the IP. Several techniques minimize this delay:

  • DNS Prefetching: Browsers scan HTML for links and speculatively resolve domains before the user clicks. You can hint this with <link rel="dns-prefetch" href="//cdn.example.com">.
  • Preconnect: Goes further than DNS — <link rel="preconnect" href="https://cdn.example.com"> does DNS + TCP + TLS handshake ahead of time.
  • Low TTLs vs. High TTLs: Short TTLs (e.g., 60s) enable fast failover but increase DNS queries. Long TTLs (e.g., 86400s = 1 day) reduce lookups but make changes slow to propagate.
  • Anycast: Major DNS providers use anycast routing, where the same IP address is advertised from multiple geographic locations. Your query automatically reaches the nearest server.

03 TCP, TLS & HTTP: The Connection

3.1 TCP Three-Way Handshake

Once we have an IP address, the browser needs to establish a reliable connection. TCP (Transmission Control Protocol) ensures data arrives in order and without corruption, using a three-step handshake:

This handshake costs one full round-trip (RTT). On a connection with 40ms latency, that's 40ms before any data is exchanged.

3.2 TLS Handshake (HTTPS)

For HTTPS connections, after TCP is established, a TLS (Transport Layer Security) handshake adds encryption. With TLS 1.3 (the modern standard), this takes just one additional round-trip:

  1. Client Hello: The browser sends supported cipher suites and a key share.
  2. Server Hello: The server picks a cipher, sends its certificate and key share.
  3. Verification: The browser verifies the server's certificate against trusted Certificate Authorities (CAs) stored in the OS/browser. If valid, both sides derive the encryption keys.
  4. Encrypted data flows.
Connection reuse matters! HTTP/1.1 introduced keep-alive to reuse TCP connections. HTTP/2 multiplexes many requests over a single connection. HTTP/3 uses QUIC (built on UDP), which combines the TCP and TLS handshakes into a single round-trip — and even supports 0-RTT for repeat connections.

3.3 HTTP Request & Response

Once the secure connection is established, the browser sends an HTTP request:

GET / HTTP/2
Host: example.com
User-Agent: Chrome/120
Accept: text/html,application/xhtml+xml
Accept-Encoding: gzip, br
Accept-Language: en-US
Cookie: session=abc123

The server processes this (through its web server, application code, database, etc.) and returns a response:

HTTP/2 200 OK
Content-Type: text/html; charset=UTF-8
Content-Encoding: br
Cache-Control: max-age=3600
Content-Length: 42381

<!DOCTYPE html>
<html>...the HTML document...</html>

This response is what the browser's rendering engine will process. The HTML is sent as a stream of bytes — the browser doesn't wait for the entire document to arrive before starting to parse it.

04 Browser Architecture

A modern browser like Chrome is not one program — it's multiple processes, each sandboxed for security and stability. If one tab crashes, others keep running.

The Critical Insight: The Main Thread Bottleneck Inside each renderer process, there is one main thread that handles HTML parsing, CSS calculation, JavaScript execution, layout, and paint. If your JavaScript runs a heavy computation, it blocks everything — the page freezes because layout and paint can't happen. This is why understanding the event loop (covered later) is so important.

5.3 Parser-Blocking Resources

While parsing HTML, the browser encounters external resources that can block the parser:

<script> (default)

BLOCKS Parsing stops. The browser must download and execute the script before continuing, because the script might modify the DOM with document.write().

<script async>

NON-BLOCKING Downloads in parallel, executes as soon as it's ready (may execute out of order). Parsing continues during download.

<script defer>

NON-BLOCKING Downloads in parallel, executes after HTML parsing is complete, in order. Best for scripts that need the full DOM.

<link rel="stylesheet">

RENDER-BLOCKING Doesn't block parsing, but blocks rendering. The browser won't paint until all CSS is loaded and the CSSOM is built — otherwise you'd see a flash of unstyled content (FOUC).

The Preload Scanner While the main parser is blocked by a script, a secondary "preload scanner" continues scanning the rest of the HTML to discover resources (images, stylesheets, scripts) and start downloading them in parallel. This is one of the most important browser optimizations.

6.2 Style Computation (Cascade & Inheritance)

The browser must figure out the final computed style for every single DOM element. This involves resolving the CSS cascade:

  1. Origin: User-agent styles (browser defaults) → Author styles → User styles → !important
  2. Specificity: Inline > ID > Class > Element (the famous specificity calculation)
  3. Order: Later rules win over earlier ones at equal specificity
  4. Inheritance: Some properties (like color, font-family) cascade down from parent to child. Others (like border, margin) don't.

The result is a "computed style" for every DOM node — every element gets hundreds of resolved CSS properties, even if you only wrote a few rules.

Why CSS blocks rendering The browser refuses to render anything until the CSSOM is complete. Why? Because without knowing all the styles, the browser can't lay anything out. A late-loading stylesheet could completely change the page layout, causing an ugly flash. So CSS is render-blocking by design.

07 The Render Pipeline: From Trees to Pixels

Once both the DOM and CSSOM are ready, the browser's rendering engine kicks into gear. This is the most complex part:

7.1 Render Tree Construction

The browser walks the DOM tree and for each visible node, looks up the computed style from the CSSOM. Nodes that aren't visible (like <head>, <meta>, or elements with display: none) are excluded. visibility: hidden elements ARE included (they take space, they're just invisible).

7.2 Layout (Reflow)

Starting from the root, the engine calculates the exact position and size of every element in the render tree. This uses the CSS box model: content → padding → border → margin. Layout is recursive — a parent's size depends on its children, and children may depend on their parent (percentage widths, flex/grid, etc.).

7.3 Paint

The layout tree is converted into a series of paint instructions — essentially a list of drawing commands: "draw a blue rectangle at (10, 20) with size 200×50", "draw text 'Hello' at (15, 35) in 16px Arial". These are organized into layers.

7.4 Layering & Compositing

The browser promotes certain elements to their own compositing layers — elements with will-change, transform, position: fixed, opacity animations, or <video>/<canvas> tags. Each layer is rasterized (turned into pixels) independently by raster threads, then the compositor thread combines them into the final image and sends it to the GPU for display.

Why this matters for performance If you animate using left/top, the browser must recalculate Layout → Paint → Composite on every frame. If you animate using transform: translateX(), only the Composite step runs — it's handled entirely on the compositor thread without touching the main thread, giving you smooth 60fps animations.

08 The JavaScript Engine (V8 Deep Dive)

The JS engine (V8 in Chrome/Edge/Node.js, SpiderMonkey in Firefox, JavaScriptCore in Safari) is what gives web pages their behavior. Here's how V8 works:

8.1 Key Concepts

  • Parsing: Source code → tokens → Abstract Syntax Tree (AST). V8 uses "lazy parsing" — it only fully parses functions when they're about to be called.
  • Ignition (Interpreter): Walks the AST and produces bytecode — a compact, platform-independent representation. Bytecode executes immediately via the interpreter, providing fast startup.
  • TurboFan (JIT Compiler): When Ignition notices a function is called frequently ("hot"), TurboFan compiles it to highly optimized machine code using the type information collected during interpretation. This is Just-In-Time compilation.
  • Deoptimization: If the assumptions TurboFan made prove wrong (e.g., a variable that was always a number suddenly gets a string), the engine "deoptimizes" — throwing away the machine code and falling back to interpreted bytecode.
  • Hidden Classes: V8 creates internal "shapes" (hidden classes) for objects. Objects with the same property names added in the same order share a hidden class, enabling fast property access. Changing object shapes at runtime hurts performance.
  • Garbage Collection: V8 uses a generational garbage collector. The "minor GC" (Scavenger) handles short-lived objects in the young generation. The "major GC" (Mark-Compact) handles the old generation. Both use concurrent and parallel techniques to minimize pauses.

09 The Event Loop: The Heart of the Browser

JavaScript is single-threaded — it has one call stack and can do one thing at a time. Yet browsers handle thousands of concurrent operations (network requests, timers, user events, animations). How? The event loop.

9.1 Practical Example

console.log('1');                          // Sync — runs immediately

setTimeout(() => console.log('2'), 0);     // Macrotask — queued

Promise.resolve().then(() => {
  console.log('3');                        // Microtask — queued
});

console.log('4');                          // Sync — runs immediately

// Output: 1, 4, 3, 2
// Why? Sync first (1, 4), then microtasks (3), then macrotasks (2)

9.2 requestAnimationFrame

requestAnimationFrame (rAF) is special — it's tied to the rendering step. Its callbacks run right before the browser does Style/Layout/Paint, making it ideal for animations. It fires at the display's refresh rate (typically 60fps = every 16.6ms).

Common mistake: infinite microtask loops If a microtask schedules another microtask endlessly, the event loop never reaches Step 3 (rendering) or Step 4 (macrotasks). The page freezes completely because the microtask queue never empties. while(true) in a Promise chain is a guaranteed freeze.

10 Rendering Strategies: SSR, CSR, SSG & More

How you deliver HTML to the browser fundamentally changes what the browser does and when. Here's the complete comparison:

10.1 Client-Side Rendering (CSR)

The server sends a nearly empty HTML shell. JavaScript downloads, parses, executes, and builds the entire page in the browser.

<!-- What the server sends -->
<html>
  <body>
    <div id="root"></div>  <!-- Empty! -->
    <script src="app.js"></script>  <!-- ~500KB bundle -->
  </body>
</html>

10.2 Server-Side Rendering (SSR)

The server runs your application code, generates full HTML, and sends it to the browser. The browser can display content immediately. Then JavaScript "hydrates" the page — attaching event listeners to the server-rendered HTML to make it interactive.

10.3 Static Site Generation (SSG)

HTML is generated at build time, not request time. Every page is pre-built into a static HTML file. Extremely fast because pages can be served from a CDN with zero server computation.

10.4 Incremental Static Regeneration (ISR)

A hybrid: pages are statically generated, but can be re-generated on the server after a set time interval. Stale-while-revalidate: serve the cached page, rebuild in the background.

10.5 Streaming SSR

Instead of waiting for the entire HTML to be generated on the server, chunks of HTML are streamed to the browser as they become ready. The browser can start parsing and rendering the top of the page while the server is still generating the bottom. React 18's renderToPipeableStream enables this.

10.6 How Each Strategy Affects the Browser

Strategy First Paint Time to Interactive SEO Server Load
CSR Slow (wait for JS) Slow (wait for JS) Poor (empty HTML) Low (static files)
SSR Fast (full HTML) Medium (hydration delay) Great (full HTML) High (per-request rendering)
SSG Fastest (CDN) Fast Great None (static files)
ISR Fast (cached HTML) Fast Great Low (revalidation only)
Streaming SSR Very fast (partial HTML) Progressive Great Medium
React Server Components (RSC) — the latest evolution RSCs stay on the server and never ship to the client's JS bundle. They can directly access databases and file systems. Only interactive components ("use client") are sent to the browser. This reduces the JavaScript bundle dramatically and eliminates hydration for server-only components.

11 Performance: Core Web Vitals & Optimization

11.1 Core Web Vitals

LCP — Largest Contentful Paint

Time until the largest visible content element (image, heading, etc.) is rendered. Target: under 2.5 seconds. Affected by: server response time, CSS/JS blocking, resource load times.

INP — Interaction to Next Paint

Time from user interaction (click, tap, key) to the next visual update. Target: under 200ms. Affected by: JavaScript execution time, main thread blocking, rendering time.

CLS — Cumulative Layout Shift

Measures visual stability — how much content unexpectedly shifts while loading. Target: under 0.1. Caused by: images without dimensions, dynamic content injection, web fonts loading.

FCP — First Contentful Paint

Time until the first text or image is rendered. Affected by: render-blocking resources, server response time, DOM size.

11.2 The Critical Rendering Path

The critical rendering path is the minimum set of steps the browser must complete before it can display the first pixel. Optimizing it means:

  • Minimize critical resources: Only include CSS/JS needed for above-the-fold content. Defer the rest.
  • Minimize critical bytes: Compress (gzip/Brotli), minify, and tree-shake.
  • Minimize critical path length: Reduce the number of sequential round-trips required. Inline critical CSS, use preload/preconnect.

11.3 Common Optimizations

  • Code splitting: Don't send 2MB of JavaScript if the user only needs 200KB for the current page. Dynamic import() loads code on demand.
  • Image optimization: Use modern formats (WebP, AVIF), lazy-load offscreen images (loading="lazy"), provide width/height to prevent CLS.
  • Font optimization: font-display: swap prevents invisible text during load. Preload critical fonts. Subset fonts to include only needed characters.
  • Caching: HTTP cache headers (Cache-Control), Service Workers for offline/custom caching, CDNs for global distribution.
  • Bundle analysis: Use tools like Webpack Bundle Analyzer, Lighthouse, or Chrome DevTools Performance tab to find bottlenecks.

12 Browser Security Model

12.1 Same-Origin Policy

The foundation of web security. An "origin" is defined as scheme + host + port (e.g., https://example.com:443). Scripts from one origin cannot read data from a different origin. This prevents a malicious page from reading your bank's cookies or DOM.

12.2 CORS (Cross-Origin Resource Sharing)

A controlled mechanism to relax the same-origin policy. The server includes headers like Access-Control-Allow-Origin: https://myapp.com to grant specific origins access to its resources. For "complex" requests (PUT, DELETE, custom headers), the browser sends a preflight OPTIONS request first to check permission.

12.3 Key Security Features

  • CSP (Content Security Policy): HTTP header that restricts where scripts, styles, and other resources can be loaded from. Prevents XSS by blocking inline scripts and unauthorized sources.
  • Process isolation: Each site gets its own renderer process (Site Isolation). Even if a malicious page exploits a renderer bug, it can't access other sites' memory.
  • HTTPS: Encrypts all data in transit. Modern browsers mark HTTP sites as "Not Secure."
  • SameSite cookies: Controls when cookies are sent cross-site, mitigating CSRF attacks.
  • Sandboxing: Renderer processes run with restricted OS permissions — they can't access the filesystem, system calls, or other processes directly.

The Complete Flow: From URL to Pixels

Let's trace the entire journey one more time, end to end:

1. URL Input & Parsing

Browser parses the URL into scheme, host, port, path. Checks HSTS preload list. Determines if this needs a new connection.

2. DNS Resolution

Browser cache → OS cache → Router → Recursive resolver → Root → TLD → Authoritative NS. Returns an IP address.

3. TCP + TLS Connection

Three-way TCP handshake (SYN → SYN-ACK → ACK), then TLS handshake for encryption. HTTP/3 does this in one step with QUIC.

4. HTTP Request/Response

Browser sends GET request with headers. Server processes (routing, application logic, DB queries) and returns HTML + headers.

5. HTML Parsing → DOM

Bytes → Characters → Tokens → Nodes → DOM Tree. Preload scanner discovers resources. Parser blocks on synchronous scripts.

6. CSS Parsing → CSSOM

All CSS parsed into CSSOM. Cascade, specificity, inheritance resolved. Computed styles calculated for every element. Render-blocking until complete.

7. Render Tree

DOM + CSSOM combined. Only visible nodes included (no display:none, no <head>).

8. Layout (Reflow)

Box model calculations: exact position and size of every element, starting from the viewport root.

9. Paint

Convert layout tree to paint records — ordered lists of drawing commands organized into layers.

10. Compositing

Layers are rasterized into bitmaps by raster threads, then the compositor thread sends them to the GPU which draws the final pixels to the screen.

11. JavaScript + Event Loop

V8 parses and executes JavaScript. The event loop manages async operations: microtasks (Promises) → render → macrotasks (timers, events). The page stays alive and interactive.
