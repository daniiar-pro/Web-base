01 What's a Package Manager? (And Why So Many?)

Imagine you want to build a LEGO castle. You could mold every brick yourself from plastic, but that would take years. Instead, you go to a LEGO store and buy bricks other people already made. A package manager is the helper who runs to the LEGO store for you, brings back the exact bricks you need, organizes them in a box, and remembers which ones you used so your friend can build the same castle.

In real terms: when you write a JavaScript app, you don't write everything from scratch. You use packages — bits of code other developers wrote and shared (React, Express, Lodash, etc.). A package manager is a tool that:

  • 📦 Downloads packages from the internet for you
  • 📋 Tracks which packages your project uses (and which versions)
  • 🌳 Figures out which packages those packages depend on (and so on, and so on...)
  • 🔒 Locks down exact versions so your teammate gets the same setup
  • 🧹 Removes packages cleanly when you don't need them
  • 🚀 Runs scripts like "start the dev server" or "build for production"

1.1 Why are there so many of them?

Each new tool was built because the previous one had a real, painful problem:

02 package.json — The Wishlist

Think of package.json as a shopping list taped to your fridge. It says "I need milk, eggs, and bread." It doesn't contain the actual food — just the list of what you want. When someone new moves into your house, they read the list and know what to buy.

Every JavaScript project has a file called package.json. It's the project's identity card and shopping list, all in one. Every package manager reads and writes to this same file — that's why they're interchangeable.

{
  "name": "my-cool-app",
  "version": "1.0.0",
  "description": "A really cool app",
  "main": "index.js",

  "scripts": {
    "start": "node index.js",
    "dev": "vite",
    "build": "vite build",
    "test": "jest"
  },

  "dependencies": {
    "react": "^18.2.0",
    "express": "^4.18.0"
  },

  "devDependencies": {
    "vite": "^5.0.0",
    "jest": "^29.0.0"
  }
}

2.1 The Important Fields

name + version

Your project's identity. If you publish to npm, this is what people install.

scripts

Shortcuts you can run. npm run dev runs whatever's in "dev". Like aliases for terminal commands.

dependencies

Packages your app needs to run in production. (React, your database driver, etc.)

devDependencies

Packages only needed while developing. (TypeScript, testing tools, build tools.) Not shipped to users.

peerDependencies

"I need this, but you should install it, not me." Used by libraries that need a host (e.g., a React plugin needs the user's React).

engines

"This project needs Node 20+." A polite warning to anyone who tries to run it with an older version.

2.2 The Magic Symbols Before Versions

Symbol Meaning Example: "react": "X18.2.0"
^ (caret) "Compatible" — minor and patch updates OK Allows 18.2.0 → 18.99.99, but NOT 19.0.0
~ (tilde) "Approximate" — only patch updates OK Allows 18.2.0 → 18.2.99, but NOT 18.3.0
(no symbol) Exact version. Don't touch it. Only 18.2.0, period.
* or latest "Whatever's newest." Risky in production. Could be anything. Don't do this.
Why ^ is the default When you run npm install react, it adds "^18.2.0" by default. The idea is that minor version bumps are "safe" — bug fixes and new features without breaking changes. (More on why this is a lie in Chapter 5.)

03 node_modules — The Toy Box

You asked the package manager for React. It went and got React. But React said "I need 47 other things to work." So the package manager got those too. Then those 47 things needed 200 other things... and they all get dumped into a giant folder called node_modules. It's notoriously huge — there's a meme that node_modules is the heaviest object in the universe.

When you run npm install, the package manager creates a folder called node_modules/ next to your package.json. This folder contains every package your project needs, plus every package those packages need, recursively. It can easily contain 1000+ folders for a small project.

3.1 The Famous Joke

node_modules is a meme A typical React project's node_modules can be 200-500MB with hundreds of thousands of files. The infamous "left-pad incident" of 2016 showed that the JS ecosystem depends on tiny packages — a single 11-line package being unpublished broke a huge chunk of the internet. This sprawl is exactly what pnpm and Bun try to fix.

3.2 The Golden Rule

Never commit node_modules to git. Always add it to your .gitignore. The lock file (next chapter) is what you commit — it lets anyone rebuild the exact same node_modules on demand.

04 Lock Files — The Receipt

Your shopping list says "milk." You went to the store and bought "Organic Valley 2% milk, 1 gallon, expires Jan 15". The lock file is the receipt — the exact thing you bought. Next time anyone wants the same groceries, they don't just buy "milk" (which could be anything!) — they buy the exact item from the receipt.

Remember those ^ symbols in package.json? They mean "any compatible version." That's a problem: if you install today and your teammate installs tomorrow, you might get different versions. The lock file solves this by recording the exact version of every single package that got installed.

Tool Lock File Name Format
npm package-lock.json JSON
Yarn (Classic) yarn.lock Custom (YAML-ish)
pnpm pnpm-lock.yaml YAML
Bun bun.lock (text) or bun.lockb (binary) Text or binary (faster)

4.1 What's Inside

A lock file records, for each package:

  • The exact version (e.g., 18.2.0, not ^18.2.0)
  • Where it was downloaded from (the registry URL)
  • A cryptographic hash (so you know nobody tampered with the package)
  • Its own dependencies (and their exact versions)
The Two Commands npm install reads package.json, decides what versions to use, and writes a new lock file.
npm ci reads the existing lock file and installs exactly that. Faster and safer for CI servers — it never updates anything.
The #1 mistake: not committing the lock file Always commit your lock file to git! It's how teammates and your production servers get identical installs. If you only commit package.json, every machine might end up with subtly different versions, leading to "works on my machine" disasters.

4.2 Lock File Conflicts

The most annoying part of lock files: when two people change them in different branches, git can't easily merge them. Most tools now offer auto-resolution:

  • npm install usually handles it automatically
  • Yarn has built-in merge logic
  • pnpm and Bun also auto-resolve in most cases

05 Semver — Version Numbers Explained

You know how video games have versions like "Mario 1.2.3"? The numbers aren't random. Semantic Versioning (semver) is a promise: each number tells you what kind of change happened. The biggest number means "things broke." The middle number means "new stuff was added." The last number means "tiny bug fix."

5.1 Semver Tells You How Worried to Be

Update Type Risk Level What to do
PATCH (1.0.0 → 1.0.1) 🟢 Low Update freely. Auto-applied with ^.
MINOR (1.0.0 → 1.1.0) 🟡 Medium Usually safe. Auto-applied with ^. Test it.
MAJOR (1.0.0 → 2.0.0) 🔴 High Read the changelog! Code may need rewrites.
Reality check: semver is a promise, not a guarantee Library authors are supposed to follow semver, but mistakes happen. A "minor" update might accidentally break your code. This is why lock files are so important — they protect you from surprises until you decide to update.

5.2 Special Versions

  • Pre-releases: 1.0.0-beta.2, 2.0.0-alpha.1, 3.0.0-rc.1. Skipped by default — you have to explicitly request them.
  • 0.x.x versions: "Anything goes." When a package is 0.x, the rules don't apply — even 0.1.0 → 0.2.0 can break things. Treat these as "early days."

06 The Registry — npm's Library

Imagine the world's biggest library, but for code. Anyone can borrow books (download packages) for free. Anyone can publish their own book (upload packages) for free. That library is npmjs.com — the npm registry. It has over 3 million packages, and it's the central hub that npm, Yarn, pnpm, and Bun all download from.

When you run npm install react, here's what really happens:

  1. The package manager asks https://registry.npmjs.org/react for info about the React package.
  2. The registry returns a JSON document listing every published version.
  3. The package manager picks the right version based on your package.json and downloads the .tgz (tarball) file.
  4. It unpacks the tarball into node_modules/react/.
  5. It repeats for every dependency. (And every dependency's dependencies.)

6.1 The Registry is Just a Server

The npm registry is owned by GitHub (which is owned by Microsoft). It's free, public, and the de facto standard. But it's not the only registry — you can run your own:

  • Verdaccio: Free, open-source private registry you run yourself. Good for companies.
  • GitHub Packages: Private registry hosted by GitHub.
  • JFrog Artifactory / AWS CodeArtifact: Enterprise registries.
  • JSR (jsr.io): A newer registry from the Deno team that supports TypeScript natively.

6.2 Scoped Packages

You'll sometimes see packages like @types/node or @vue/cli. The @something/ prefix is a "scope" — it groups packages under an organization. Scopes prevent name collisions: @google/maps and @apple/maps can both exist.

Supply chain risk Because anyone can publish to npm, malicious packages occasionally appear. A bad package can run scripts during install (via postinstall) and steal data. Modern protections: npm audit, lock files with hashes, and --ignore-scripts for paranoid installs. Bun and pnpm by default don't run install scripts of transitive dependencies for safety.

07 npm — The Original

npm

npm

2010 · "Node Package Manager" · Bundled with Node.js

The granddaddy. Comes pre-installed with Node.js — if you have Node, you have npm. The default everyone starts with.

7.1 The Story

npm was created by Isaac Schlueter in 2010, around the same time as Node.js itself. It quickly became the de facto package manager for JavaScript. By 2020, GitHub bought npm Inc., the company behind it.

For a long time, npm was the only option. It got the job done, but had real problems:

  • 🐌 Slow installs — single-threaded, lots of network round-trips
  • 🎲 Non-deterministic — same package.json could give different installs on different machines (before lock files existed in npm 5)
  • 📁 Confusing nested folders — early npm created deeply nested node_modules trees

7.2 What npm Does Well

  • ✅ Always available (ships with Node)
  • ✅ Largest ecosystem support — tutorials, docs, tooling all default to npm
  • ✅ Has caught up significantly on speed since v7
  • ✅ npx (bundled with npm) lets you run packages without installing them: npx create-react-app my-app

7.3 Common npm Commands

npm init                  # Create a new package.json
npm install               # Install everything from package.json
npm install react         # Add a new package
npm install -D vite       # Add as devDependency
npm uninstall lodash      # Remove a package
npm update                # Update packages within semver ranges
npm run dev               # Run a script defined in package.json
npm ci                    # Clean install from lock file (for CI)
npx create-vite my-app    # Run a package without installing it
npx — npm's secret weapon npx downloads and runs a package without permanently installing it. Perfect for one-off tools like project generators. It's why you can type npx create-react-app without installing it first.

08 Yarn — The Challenger

Y

Yarn

2016 · Built by Facebook · "Fix npm's problems"

In 2016, npm was slow and unreliable. Facebook (now Meta), Google, and a few others teamed up to build Yarn — a faster, more deterministic alternative that read the same package.json files.

8.1 What Yarn Brought to the Table

  • ⚡ Parallel downloads — multiple packages at once
  • 🔒 Lock files by default — npm didn't have these yet!
  • 📦 Offline cache — installed packages get cached, so re-installs work without internet
  • 🎨 Better terminal output — clearer, more colorful

Within a year, half the JavaScript world switched. npm panicked, and most of Yarn's innovations made it back into npm 5+.

8.2 Yarn Classic vs Yarn Berry

Plot twist: there are two Yarns In 2020, Yarn split into two completely different projects.

Yarn Classic (Yarn 1.x)

The original. Still widely used. Stable, familiar, works like npm. Most "Yarn" tutorials online refer to this.

Stable Most common

Yarn Berry (Yarn 2/3/4)

A complete rewrite. Introduced "Plug'n'Play" (PnP) — a radical mode that eliminates node_modules entirely, replacing it with a single .pnp.cjs file. Fast and disk-efficient, but breaks tools that expect node_modules to exist.

Experimental feel Niche but loved

8.3 Yarn Today

Yarn is still actively maintained, but it lost momentum after the Berry split confused users. Many teams have migrated to pnpm or back to npm. That said, it's still solid and widely deployed.

yarn                      # Install everything (no "install" needed)
yarn add react            # Add a package
yarn add -D vite          # Add as devDependency
yarn remove lodash        # Remove a package
yarn dev                  # Run a script (no "run" needed for unique names)
yarn dlx create-vite      # Like npx — run without installing

09 pnpm — The Smart One

p

pnpm

2017 · "Performant npm" · The disk space hero

pnpm noticed something embarrassing: if you have 50 projects, each with React installed, you're storing 50 copies of React on your disk. That's wasteful. pnpm stores each package once on your computer and uses links to share it.

9.1 The Genius Idea: A Global Store

Imagine a library where every book exists in only one copy. When you want to "borrow" it, instead of getting your own copy, you get a magical portal in your house that links to the one true book in the library. Open your portal, you see the book. Your neighbor opens their portal, they see the same book. The library only stores one copy. That's pnpm.

9.2 How It Actually Works

pnpm uses two file system tricks:

  • Hard links: Multiple folders can point to the exact same file on disk. The file isn't duplicated — it just appears in many places.
  • Symbolic links (symlinks): A folder that's secretly a "shortcut" to another folder.

Your project's node_modules still exists, but it's mostly symlinks to a global store at ~/.pnpm-store. Same packages, drastically less disk usage.

9.3 The Strict Bonus

pnpm has another superpower: it's strict about phantom dependencies. With npm/Yarn, your code can accidentally import packages you didn't list in package.json (because they happen to be in node_modules as a sub-dependency). pnpm prevents this — you can only import what you explicitly declared.

Why developers love pnpm Faster than npm/Yarn, uses way less disk space, catches dependency mistakes, and is fully npm-compatible. It's become the favorite of monorepo users (Vue, Vite, Next.js codebases all use it). 
pnpm install              # or just: pnpm i
pnpm add react            # Add a package
pnpm add -D vite          # Add as devDependency
pnpm remove lodash        # Remove
pnpm dev                  # Run a script
pnpm dlx create-vite      # Like npx

10 Bun — The Speed Demon

🥯

Bun

2022 · Written in Zig · "Be fastest at everything"

Bun is the new kid (and arguably the most ambitious). It's not just a package manager — it's a complete JavaScript runtime that wants to replace Node.js, npm, Webpack, Jest, and ts-node all at once. The package manager part is just one feature.

10.1 What Makes Bun Different

Bun is written in Zig (a systems programming language, like C but modern), not JavaScript. This means it can do things at native speed. Where npm and Yarn are themselves JavaScript programs running on Node.js, Bun is a compiled binary.

🚀 Runtime

A drop-in replacement for Node.js. Runs your JS/TS files directly. No ts-node needed.

📦 Package Manager

Reads package.json, installs to node_modules. Often 10-30x faster than npm.

🛠 Bundler

Built-in JS/TS bundler — competes with esbuild, Vite.

🧪 Test Runner

Built-in test runner with Jest-compatible API. No separate testing tool needed.

10.2 Why It's So Fast

  • Native code (Zig) instead of running on a JS runtime
  • Aggressive parallelism for downloads and file operations
  • Custom binary lock file (bun.lockb) — faster to read/write than text formats
  • Smarter caching — uses a global content-addressable store like pnpm

10.3 The Tradeoffs

Bun is fast, but young Bun aims for full Node.js compatibility but isn't 100% there yet. Some npm packages with native bindings (database drivers, image processors) might not work. Production deployment tooling is still catching up. Most teams use Bun for development and ship to Node.js for safety. 
bun init                  # Create a project
bun install               # or: bun i
bun add react             # Add a package
bun add -d vite           # devDependency (lowercase d!)
bun remove lodash         # Remove
bun run dev               # Run a script
bun dev                   # Shortcut works too
bunx create-vite          # Like npx
bun test                  # Run tests (no Jest needed)

11 Other Players Worth Knowing

11.1 Deno

Deno (made by the same person who created Node.js, Ryan Dahl) takes a radical stance: no node_modules, no package.json, no package manager at all. You import packages directly from URLs:

import { serve } from "https://deno.land/std/http/server.ts";

It caches downloads automatically. Recently though, Deno added deno install and now supports package.json too — converging back toward the npm world.

11.2 JSR (JavaScript Registry)

A new registry (jsr.io) launched by the Deno team. It supports TypeScript natively (no compilation needed before publishing) and works with npm, Yarn, pnpm, Bun, AND Deno. A potential glimpse of the future.

11.3 Corepack

Corepack ships with Node.js. It's not a package manager — it's a manager of package managers. You declare which one your project uses in package.json:

{
  "packageManager": "pnpm@8.15.0"
}

Corepack auto-downloads the right version. No more "install pnpm first" instructions in READMEs.

11.4 Version Managers (Different Beast!)

Don't confuse package managers with Node version managers:

  • nvm — switches between Node.js versions (Node 18, 20, 22, etc.)
  • fnm — same as nvm, but faster (written in Rust)
  • Volta — manages Node and package managers per project, automatically

These manage which version of Node you're running. They don't install packages — they install Node itself.

The full toolbelt A modern setup might be: fnm (manages Node versions) + Corepack (manages package manager versions) + pnpm (the actual package manager) + npm registry (where packages live). All four work together.

12 How Dependency Resolution Actually Works

You want React. React wants Object-Assign. Object-Assign wants Polyfill. Polyfill wants... and so on. The package manager has to figure out which version of everything to install — and what to do when two packages need different versions of the same dependency. This is the hardest job a package manager has.

12.1 The Diamond Problem

Here's a classic puzzle:

Your app
├── package-A wants lodash@^3.0.0
└── package-B wants lodash@^4.0.0

Both packages need lodash, but different major versions. You can't pick one — they really do need different versions! Solutions:

  1. Nest them: Put lodash@3 inside package-A's folder, lodash@4 inside package-B's. Each gets its own copy. Wasteful but always works. (npm's old strategy.)
  2. Hoist + nest: Put one version at the top level, nest the other one. (Modern npm/Yarn.)
  3. Symlink trick: Use the global store with two folder names pointing to the right versions. (pnpm's approach.)

12.2 Hoisting (and the Phantom Dependency Problem)

To save space, npm and Yarn "hoist" packages: even though lodash is technically a dependency of express, they put it at the top level of node_modules. This works... but creates a sneaky bug:

// Your code
import _ from 'lodash';   // 😱 You never declared lodash in package.json!
                          // It just happens to be there because express needs it.

This is a phantom dependency. Your code works today, but if you upgrade express and the new version stops needing lodash, your code breaks mysteriously. pnpm fixes this by only making your direct dependencies importable from the top level.

12.3 The SAT-Solver Inside

Picking the right combination of versions across hundreds of packages is mathematically the same problem as solving a Sudoku puzzle (it's NP-hard!). Modern package managers use sophisticated constraint solvers to find a valid combination. This is why npm install can sometimes take a while to "resolve" before downloads even start — it's doing serious computation.

13 Monorepos & Workspaces

Sometimes you have several related projects (frontend, backend, shared utilities) that you want to keep in one big folder. That's a monorepo. Each project has its own package.json, but they share a single node_modules. Workspaces are how the package manager handles this.

All four package managers support workspaces. You declare them in your root package.json:

{
  "name": "my-monorepo",
  "workspaces": [
    "apps/*",
    "packages/*"
  ]
}

Folder structure:

my-monorepo/
├── package.json          ← root, defines workspaces
├── node_modules/         ← shared dependencies
├── apps/
│   ├── web/
│   │   └── package.json  ← can depend on @my/utils
│   └── api/
│       └── package.json
└── packages/
    └── utils/
        └── package.json  ← reusable, internal to the repo

Apps in the workspace can import from each other directly, no publishing needed. The package manager creates symlinks between them.

13.1 Why Monorepos?

  • 🔄 Share code between projects without publishing to npm
  • 🔧 Update a shared library and instantly see effects in all apps
  • 📋 One place to install, lint, test all your projects
  • 🚀 Used by huge companies — Google, Meta, and most startups with multiple products

13.2 Specialized Monorepo Tools

  • Turborepo (by Vercel) — adds smart caching and parallel task running on top of any package manager
  • Nx — full monorepo platform with code generation, dependency graphs, and caching
  • Lerna — older tool, mostly absorbed into Nx now
  • Changesets — handles versioning and changelogs across packages in a monorepo
The pnpm + Turborepo combo This pairing has become the de facto standard for new monorepos. pnpm handles the install efficiently; Turborepo runs tasks (build, test, lint) in parallel with smart caching. Many big-name projects use this exact setup.

15 Which Should You Pick?

Honest answer: any of them will work fine for most projects. They all do the same job. The differences matter most when you're working at scale or have specific needs. Pick one and stick with it.

15.1 Honest Recommendations

🟢 You're new / learning

Use npm. It's pre-installed with Node, every tutorial uses it, and "speed" doesn't matter when you're learning. You can always switch later — your package.json will be portable.

🔵 You want the best modern default

Use pnpm. Fast, saves disk space, catches phantom dependency bugs, supports monorepos beautifully. The choice of most cool projects in 2026.

🟠 You want maximum speed

Try Bun. Especially for new projects. Be aware some libraries with native bindings might have rough edges. Excellent for development; many teams still ship with Node.js to production.

🟣 You're maintaining an existing project

Don't switch. Whatever the project uses now, keep using. Switching has costs (lock file conflicts, tooling differences) that are rarely worth the upside. Pick a new tool for new projects.

15.2 The Core Truth

It's all the same packages npm, Yarn, pnpm, and Bun all download the same packages from the same registry. The actual code your app runs is identical regardless. The differences are in: speed, disk usage, lock file format, and a few quality-of-life features. Your users will never notice.

15.3 The Big Trends in 2026

  • 📈 pnpm is winning the disk-space-conscious crowd — Vue, Vite, Next.js, and most major OSS projects use it
  • ⚡ Bun is gaining fast for startups who value speed over battle-tested stability
  • 🔧 Corepack is becoming standard — declaring your package manager in package.json is the new normal
  • 📦 JSR may reshape registries — TypeScript-native packages without compilation are appealing
  • 🦕 Deno's "URL imports" idea is fading as Deno itself moves toward npm compatibility

15.4 Final Mental Model

Remember the four-step picture from Chapter 1:

You ask
→
Manager fetches
→
Registry serves
→
Code lives in node_modules

Everything else — semver, lock files, hoisting, workspaces, symlink stores — is detail about how the manager does its job. The job itself is simple: get the right code, organize it, and remember exactly what you got.

Now go build something. 🚀
