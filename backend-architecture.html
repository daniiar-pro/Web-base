1. Architecture Styles (Complete)

The architecture style determines how your backend is decomposed, deployed, and scaled. This single decision has cascading effects on team structure, deployment cadence, infrastructure cost, fault tolerance, and developer experience. Below is every major backend architecture style used in modern systems.

1.1 Monolith (Single Deployable Unit)

One codebase, one process, one deployment. All features share the same memory, database, and lifecycle. The classical and simplest architecture — and still the right choice for most applications.

  ┌──────────────────────────────────────────────┐
  │               MONOLITH (one app)              │
  │  ┌─────────┐ ┌─────────┐ ┌─────────────────┐ │
  │  │  Users   │ │ Orders  │ │   Catalog       │ │
  │  │  Module  │ │ Module  │ │   Module        │ │
  │  └─────────┘ └─────────┘ └─────────────────┘ │
  │  ┌─────────┐ ┌─────────┐ ┌─────────────────┐ │
  │  │  Auth    │ │Payments │ │   Notifications │ │
  │  └─────────┘ └─────────┘ └─────────────────┘ │
  │              In-process function calls        │
  │              ↓        ↓        ↓             │
  │          Single shared database               │
  └──────────────────────────────────────────────┘
  
  Pros: Simple deploy, easy debugging, fast in-process calls.
  Cons: Whole app rebuilds for any change. Scaling = scaling everything.
Monolith — one process, one DB, function calls between modules
Tech Stacks: Ruby on Rails, Django, Laravel, Spring Boot, NestJS, ASP.NET, Phoenix. Most CRUD apps and SaaS startups should start here.
Real-World Usage: GitHub (mostly Rails monolith for years), Shopify (Ruby monolith), Basecamp (Rails), early Twitter/Instagram. "Majestic Monolith" has had a renaissance — DHH, Shopify, and others actively defend this style.
Advantages
  • Simplest possible architecture
  • One deploy, one log stream, one DB
  • Refactoring across modules is trivial
  • Local dev = one process
  • Transactions are easy (single DB)
Disadvantages
  • Bottleneck: all teams ship to one binary
  • Tech stack lock-in (one language)
  • Scaling is all-or-nothing
  • Tight coupling creeps in over years
  • Long startup times when huge

1.2 Modular Monolith

Still one deployable, but with strict internal boundaries between modules. Modules expose explicit APIs to each other and own their tables. The pragmatic middle ground — most of monolith's simplicity, most of microservices' clean boundaries.

  ┌────────────────────────────────────────────────────┐
  │             MODULAR MONOLITH (one app, strict)     │
  │                                                    │
  │  ┌────────────────┐    ┌────────────────┐          │
  │  │ Users Module   │    │ Orders Module  │          │
  │  │  ├ public API  │◄──►│  ├ public API  │          │
  │  │  ├ internal    │    │  ├ internal    │          │
  │  │  └ users table │    │  └ orders table│          │
  │  └────────────────┘    └────────────────┘          │
  │                                                    │
  │  ┌────────────────┐    ┌────────────────┐          │
  │  │ Catalog Module │    │ Billing Module │          │
  │  └────────────────┘    └────────────────┘          │
  └────────────────────────────────────────────────────┘
  
  RULES:
  • Modules talk via PUBLIC interfaces only (typed contracts)
  • Each module OWNS its tables — no cross-module SQL joins
  • Database can be split later (1 schema → N services) easily
  • Compile-time enforcement (e.g., NestJS modules, .NET assemblies)
Modular Monolith — single deploy, but each module is a "future microservice"
Tech Stacks: NestJS (modules + DI), .NET (project references), Spring Boot (Spring Modulith), Ruby (packwerk). Tools like ArchUnit (Java), ts-arch (TypeScript), and dependency-cruiser enforce module boundaries automatically.
Real-World Usage: Shopify (modular Rails monolith with packs), GitHub (similar evolution), Stack Overflow. Increasingly the recommended starting point in 2025-2026.

1.3 Microservices

The application is decomposed into independently deployable services, each owning its data and bounded context. Services communicate over the network (HTTP, gRPC, message queues). Each can use its own language, database, and team.

  ┌─────────────┐      ┌──────────────────┐      ┌──────────────┐
  │  API GATEWAY │ ───► │   Users Service   │      │ Orders Service│
  │   (routing,  │      │  (Node.js + PG)   │      │ (Go + Mongo)  │
  │    auth)     │      └────────┬──────────┘      └──────┬────────┘
  └─────────────┘               │                         │
        │                       │  HTTP / gRPC / events  │
        │                       │                         │
        ▼              ┌────────▼────────┐    ┌──────────▼──────┐
  ┌─────────────┐      │ Catalog Service │    │ Payment Service │
  │   Frontend   │      │ (Python + ES)   │    │ (Java + MySQL)  │
  └─────────────┘      └─────────────────┘    └─────────────────┘
  
  Each service:
  • Owns its database (no shared schema!)
  • Has its own deploy pipeline
  • Can be scaled independently
  • Can be rewritten without touching others
  • Communicates over the network only
Microservices — independent services, network communication, polyglot
Tech Stacks: Kubernetes + Docker, service mesh (Istio, Linkerd), API Gateway (Kong, Envoy, AWS API GW), gRPC, Kafka, RabbitMQ, distributed tracing (Jaeger, Tempo). Languages mix freely: Go, Java, Node, Python, Rust.
Real-World Usage: Netflix (the canonical example, ~1000+ services), Uber, Amazon, Spotify, Airbnb. Generally only worth the complexity at significant scale or organizational size.
Advantages
  • Independent deploys per team
  • Per-service scaling (scale only what's hot)
  • Polyglot — pick best tool per service
  • Fault isolation (one service down ≠ system down)
  • Smaller, focused codebases
Disadvantages
  • Distributed systems = much harder
  • Network calls fail / are slow
  • Distributed transactions are nightmare-tier
  • Eventual consistency everywhere
  • Operational overhead (monitoring × N)

1.4 Service-Oriented Architecture (SOA)

The grandparent of microservices. Services are larger, more coarse-grained, and traditionally communicate through an Enterprise Service Bus (ESB) using SOAP/XML. Modern variants use REST/messaging instead. Microservices = SOA done right (smaller services, no central ESB, decentralized governance).

  Classic SOA                          Microservices
  ───────────                          ─────────────
  ┌─────┐ ┌─────┐ ┌─────┐              ┌─────┐ ┌─────┐ ┌─────┐
  │ Svc │ │ Svc │ │ Svc │              │ Svc │ │ Svc │ │ Svc │
  └──┬──┘ └──┬──┘ └──┬──┘              └──┬──┘ └──┬──┘ └──┬──┘
     └──────┼──────┘                      └──────┼──────┘
            ▼                                    ▼
  ┌──────────────────────┐              direct HTTP/gRPC/Kafka
  │  Enterprise Service  │              (no central pipe!)
  │       Bus (ESB)      │              
  │  routing, transform, │              Smarter endpoints,
  │   orchestration      │              dumber pipes.
  └──────────────────────┘
  
  "Smart pipes, dumb endpoints"        "Dumb pipes, smart endpoints"
SOA's centralized bus → Microservices' decentralized communication

1.5 Serverless / FaaS (Functions as a Service)

You write functions; the cloud runs them. No servers to manage, no idle costs — you pay per invocation. Functions auto-scale from zero to thousands of concurrent executions. Cold starts and stateless constraints are the trade-offs.

  Traditional Server                   Serverless / FaaS
  ──────────────────                   ─────────────────────
  Always-on VM/container               Function dormant when not called
       │                                       │
       │  ↑ pays $$$ even idle                 │  ↑ $0 when idle
       │                                       │
  Request comes in                     Request comes in
       ↓                                       ↓
  Handled immediately                   Cold start (50-2000ms)
       ↓                                       ↓
  Response                             Function instantiated
                                              ↓
  Capacity: fixed                       Handle request
  Scaling:  manual / autoscaler         ↓
                                       Response
                                       (instance reused for ~minutes)
                                       
  Capacity: 0 to ∞ automatically
  Scaling:  free, instant, automatic
  
  ┌──────────────────────────────────────────────────────┐
  │  Lambda function example                            │
  │  exports.handler = async (event) => {              │
  │    const order = await db.orders.create(event.body)│
  │    await sqs.send({ orderId: order.id })           │
  │    return { statusCode: 200, body: order }          │
  │  }                                                  │
  │  Limits: 15min max execution, 10GB RAM, stateless   │
  └──────────────────────────────────────────────────────┘
Serverless — pay-per-use, auto-scale, but stateless and cold-start prone
Tech Stacks: AWS Lambda, Azure Functions, Google Cloud Functions, Cloudflare Workers (V8 isolates — almost no cold start), Vercel/Netlify Functions, Deno Deploy. Frameworks: Serverless Framework, SST, AWS CDK, Pulumi.
Real-World Usage: Image processing pipelines, scheduled jobs, webhooks, glue code between SaaS, low-traffic APIs, edge personalization. Bad fit for: long-running tasks, WebSocket servers, heavy stateful workloads, latency-critical services.

1.6 Event-Driven Architecture

Services don't call each other directly — they publish events ("OrderPlaced", "PaymentReceived") to a message broker. Other services subscribe to events they care about. Loosely coupled, asynchronous, and scalable.

  Synchronous (request/response)        Event-Driven
  ─────────────────────────────          ──────────────
  Order Svc ──► Payment Svc              Order Svc ─► [event bus]
              ├─► Email Svc                              │   │   │
              ├─► Inventory                              │   │   ▼
              ├─► Analytics                              │   │  Email Svc
              └─► Recommend                              │   ▼
                                                         │  Inventory Svc
  Order Svc must KNOW about each            ─────────────┘
  downstream. Adding a 6th = touch         Order Svc just FIRES the event.
  Order Svc.                                It doesn't know who listens.
                                            Add a 6th consumer? No code changes.
  
  Event bus options:
  ┌────────────────────────────────────────────────────────┐
  │ Kafka       — log-based, replayable, high throughput   │
  │ RabbitMQ    — classic queue/exchange model             │
  │ NATS        — lightweight, simple, fast                │
  │ AWS SNS/SQS — managed, pub/sub + queue                 │
  │ Redpanda    — Kafka-compatible, no JVM, fast           │
  │ Pulsar      — multi-tenant, geo-replication            │
  └────────────────────────────────────────────────────────┘
Event-driven — publish events, subscribers react, no direct coupling
Real-World Usage: LinkedIn (Kafka was born there), Uber, Netflix, Stripe (events for everything: subscription.created, charge.succeeded, etc.). The norm at any scale where many services react to user actions. See the dedicated Async Architecture A→Z page for queues, jobs, DLQs, idempotency, and saga details.

1.7 Hexagonal / Clean / Onion Architecture (as a style)

The application's domain logic sits at the center, isolated from frameworks, databases, and external services. The outer layers ("adapters") implement interfaces ("ports") defined by the domain. Dependencies flow inward — domain knows nothing about Postgres or HTTP.

                ┌─────────────────────────────────────────┐
                │       Frameworks & Drivers (outer)      │
                │  HTTP, CLI, Cron, Message Queue          │
                │  ┌───────────────────────────────────┐  │
                │  │   Interface Adapters              │  │
                │  │   Controllers, Presenters         │  │
                │  │  ┌─────────────────────────────┐  │  │
                │  │  │   Application / Use Cases   │  │  │
                │  │  │   PlaceOrder, CancelOrder   │  │  │
                │  │  │  ┌─────────────────────┐    │  │  │
                │  │  │  │   Domain (entities) │    │  │  │
                │  │  │  │   Order, Money,     │    │  │  │
                │  │  │  │   Customer          │    │  │  │
                │  │  │  └─────────────────────┘    │  │  │
                │  │  └─────────────────────────────┘  │  │
                │  └───────────────────────────────────┘  │
                │  Postgres adapter, Stripe adapter, etc. │
                └─────────────────────────────────────────┘
  
  Dependencies point INWARD only:
  Domain ◄─── Application ◄─── Adapters ◄─── Frameworks
  
  Domain is pure: no `import express`, no `import @prisma/client`.
  Swap Postgres → Mongo? Only the Postgres adapter changes.
  Swap REST → GraphQL? Only the controller adapter changes.
Hexagonal — pure domain at center, frameworks at the edges
Real-World Usage: The default for any team that practices DDD or expects to outlive its current framework choice. Common in fintech, healthcare, and any domain where the rules of the business outlast specific tech. We cover it again as a "structural pattern" in §2 — it can be both an overall style AND a pattern within any service.

1.8 Cell-Based Architecture

The system is composed of independent "cells" — fully self-contained vertical slices that include compute, storage, and networking. Each cell handles a partition of users/traffic. If one cell fails, only that subset of users is affected. Used by AWS, Slack, and others at extreme scale for blast-radius reduction.

  ┌────────────────────────── Cell-Based Architecture ──────────────────────────┐
  │                                                                             │
  │   Router (sharded by user ID hash)                                          │
  │       │                                                                     │
  │       ├──► Cell A ────────►  Self-contained: services + DB + cache + queue │
  │       │    Users 0-25%       (own everything; no shared state with others) │
  │       │                                                                     │
  │       ├──► Cell B ────────►  Same stack, different users (25-50%)          │
  │       │                                                                     │
  │       ├──► Cell C ────────►  Same stack, different users (50-75%)          │
  │       │                                                                     │
  │       └──► Cell D ────────►  Same stack, different users (75-100%)         │
  │                                                                             │
  │   If Cell B has a bad deploy → only 25% of users impacted.                  │
  │   Add a new cell to scale out, drain & rebuild a cell to upgrade.           │
  └─────────────────────────────────────────────────────────────────────────────┘
Cell-based — partition users into independent stacks, contain blast radius
Real-World Usage: AWS DynamoDB, S3, Lambda (all cell-based internally). Slack moved to cells after a major outage. Doordash, Roblox. Generally only worth it at scale where uptime SLAs matter more than infrastructure cost.

1.9 Backend for Frontend (BFF)

Instead of one API serving every client (web, mobile, smart TV), each client gets its own dedicated backend — a "BFF" — that aggregates downstream services and tailors responses to that specific client's needs. Mobile apps want trimmed payloads; web apps want different data shapes. BFFs prevent over-/under-fetching.

  Without BFF (one API for all)         With BFF (one API per client type)
  ──────────────────────────────         ──────────────────────────────────
  Web ─┐                                 Web ───────► Web BFF ─┐
       │                                                       │
  iOS ─┼──► Single API ──► services      iOS ───────► iOS BFF ─┼──► services
       │                                                       │
  TV  ─┘                                 TV  ───────► TV BFF  ─┘
  
  Problem:                              Each BFF:
  • Mobile gets bloated payloads        • Tailored response shape
  • Web has to do many round trips      • Aggregates calls server-side
  • One API change breaks all clients   • Owned by client team
                                        • Versioned with the client
BFF — one backend per client type, optimized payloads, no over-fetching
Real-World Usage: SoundCloud (originated the term), Netflix, Spotify, Booking.com. Often paired with GraphQL (each BFF is a small GraphQL server) or tRPC. Mobile teams especially benefit.

1.10 Edge Computing / Distributed Backend

Instead of a centralized backend in one or two regions, your code runs at hundreds of edge locations close to users. V8 isolates (Cloudflare Workers, Deno Deploy) make this practical for stateful logic too — sub-50ms TTFB globally.

  Centralized Backend                    Edge Backend
  ────────────────────                    ─────────────────────
  ┌──────────┐                            User (Tokyo) ─► nearest edge (15ms)
  │  Origin   │ ◄── User in Tokyo         User (Berlin) ─► nearest edge (12ms)
  │ (US-East) │       (300ms RTT)         User (NYC)    ─► nearest edge (8ms)
  │ Postgres  │
  └──────────┘                            Each edge runs your code in V8 isolates.
                                          Cold start: ~5ms (vs Lambda's 200-2000ms).
  All requests go to one place.
                                          State: edge KV stores, Durable Objects,
                                                 D1 (SQLite at edge), Turso.
  
  Hybrid pattern:
  • Static assets & auth   → edge
  • Read-heavy queries     → edge with cache
  • Writes & transactions  → origin (write-through to edge cache)
Edge computing — code runs near users, sub-50ms TTFB globally
Tech Stacks: Cloudflare Workers + D1 + KV + R2 + Durable Objects, Vercel Edge Functions, Deno Deploy + Deno KV, Fastly Compute@Edge (Wasm), AWS Lambda@Edge, Netlify Edge Functions. Frameworks: Hono, itty-router, Elysia.

1.11 CQRS & Event Sourcing (as a style)

Two related but separable ideas, often used together. CQRS (Command Query Responsibility Segregation) means writes and reads use separate models. Event Sourcing means the source of truth is an append-only log of events, not the current state — current state is derived by replaying events.

  Traditional CRUD                       CQRS + Event Sourcing
  ─────────────────                       ──────────────────────
  Update users                            Command: "ChangeEmail"
  SET email = 'new@x.com'                       │
  WHERE id = 42                                 ▼
                                          ┌──────────────┐
  Old email is GONE.                       │ Event Store  │ ← append-only
  No history.                              │ (log of all  │
                                          │  events)      │   UserCreated  ─┐
  ┌─────────────────┐                     └──────┬───────┘   EmailChanged ─┤
  │   Users Table    │                            │            EmailChanged─┤
  │  id, name, email │                            │                         │
  └─────────────────┘                       Project to read models          │
  Same model for reads & writes.                  │                         │
                                                  ▼                         │
                                          ┌──────────────┐ ┌──────────┐    │
                                          │User Profile   │ │Audit Log │ ◄──┘
                                          │ (read model)  │ │(another  │
                                          │  current view │ │ read model)
                                          └──────────────┘ └──────────┘
                                          
                                          Replay events to rebuild state.
                                          Add new read models any time.
CQRS + Event Sourcing — events are truth, current state is derived
Real-World Usage: Banking and ledgers (every transaction is an event), e-commerce order systems, audit-heavy domains. Heavy lift — only worth it when audit history, time-travel, or complex projections are core requirements. Frameworks: EventStoreDB, Axon, Marten, Equinox.

2. Structural / Layering Patterns

Within any architecture style, you still need to organize the code inside a service. These patterns govern how you separate concerns, where business rules live, and which layer can call which.

2.1 Layered (N-tier) Architecture

  ┌─────────────────────────────────────────────────┐
  │           Presentation Layer                     │
  │  Controllers, request validation, serialization  │
  │  (Express handlers, NestJS controllers, etc.)    │
  └────────────────────┬────────────────────────────┘
                       │ depends on ▼
  ┌─────────────────────────────────────────────────┐
  │             Service / Application Layer         │
  │  Use cases, transactions, orchestration          │
  └────────────────────┬────────────────────────────┘
                       │ depends on ▼
  ┌─────────────────────────────────────────────────┐
  │              Domain Layer                        │
  │  Entities, value objects, business rules         │
  │  Pure — no framework, no DB                      │
  └────────────────────┬────────────────────────────┘
                       │ depends on ▼
  ┌─────────────────────────────────────────────────┐
  │            Infrastructure Layer                  │
  │  ORM, HTTP clients, message queue, file storage  │
  └─────────────────────────────────────────────────┘
Layered architecture — top-down dependency, no skipping or backward arrows

2.2 Clean Architecture / Onion Architecture

Same concentric ring as hexagonal — domain at the center, dependencies point inward. The difference from layered: infrastructure depends on domain interfaces, not the other way around. The "dependency inversion" makes the domain testable without any infrastructure.

                ┌─────────────────────────────────────┐
                │  Frameworks & Drivers (Express, K8s)│
                │ ┌─────────────────────────────────┐ │
                │ │    Adapters (DB, HTTP clients)  │ │
                │ │ ┌─────────────────────────────┐ │ │
                │ │ │  Application Services / UC  │ │ │
                │ │ │ ┌─────────────────────────┐ │ │ │
                │ │ │ │       Domain            │ │ │ │
                │ │ │ │  Entities + Value Obj   │ │ │ │
                │ │ │ │  Business Rules (pure)  │ │ │ │
                │ │ │ └─────────────────────────┘ │ │ │
                │ │ └─────────────────────────────┘ │ │
                │ └─────────────────────────────────┘ │
                └─────────────────────────────────────┘
  
  Domain layer:        const order = Order.place(items, customer)  // pure logic
  Application:         await orderRepo.save(order)                 // calls interface
  Adapter (Postgres):  class PgOrderRepo implements OrderRepository { ... }
  Test (in-memory):    class InMemOrderRepo implements OrderRepository { ... }
Clean / Onion — same shape as hexagonal, "use cases" layer is explicit

2.3 Hexagonal / Ports & Adapters

Functionally identical to Clean / Onion. The vocabulary is different: "ports" are interfaces the domain defines (e.g., OrderRepository); "adapters" are implementations of those ports (e.g., PostgresOrderRepository, RedisOrderRepository, InMemoryOrderRepository). The metaphor: the domain has ports like a USB hub; you can plug in any compatible adapter.

2.4 Domain-Driven Design (DDD)

Not strictly a layering pattern but the philosophy that drives them. Identify bounded contexts (parts of the business with their own ubiquitous language), build a rich domain model in each, and treat external systems as detail. Tactical patterns: Entity, Value Object, Aggregate, Repository, Domain Event, Domain Service.

  Bounded Context: Sales              Bounded Context: Shipping
  ──────────────────────              ───────────────────────────
  • Order (aggregate root)            • Shipment (aggregate root)
  • OrderLine                         • Package
  • Money (value object)              • Address (value object)
  • Customer                          • Carrier
  
  "Order" means something different in each context:
  Sales:    items, prices, discounts, customer
  Shipping: weight, dimensions, address, tracking number
  
  ┌──────────────────────────────────────────────────────┐
  │ Communication BETWEEN contexts: events or APIs.      │
  │ NEVER share entities or DB tables across contexts.   │
  │                                                      │
  │ Sales publishes "OrderConfirmed" event.              │
  │ Shipping subscribes & creates its OWN Shipment.      │
  └──────────────────────────────────────────────────────┘
DDD bounded contexts — same word, different meaning, separate models

2.5 Vertical Slice Architecture

Instead of horizontal layers (presentation → service → domain → infra), organize each feature as a self-contained vertical slice. Each slice has its own controller, handler, validation, and data access — minimal sharing between slices. Popular with CQRS and "Request → Handler" patterns (MediatR in .NET).

  Layered                              Vertical Slice
  ────────                              ──────────────
  src/                                  src/features/
  ├── controllers/                      ├── place-order/
  │   ├── orders.controller.ts          │   ├── PlaceOrderCommand.ts
  │   └── users.controller.ts           │   ├── PlaceOrderHandler.ts
  ├── services/                         │   ├── PlaceOrderValidator.ts
  │   ├── order.service.ts              │   └── place-order.endpoint.ts
  │   └── user.service.ts               ├── cancel-order/
  ├── repositories/                     │   ├── CancelOrderCommand.ts
  │   ├── order.repository.ts           │   ├── CancelOrderHandler.ts
  │   └── user.repository.ts            │   └── ...
  └── domain/                           └── get-order-history/
      ├── order.ts                          ├── GetOrderHistoryQuery.ts
      └── user.ts                           ├── GetOrderHistoryHandler.ts
                                            └── ...
  
  Adding a feature touches 4+ folders.   Adding a feature = one folder.
  Removing a feature is risky.           Removing = delete folder.

3. Design Principles

Universal guidelines that apply regardless of language or framework. These keep your codebase healthy as it grows.

3.1 SOLID Principles (Backend Edition)

S — Single Responsibility Principle

A class/module should have one reason to change. A service that handles HTTP, validates input, talks to the DB, sends emails, and renders PDFs has five reasons to change.

SRP — split by reason to change
// ❌ Violates SRP — controller does everything
class OrderController {
  async placeOrder(req, res) {
    if (!req.body.items?.length) return res.status(400).json({error: '...'})
    const total = req.body.items.reduce((s, i) => s + i.price * i.qty, 0)
    const order = await db.orders.create({ ...req.body, total })
    await sendgrid.send({ to: req.user.email, subject: 'Confirmed', ... })
    await analytics.track('order_placed', { orderId: order.id })
    res.json(order)
  }
}

// ✅ SRP applied
class OrderController {
  constructor(private placeOrderUseCase: PlaceOrderUseCase) {}
  async placeOrder(req, res) {
    const order = await this.placeOrderUseCase.execute(req.body, req.user)
    res.json(order)
  }
}
class PlaceOrderUseCase {
  constructor(
    private orders: OrderRepository,
    private events: EventBus,
  ) {}
  async execute(input, user) {
    const order = Order.place(input.items, user)   // domain logic
    await this.orders.save(order)                   // persistence
    await this.events.publish(new OrderPlacedEvent(order))  // side effects async
    return order
  }
}

O — Open/Closed Principle

Open for extension, closed for modification. Use strategies/plugins for new behavior instead of editing existing code.

OCP — strategy pattern for payment methods
// ✅ Add new payment methods without touching existing code
interface PaymentProvider {
  charge(amount: Money, customerId: string): Promise<PaymentResult>
}
class StripeProvider implements PaymentProvider { ... }
class PayPalProvider implements PaymentProvider { ... }
class CryptoProvider implements PaymentProvider { ... }   // ← new, no edits to others

class PaymentService {
  constructor(private providers: Map<string, PaymentProvider>) {}
  charge(method: string, amount: Money, customerId: string) {
    return this.providers.get(method)!.charge(amount, customerId)
  }
}

L — Liskov Substitution Principle

Subtypes must be usable wherever the base type is expected, without surprising behavior. A SqlOrderRepository and an InMemoryOrderRepository implementing the same OrderRepository interface must both save orders the same way semantically — same error contract, same idempotency, same observable behavior.

I — Interface Segregation Principle

Don't force consumers to depend on methods they don't use. Many small interfaces beat one fat interface.

ISP — split fat interfaces
// ❌ Fat interface
interface UserService {
  create(input): Promise<User>
  update(id, input): Promise<User>
  delete(id): Promise<void>
  changePassword(id, pw): Promise<void>
  resetPassword(email): Promise<void>
  uploadAvatar(id, file): Promise<string>
  exportGdprData(id): Promise<Buffer>
}
// Read-only consumer (e.g., a reporting service) doesn't need any of this.

// ✅ Segregated
interface UserReader { findById(id): Promise<User>; findByEmail(email): Promise<User> }
interface UserWriter { create(input): Promise<User>; update(id, input): Promise<User> }
interface UserAuthOps { changePassword(id, pw): Promise<void>; resetPassword(email) }
interface UserGdprOps { exportData(id): Promise<Buffer>; deleteAccount(id) }

// Reporting service depends on UserReader only.

D — Dependency Inversion Principle

High-level modules should not depend on low-level modules. Both depend on abstractions. In backend: business logic depends on repository interfaces, not Postgres clients.

  ❌ Direct dependency                   ✅ Inverted dependency
  ──────────────────                      ──────────────────────
  PlaceOrderUseCase                      PlaceOrderUseCase
       │                                      │
       │ import { Pool } from 'pg'             │ depends on
       │ pool.query('INSERT ...')              ▼
       ▼                                  OrderRepository (interface)
  [pg driver]                                  ▲        ▲
                                                │        │
  Switch to Mongo or Cosmos?                Pg-Repo   InMem-Repo
  PlaceOrderUseCase changes.            (production)  (tests)
                                        Use case never changes when DB does.
DIP — domain depends on abstractions, infra implements them

3.2 DRY, KISS, YAGNI

DRY — Don't Repeat Yourself

Extract repeated logic. But "Rule of Three" first — wait until you've seen the same thing three times before extracting. Premature DRY creates wrong abstractions, and a wrong abstraction is more expensive than duplication.

KISS — Keep It Simple, Stupid

Pick the simplest thing that works. Don't reach for Kafka if a Postgres queue table is enough. Don't build microservices for a 5-person team. Complexity must be justified by current pain, not anticipated future pain.

YAGNI — You Aren't Gonna Need It

Don't add hooks, plugin systems, or configuration knobs for hypothetical future use cases. Half of "future-proofing" gets thrown away when the actual future arrives looking nothing like the prediction.

3.3 Composition over Inheritance

Inheritance creates rigid hierarchies and tight coupling. Composition (combining small, focused pieces) is more flexible. In backend: prefer middleware chains, decorators, dependency injection, and small interfaces over deep class trees.

Composition via middleware chain (Express/Hono/etc.)
// ❌ Deep inheritance
class ApiController { ... }
class AuthenticatedController extends ApiController { ... }
class AdminAuthenticatedController extends AuthenticatedController { ... }
class RateLimitedAdminAuthenticatedController extends AdminAuthenticatedController { ... }
// Adding "logged" or "cached"? Combinatorial explosion.

// ✅ Composition
app.post('/admin/users',
  authenticate,           // adds req.user
  authorize('admin'),     // checks role
  rateLimit({ rpm: 60 }), // throttles
  validate(CreateUserSchema),
  cache({ ttl: 30 }),
  createUserHandler
)
// Each piece is independent. Mix-and-match per route.

3.4 Law of Demeter

"Talk to friends, not strangers." A method should call methods on: itself, its parameters, objects it creates, or its direct dependencies. Avoid order.customer.address.country.taxRate chains — you've now coupled to four levels of internal structure. Either ask for what you need (order.getTaxRate()) or pass the right object directly.

3.5 Twelve-Factor App

The classical checklist for cloud-native services. Worth knowing by heart.

   1. Codebase     — one app per repo, many deploys
   2. Dependencies — declared explicitly, isolated (lockfile, container)
   3. Config       — in environment variables, NEVER in code
   4. Backing svcs — DB, Redis, etc. as attached resources via URL
   5. Build/release/run — strict separation; no editing in production
   6. Processes    — stateless; share nothing (state goes to DB/cache)
   7. Port binding — app exposes itself via HTTP port; no Apache/Nginx baked in
   8. Concurrency  — scale by adding processes, not threads
   9. Disposability — fast startup, graceful shutdown (SIGTERM)
  10. Dev/prod parity — same backing services in dev as in prod
  11. Logs        — write to stdout; let infra collect & route
  12. Admin tasks — run as one-off processes (db migrate, cron)
Twelve-Factor — the canonical cloud-native checklist

3.6 Separation of Concerns

Each module should care about one slice of the problem. Authentication ≠ business logic ≠ persistence ≠ logging. Mixing concerns is the #1 cause of "I can't change X without breaking Y."

4. API Design Patterns

How clients talk to your backend. The choice impacts coupling, type safety, performance, and developer ergonomics.

4.1 REST

Resources identified by URLs, manipulated with HTTP verbs. Stateless. The default for public APIs since 2005. Best when resources map cleanly to nouns and clients are diverse.

  GET    /api/orders            → list orders
  GET    /api/orders/42         → fetch order 42
  POST   /api/orders            → create order
  PUT    /api/orders/42         → replace order 42
  PATCH  /api/orders/42         → partial update
  DELETE /api/orders/42         → delete order 42
  
  GET /api/orders/42/items      → nested resource
  
  Status codes:
  2xx success | 3xx redirect | 4xx client error | 5xx server error
  
  Levels of REST (Richardson Maturity Model):
  Level 0: HTTP-RPC (one URL, one verb)
  Level 1: Resources (many URLs)
  Level 2: HTTP verbs + status codes (most "REST" APIs are here)
  Level 3: HATEOAS — responses include links to next actions
REST — resources + HTTP verbs, stateless, cacheable

4.2 GraphQL

One endpoint (/graphql). Clients send a query specifying exactly the fields they need. Solves over-fetching and under-fetching elegantly. Great for mobile clients, complex front-ends, and federating multiple backends.

  REST (multiple round trips)            GraphQL (one query)
  ───────────────────────────             ─────────────────────
  GET /api/users/42         → { id, name, email, ... 30 fields }
  GET /api/users/42/posts   → [{ ... }, { ... }]
  GET /api/users/42/orders  → [{ ... }, { ... }]
  
  3 requests, lots of unused fields.     query {
                                          user(id: 42) {
                                            name
                                            posts(limit: 5) { title }
                                            orders(status: SHIPPED) { id, total }
                                          }
                                         }
                                         
                                         1 request. Exactly the fields needed.
  
  Trade-offs:
  • Caching is harder (HTTP cache layer doesn't work)
  • Authorization per field (n+1 risks if naive)
  • Tooling: Apollo, urql, Relay; servers: Apollo Server, Yoga, Mercurius, Hot Chocolate
GraphQL — one endpoint, client-driven queries, no over/under-fetching

4.3 gRPC

Binary RPC over HTTP/2 using Protocol Buffers (Protobuf) for schema. Extremely fast and type-safe. The default for service-to-service communication in microservice landscapes. Bad fit for browsers (need a gRPC-Web proxy).

orders.proto

service OrderService {
  rpc PlaceOrder(PlaceOrderRequest) returns (Order);
  rpc StreamOrders(StreamOrdersRequest) returns (stream OrderEvent);
}

message Order {
  string id = 1;
  string customer_id = 2;
  repeated OrderItem items = 3;
  Money total = 4;
}

// Generates type-safe clients/servers in 11+ languages.
// Wire format is binary — much smaller than JSON.
// Built-in: streaming, deadlines, cancellation, retries.

5. Data Access Patterns

How your code talks to the database. The pattern you pick affects testability, swappability, and how clean your domain layer stays.

5.1 Repository Pattern

Define a domain interface (OrderRepository) that describes what you do with orders. Implement it once for the real DB and once for tests (in-memory). Domain code never touches SQL.

Repository — abstraction over data access
// Domain layer — pure
interface OrderRepository {
  findById(id: OrderId): Promise<Order | null>
  save(order: Order): Promise<void>
  findByCustomer(customerId: CustomerId): Promise<Order[]>
}

// Infrastructure layer
class PostgresOrderRepository implements OrderRepository {
  constructor(private db: Pool) {}
  async findById(id: OrderId) {
    const row = await this.db.query('SELECT * FROM orders WHERE id = $1', [id])
    return row ? Order.fromRow(row) : null
  }
  // ...
}

// Test layer
class InMemoryOrderRepository implements OrderRepository {
  private orders = new Map<string, Order>()
  async findById(id) { return this.orders.get(id) ?? null }
  async save(order) { this.orders.set(order.id, order) }
}

// Use case never knows which one is plugged in.

5.2 Unit of Work

Group multiple repository operations into a single transaction. Think of it as a "shopping cart for DB changes" — collect all the changes, then commit them atomically (or roll them back if anything fails).

Unit of Work — transactional boundary
await uow.transaction(async (tx) => {
  const order = await tx.orders.save(newOrder)
  await tx.inventory.decrement(items)
  await tx.payments.charge(customer, total)
  // All committed atomically. Any throw → all rolled back.
})

// Implementations: TypeORM & MikroORM "EntityManager", Prisma transactions,
// Spring's @Transactional, .NET's IUnitOfWork.

5.4 CQRS — Command Query Responsibility Segregation

Use different models for reads and writes. Writes go through commands that mutate the write model (often normalized). Reads come from a read model optimized for the query (often denormalized, possibly in a different DB).

  Command (write side)                Query (read side)
  ────────────────────                ───────────────────
  PlaceOrderCommand                   GetOrderHistoryQuery
       │                                       │
       ▼                                       ▼
  PlaceOrderHandler                   GetOrderHistoryHandler
       │                                       │
       ▼                                       ▼

       │                                       ▲
       ▼                                       │
  Postgres (write store) ───── projector ─────►│
                                          Read store
                                          (Postgres view, Mongo,
                                           Elasticsearch, Redis...)

  Why: read patterns ≠ write patterns.
  • Writes need invariants, validation, transactions
  • Reads need speed, denormalization, aggregations
  Optimize each side independently.
CQRS — separate models, separate paths, optimized independently

5.5 Event Sourcing

6.3 Indexing

An index is a data structure (typically a B-tree) that lets the DB find rows without scanning every row. The cost: writes get slower (each insert updates the index too), and indexes take disk space. The benefit: reads can go from seconds to microseconds.

  Without index                           With index on (email)
  ──────────────                          ──────────────────────
  SELECT * FROM users                     SELECT * FROM users
  WHERE email = 'a@b.com'                 WHERE email = 'a@b.com'
  
  → SCAN entire table                     → B-tree lookup (log n)
    1M rows, ~500ms                          1M rows, <1ms
  
  Index types:
  ┌─────────────────────────────────────────────────────────────┐
  │ B-tree    — default; great for =, <, >, BETWEEN, ORDER BY  │
  │ Hash      — only =; faster but no range queries            │
  │ GIN       — JSONB, arrays, full-text (Postgres)            │
  │ GiST      — geo, ranges, custom types                      │
  │ BRIN      — huge tables with natural ordering (time series)│
  │ Composite — index on (col_a, col_b) — order matters!       │
  │ Partial   — WHERE active = true                            │
  │ Covering  — INCLUDE extra cols to satisfy query from index │
  └─────────────────────────────────────────────────────────────┘

  Rule of thumb: index columns used in WHERE, JOIN, ORDER BY.
  Don't over-index — every index slows writes and uses disk.
Indexes — trade write speed and disk for fast reads

6.4 Replication

Maintain copies of your data on multiple machines. The most common pattern: one primary handles writes, multiple replicas handle reads. Reads scale horizontally; failover lets you survive primary loss.

  Primary-Replica (most common)
  ─────────────────────────────────
  
       Writes                   Reads
         │                        │
         ▼                  ┌─────┼─────┐
  ┌─────────────┐           ▼     ▼     ▼
  │   PRIMARY   │ ── replication stream (WAL/binlog) ──►
  │  (writeable)│ ──►  Replica 1   Replica 2   Replica 3
  └─────────────┘                              (read-only)
  
  Replication modes:
  • Sync     — primary waits for replicas to ACK. Safer, slower.
  • Async    — primary doesn't wait. Faster, may lose recent writes.
  • Semi-sync — wait for at least one replica.
  
  Multi-primary (writeable replicas):
  • CockroachDB, Cassandra, Yugabyte — eventual or tunable consistency
  • Powerful but conflicts must be resolved (last-write-wins, vector clocks, CRDTs)
Replication — scale reads, survive failure, replication lag is the catch

6.5 Sharding (Horizontal Partitioning)

Split data across multiple database servers, each holding a subset. Necessary when one server can't hold or serve all the data. Brings real complexity — cross-shard queries, rebalancing, hot shards.

  Single DB                              Sharded DB
  ──────────                             ────────────
  All users on one server                Shard 1: users with id % 4 == 0
                                         Shard 2: users with id % 4 == 1
  ┌────────────┐                         Shard 3: users with id % 4 == 2
  │   100M     │                         Shard 4: users with id % 4 == 3
  │   users    │                         
  └────────────┘                         ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  500GB, slow                            │ 25M  │ │ 25M  │ │ 25M  │ │ 25M  │
                                         └──────┘ └──────┘ └──────┘ └──────┘
                                         Each shard fast.
  
  Sharding strategies:
  • Hash-based   — hash(key) % N. Even distribution. Hard to add shards.
  • Range-based  — id 0-1M shard 1, id 1M-2M shard 2. Hot shards risk.
  • Geo-based    — EU users in EU shard. Compliance + latency win.
  • Consistent hashing — minimize re-shuffling when adding/removing nodes.
  
  Trade-offs: cross-shard joins are painful, rebalancing is hard.
  Tools: Vitess (MySQL), Citus (Postgres), MongoDB sharding, CockroachDB (auto).
Sharding — split data across servers when one can't hold it all

6.6 The N+1 Query Problem

The most common ORM performance trap. Loading a list of N items, then triggering one extra query per item for related data. 1 query becomes N+1 queries.

N+1 — and how to fix it
// ❌ N+1: 1 query for users, N queries for each user's posts
const users = await User.findAll()         // 1 query
for (const u of users) {
  const posts = await u.getPosts()         // 1 query EACH (N queries)
}
// 100 users? 101 queries.

// ✅ Eager loading: 2 queries total (or 1 with JOIN)
const users = await User.findAll({
  include: [{ model: Post }]
})
// Or in Prisma:
const users = await prisma.user.findMany({ include: { posts: true } })

// For GraphQL: use DataLoader to batch & cache within a request
const userLoader = new DataLoader(ids => User.findAll({ where: { id: ids }}))

6.7 Connection Pooling

Opening a new DB connection per request is slow (~100ms TCP+auth). A connection pool keeps a set of warm connections ready to serve requests. Critical for serverless — without pooling (e.g., PgBouncer, Prisma Data Proxy, Supabase pooler), your DB will run out of connections fast.

6.8 Database Migrations

  Migration files (versioned, in git):
  ─────────────────────────────────────
  migrations/
  ├── 001_create_users.sql
  ├── 002_add_email_index.sql
  ├── 003_split_name_columns.sql
  └── 004_drop_legacy_field.sql
  
  Tools: Prisma Migrate, Knex, Flyway, Alembic, Liquibase, Atlas, Sqitch

  ┌─────────────────────────────────────────────────────────┐
  │ 1. Add new column (nullable, default null)              │
  │ 2. Deploy app that writes BOTH old and new columns      │
  │ 3. Backfill old data into new column                    │

  │ 5. Stop writing to old column                            │
  │ 6. Drop old column (next deploy)                         │
  └─────────────────────────────────────────────────────────┘

Migrations — versioned, reversible, deployed step-by-step for zero downtime

7. Caching Strategies

"There are only two hard things in computer science: cache invalidation and naming things." — Phil Karlton. Caching is the highest-leverage optimization in backends. Used right, it cuts database load by 95%. Used wrong, it serves stale data forever.

7.1 The Five Caching Patterns

Cache-Aside (Lazy Loading)

  App reads:
  ──────────
  1. Check cache for key
  2. HIT  → return cached value
  3. MISS → query DB → write to cache → return
  
  App writes:
  ──────────
  1. Write to DB
  2. Invalidate (or update) cache
  
  Most common pattern. Used by Redis, Memcached at most companies.
  Simple, but: cache misses are expensive (extra round trip).
Cache-aside — app manages cache explicitly, lazy population

Read-Through

The cache layer itself fetches from DB on miss. App talks only to the cache. Used by Redis with modules, DataLoader-style libraries.

Write-Through

Writes go through the cache to the DB synchronously. Cache and DB are always in sync. Slow writes, but cache is always fresh.

Write-Behind (Write-Back)

Writes go to the cache; cache flushes to DB asynchronously. Very fast writes, but risky — if the cache crashes before flushing, writes are lost. Used in low-stakes scenarios (analytics counters, etc.).

Refresh-Ahead

The cache predicts which keys are about to expire and proactively re-fetches them before they go stale. Useful for hot keys with predictable access patterns.

7.2 Cache Invalidation Strategies

  TTL (Time-to-live)                  Event-based
  ──────────────────                  ──────────────
  SET user:42 {...} EX 300            On user update:
  Auto-expires after 5 min.           DEL user:42
  
  Simple. Eventually consistent.       Complex. Strongly consistent.
  Stale window = TTL.                  Must remember every key affected.
  
  Tagging                             Versioning
  ──────────                          ──────────
  Tag: "user:42"                      Cache key includes version:
  SET user:42:profile  ...            user:42:v3 → updates create v4 key
  SET user:42:settings ...            Old keys age out via TTL.
  Invalidate all "user:42" tags.      No explicit deletes needed.
                                      Used heavily by Vercel, Netlify.
Invalidation — TTL is simple; events/tags are precise; versioning is bulletproof

7.3 Multi-Level (L1/L2) Caching

  Request flow:
  ─────────────
  Client → CDN (L1: edge cache, milliseconds)
            └─ MISS → Application
                       └─ In-memory cache (L2: same-process, microseconds)
                          └─ MISS → Redis (L3: shared cache, ~1ms)
                                     └─ MISS → Database (~10-100ms)
  
  Each layer absorbs different traffic patterns.
  Static assets:  CDN handles 99.9%
  API responses:  Redis handles 95%, DB sees 5%
  Computed data:  In-process cache (LRU) handles repeats within seconds
  
  Tools:
  L1 (edge):     Cloudflare, Fastly, CloudFront, Vercel
  L2 (in-proc):  node-cache, lru-cache, Caffeine (Java)
  L3 (shared):   Redis, Memcached, Hazelcast, AWS ElastiCache
Multi-level caching — each layer absorbs traffic before reaching the DB

7.4 Stale-While-Revalidate

Serve cached data immediately even if it's stale; refresh in the background. The user always gets a fast response; the cache stays warm.

  Time t=0     CACHE: fresh "Apple Watch $399"
  Time t=300   CACHE: stale (TTL elapsed)
  Time t=301   USER REQ → return stale value INSTANTLY ✅
                          ↓
                       trigger background refresh
                          ↓
                       UPDATE CACHE with fresh value
  Time t=302   USER REQ → return now-fresh value
SWR — instant response, freshness in background

8. Async & Messaging

The full treatment lives in the dedicated Async Architecture A→Z guide (queues, jobs, DLQs, idempotency, events, sagas). Here's the executive summary.

8.1 Sync vs Async Calls

  Synchronous                            Asynchronous (queue)
  ──────────────                          ─────────────────────
  Service A ──HTTP──► Service B          Service A ──msg──► Queue ──► Service B
  
  A waits for B's response.              A drops a message and moves on.
  If B is down → A fails.                B picks it up when ready.
  If B is slow → A is slow.              If B is down → messages buffer.
  
  Use when:                              Use when:
  • You need the result NOW              • Result not needed immediately
  • Simple request/response              • B might be slow or unreliable
  • Latency matters                       • Many consumers (pub/sub)
                                          • Burst smoothing
Sync = "do it now and tell me"; Async = "fire and forget"

8.2 Queues vs Pub/Sub

  Queue (each msg → ONE consumer)           Pub/Sub (each msg → ALL consumers)
  ──────────────────────────────           ───────────────────────────────────
  Producer ──► [msg, msg, msg] ─►           Producer ──► Topic ──┬─► Consumer A
                  Worker (one of many)                            ├─► Consumer B
                                                                  └─► Consumer C
  Used for: "do this work"                  Used for: "this happened"
  E.g., send email, process video           E.g., OrderPlaced → email, analytics, fraud
Queue (work) vs Pub/Sub (notification)

8.3 Saga Pattern (Distributed Transactions)

You can't run a database transaction across services. The saga pattern coordinates a multi-step business process via events, with compensating actions for rollback. Two flavors: choreography (services react to events) and orchestration (a central coordinator drives the steps).

  Place order saga:
  ─────────────────
  Step 1: Reserve inventory   ✓
  Step 2: Charge payment       ✓
  Step 3: Create shipment      ✗  ← failure!
  
  Compensation (run in reverse):
  ── refund payment      ✓
  ── release inventory   ✓
  
  Each service publishes events; saga coordinator (or other services)
  trigger compensations on failure. NEVER rely on distributed XA transactions.
Saga — multi-step distributed flow with compensating actions on failure

8.4 Outbox Pattern

Solves the "did the DB update and the event publish?" problem. Insert the event into an outbox table in the same DB transaction as your business write. A separate poller reads from the outbox and publishes events. Atomic, exactly-once semantics.

9. Performance & Scalability

9.1 Vertical vs Horizontal Scaling

  Vertical (scale up)                    Horizontal (scale out)
  ────────────────────                    ──────────────────────
  Bigger machine                         More machines
  
  ┌────┐         ┌──────────┐           ┌────┐ ┌────┐ ┌────┐ ┌────┐
  │ 4   │  →     │  16      │           │    │ │    │ │    │ │    │
  │ CPU │        │  CPU     │           │    │ │    │ │    │ │    │
  │ 8GB │        │  64GB    │           └────┘ └────┘ └────┘ └────┘
  └────┘         └──────────┘            (load balanced)
  
  Pros: simple, no app changes.          Pros: ∞ scale, fault tolerance.
  Cons: ceiling exists, expensive,       Cons: app must be stateless,
        single point of failure.               distributed concerns.
Vertical = bigger box; Horizontal = more boxes

9.2 Load Balancing

Distribute incoming requests across multiple instances. Algorithms:

  ┌──────────────────────────────────────────────────────────────────┐
  │ Round Robin     — alternate between servers in order             │
  │ Least Connections — pick server with fewest active connections   │
  │ Least Response Time — pick fastest server                        │
  │ IP Hash         — same client always goes to same server (sticky)│
  │ Random          — uniform random pick                            │
  │ Weighted        — bigger servers get more traffic                │
  │ Power of Two Choices — pick 2 random, choose less loaded         │
  └──────────────────────────────────────────────────────────────────┘
  
  Layer 4 LB (TCP):    HAProxy, AWS NLB, Envoy
  Layer 7 LB (HTTP):   Nginx, AWS ALB, Envoy, Caddy, Traefik
  Service mesh:        Istio, Linkerd (built-in LB per-service)
Load balancing — picking the right algorithm matters at scale

9.3 Rate Limiting

Cap how many requests a client can make in a window. Protects the service from abuse and runaway clients.

  ┌──────────────────────────────────────────────────────────────────┐
  │ Algorithms                                                       │
  │ ─────────                                                        │
  │ Fixed Window    — count per minute window. Simple. Bursty.       │
  │ Sliding Window  — smoother. More accurate.                       │
  │ Token Bucket    — refill tokens at rate; each request costs one. │
  │ Leaky Bucket    — fixed-rate processor; queue overflows = drop.  │
  └──────────────────────────────────────────────────────────────────┘
  
  Where to apply:
  • Edge (CDN/API Gateway)  — first line of defense
  • App level (per user)    — fairness across customers
  • DB connection pool      — prevent overload
  
  Tools: Cloudflare, Kong, NGINX limit_req, AWS WAF, express-rate-limit, 
         Upstash, Redis (sliding window via sorted sets)
Rate limiting — protect the service, isolate noisy neighbors

9.4 Async I/O & Event Loops

Non-blocking I/O lets a single thread handle thousands of concurrent connections. Each request doesn't get its own thread; it gets a slice of time. This is why Node.js (single-threaded event loop) can serve thousands of requests per second.

9.5 Connection Pooling

Already mentioned in §6.7 — but worth repeating: opening a DB connection per request is the #1 cause of slow APIs. Always pool. For serverless, use an external pooler (PgBouncer, Supabase Pooler, Neon, Prisma Accelerate).

9.6 CDN & Edge Caching

Push static and cacheable dynamic content to edge locations near users. Cuts global latency from hundreds of ms to tens of ms. Modern CDNs (Cloudflare, Fastly, Vercel) cache HTML, JSON, and even render at the edge.

10. Resilience Patterns

Networks fail. Services restart. Disks fill up. Resilience patterns let your system degrade gracefully instead of catastrophically.

10.1 Retry with Exponential Backoff

  Naive retry                            Exponential backoff + jitter
  ────────────                            ──────────────────────────────
  on fail:                               on fail:
    sleep 1s; retry                        attempt N: sleep (base * 2^N) + random
    sleep 1s; retry                        attempt 1: sleep ~1s
    sleep 1s; retry                        attempt 2: sleep ~2s
                                          attempt 3: sleep ~4s
  Problem: thundering herd —              attempt 4: sleep ~8s
  many clients retry simultaneously       
  and DOS the recovering service.        Jitter (randomization) spreads
                                          retries so they don't pile up.
                                          
  Always set a max retry count + max total time.
  Only retry idempotent operations or use idempotency keys.
Exponential backoff with jitter — the right way to retry

10.2 Circuit Breaker

If a downstream service is failing, stop calling it for a while. Like an electrical breaker — trip the circuit, save the system. Three states: Closed (calls pass through), Open (calls fail fast without trying), Half-Open (try a few requests to see if service recovered).

  Circuit Breaker State Machine:
  ──────────────────────────────────────────────────────
  
                    failure rate > threshold
       CLOSED ─────────────────────────────────► OPEN
         ▲                                          │
         │ success                                  │ wait timeout
         │                                          ▼
         └──────────────────────── HALF-OPEN ───────┘
                  failure
  
  CLOSED:    Normal. Requests go through. Track failure rate.
  OPEN:      Don't even try. Return cached/default/error fast.
             Skip the dead service entirely.
  HALF-OPEN: Send a few "probe" requests. If they succeed → CLOSED.
             If they fail → back to OPEN.
  
  Tools: Hystrix (Netflix, deprecated), Resilience4j (Java), 
         Polly (.NET), opossum (Node), Istio (mesh-level)
Circuit breaker — fail fast when downstream is dying

10.3 Bulkhead

Like watertight compartments on a ship — if one compartment floods, others stay dry. Isolate resources (thread pools, connection pools) per dependency. If one downstream service is slow, it can't exhaust all your threads.

  Without bulkheads                       With bulkheads
  ──────────────────                      ────────────────
  ┌─────────────────────┐                 ┌─────────────────────┐
  │  Thread pool: 100    │                 │ Pool A (auth):  20   │
  │                      │                 │ Pool B (search): 30  │
  │  Auth slow → all     │                 │ Pool C (search): 30  │
  │  threads blocked     │                 │ Pool D (other):  20  │
  │  → entire app frozen │                 │                      │
  └─────────────────────┘                 │ Auth slow → only Pool│
                                          │ A blocked. Search still│
                                          │ works.                 │
                                          └─────────────────────┘
Bulkhead — isolate failures, prevent system-wide crashes

10.4 Timeout

The single most important resilience pattern. Every external call must have a timeout. No exceptions. Without timeouts, a slow downstream blocks threads forever, then queue depth grows, then the whole app dies.

10.5 Fallback & Graceful Degradation

  Recommendation Service down?           Fallback chain:
  ─────────────────────────              ─────────────────
  ❌ Show error to user                  Try personalized recs from ML service
                                          ↓ fail
  ✅ Show "trending products"             Try recently viewed from cache
     instead.                             ↓ fail
                                          Try generic "popular this week"
                                          ↓ fail
                                          Show empty state ("Browse categories")
  
  Each fallback is cheaper and more reliable than the previous.
  Users get something — never nothing.
Graceful degradation — give users a worse experience instead of no experience

10.6 Health Checks

  Two kinds of health checks:
  ────────────────────────────
  
  /health/live (Liveness)              /health/ready (Readiness)
  ─────────────────────                ────────────────────────
  "Am I alive?"                         "Am I ready to serve?"
  
  Returns 200 if process is running.   Returns 200 if dependencies (DB, cache,
  Returns 5xx → orchestrator restarts. queues) are reachable AND I'm warmed up.
                                        Returns 5xx → load balancer skips me.
  
  Used by:                              Used by:
  Kubernetes liveness probe             Kubernetes readiness probe
  ECS health check                      Load balancers
                                        Service mesh
  
  Don't make liveness check too strict! "DB unreachable" should fail readiness,
  not liveness — restarting won't help if DB is down.
Liveness (am I alive) vs Readiness (can I serve traffic)

12.4 SLI / SLO / SLA + Error Budgets

  SLI — Service Level INDICATOR
  ──────────────────────────────
  A measurable quality of service.

  SLO — Service Level OBJECTIVE
  ──────────────────────────────
  Target value for an SLI you commit to internally.
  E.g., "99.9% of requests successful in any 30-day window."
  
  SLA — Service Level AGREEMENT
  ──────────────────────────────

  ────────────

13. Security Patterns

Security is architectural, not bolt-on. The bigger the bolt-ons, the bigger the holes.

13.1 Authentication Patterns

  ┌──────────────────────────────────────────────────────────────────┐
  │ Sessions (server-stateful)                                       │
  │ ────────                                                         │
  │ Server creates session, stores in DB/Redis, returns cookie.      │
  │ Pros: revocable, simple, secure (HTTP-only cookies).             │
  │ Cons: state on server (Redis dependency), CSRF protection needed.│
  │ Best for: traditional web apps.                                  │
  ├──────────────────────────────────────────────────────────────────┤
  │ JWT (stateless tokens)                                           │
  │ ───                                                              │
  │ Server signs a payload, gives it to client.                      │
  │ Client sends it back; server verifies signature.                 │
  │ Pros: stateless, easy across services, no DB lookup.             │
  │ Cons: hard to revoke, payload visible (don't put secrets!),      │
  │       refresh token logic needed.                                │
  │ Best for: APIs, microservices, mobile.                           │
  ├──────────────────────────────────────────────────────────────────┤
  │ OAuth 2.0 / OIDC                                                 │
  │ ──────────                                                       │
  │ Delegated authentication via third party (Google, GitHub, etc.). │
  │ "Sign in with Google" → Google issues a token → your app trusts  │
  │ the token (after verifying it).                                  │
  │ OIDC adds identity (who) on top of OAuth (what they can do).     │
  │ Best for: anything user-facing.                                  │
  ├──────────────────────────────────────────────────────────────────┤
  │ Passkeys (WebAuthn)                                              │
  │ ────────                                                         │
  │ Cryptographic key pairs stored in device secure enclaves.        │
  │ No password ever transmitted. Phishing-proof. Biometrics or PIN. │
  │ Becoming the default for new auth flows in 2025-2026.            │
  ├──────────────────────────────────────────────────────────────────┤
  │ API keys / Tokens                                                │
  │ ────────                                                         │
  │ Static long-lived secret. For server-to-server or developer use. │
  │ Always scope (read-only vs write), rotate, store hashed.         │
  └──────────────────────────────────────────────────────────────────┘
Auth options — pick by trust model and threat profile

13.2 Authorization Models

  RBAC — Role-Based Access Control
  ─────────────────────────────────
  User has roles. Roles have permissions.
    User Alice → role "editor" → can [edit, publish, view]
  Simple. Coarse.
  
  ABAC — Attribute-Based Access Control
  ─────────────────────────────────────
  Decision based on attributes of user, resource, action, context.
    "Alice can edit Document 42 IF Alice.dept == Document.dept
     AND now() between 9am-5pm AND Alice.location == 'office'"
  Powerful. Complex.
  
  ReBAC — Relationship-Based Access Control
  ─────────────────────────────────────────
  Decisions based on relationships in a graph.
    Alice owns Folder X. X contains Doc Y. Therefore Alice can edit Y.
  Models permissions like Google Drive, GitHub, Notion.
  Tools: SpiceDB, OpenFGA (Auth0), Cerbos.
  
  ACL — Access Control List
  ─────────────────────────
  Per-resource list of who can do what. Old-school. Doesn't scale.
RBAC vs ABAC vs ReBAC — pick based on permission complexity

13.5 Rate Limiting & DDoS

Layers of defense: CDN/edge (block volumetric attacks before they reach you), API gateway (per-API-key limits), application (per-user limits), DB (connection limits). Cloudflare, AWS Shield, Akamai handle volumetric DDoS at the edge for ~free.

13.6 Input Validation

Validate at the boundary. Schema-based validation (Zod, Yup, class-validator, Pydantic, Joi) is now the norm. Reject early, reject hard — the deeper bad data flows, the harder it is to undo damage.

14. Deployment Patterns

14.1 Blue-Green Deployment

  Two identical environments. Switch traffic atomically.
  
  Time t=0          Blue (LIVE)  ◄── 100% traffic
                    Green        ◄── 0% (idle)
  
  Deploy v2          Blue         ◄── 100% traffic
                    Green        ── deploy v2 here
                    Smoke test   ── verify Green
  
  Switch traffic    Blue         ── 0% (now standby for rollback)
                    Green (LIVE) ◄── 100% traffic
  
  Rollback?         Switch back to Blue. Instant.
  
  Pros: zero downtime, instant rollback.
  Cons: 2x infrastructure cost during deploys, DB migrations tricky.
Blue-green — full duplicate environment, atomic switch, instant rollback

14.2 Canary Deployment

  Gradually shift traffic to new version. Monitor at each step.
  
  ──────────────────────────────────────────────────────────────────
  Step 1:  v1 ████████████████████████████████████████████ 99%
           v2 █                                              1%
           Watch: errors? latency? business metrics?
  ──────────────────────────────────────────────────────────────────
  Step 2:  v1 ███████████████████████████████████████ 90%
           v2 ████                                     10%
  ──────────────────────────────────────────────────────────────────
  Step 3:  v1 █████████████████ 50%
           v2 █████████████████ 50%
  ──────────────────────────────────────────────────────────────────
  Step 4:  v2 100% (full rollout)
  
  Tools: Argo Rollouts, Flagger, Spinnaker, manual via service mesh.
  Auto-rollback if error rate spikes.
Canary — gradual traffic shift, automated rollback if metrics degrade

14.3 Rolling Deployment

Replace instances one at a time (or in small batches) until all run the new version. The Kubernetes default. Slower than blue-green, less infrastructure cost.

14.4 Feature Flags

  Deploy ≠ Release.
  
  Code is deployed → behind a flag → flag controls who sees it.
  
  if (flags.isEnabled('new-checkout', { userId, plan })) {
    return newCheckoutFlow()
  }
  return oldCheckoutFlow()
  
  Use cases:
  ┌──────────────────────────────────────────────────────────────┐
  │ Trunk-based development — merge to main, hide behind flag    │
  │ Kill switch — disable features instantly without deploy      │
  │ A/B testing — % rollout, measure impact                      │
  │ Gradual rollout — 1% → 10% → 50% → 100%                      │
  │ Permission tier — feature only for paid users                │
  │ Beta program — opt-in users only                             │
  └──────────────────────────────────────────────────────────────┘
  
  Tools: LaunchDarkly, Unleash, Flagsmith, ConfigCat, GrowthBook,
         Vercel Edge Config, OpenFeature (open standard)
Feature flags — decouple deploy from release

14.5 Zero-Downtime Database Migrations

Already covered in §6.8 — but the pattern bears repeating because it's the #1 cause of deployment outages: never make a breaking schema change in one deploy. Always: add nullable → backfill → switch reads → stop writes to old → drop, across multiple deploys.

  Before going to production, verify:
  ─────────────────────────────────────
  
  ☐ Every external call has a timeout
  ☐ Every external call has retry logic (idempotent + backoff)
  ☐ Every endpoint has authentication
  ☐ Every endpoint has authorization
  ☐ Every endpoint has input validation
  ☐ Every endpoint has rate limiting
  ☐ Every DB query is indexed appropriately
  ☐ Every DB write is in a transaction (where it matters)
  ☐ Every secret is in a secret manager (not env vars on disk)
  ☐ Every log is structured (JSON)
  ☐ Every log line has requestId / traceId
  ☐ Liveness + readiness health checks exist
  ☐ Graceful shutdown (SIGTERM → drain → exit)
  ☐ Connection pooling configured
  ☐ Backups + restore procedure tested
  ☐ Monitoring alerts on errors, latency, saturation
  ☐ Migrations are zero-downtime (additive)
  ☐ Idempotency keys on critical mutations
  ☐ DLQ / poison queue monitored
  ☐ Disaster recovery plan documented

Final Principle: Backend architecture is about trade-offs, not best practices. The "best" architecture is the simplest one that handles your current requirements while leaving room to evolve. Start with a modular monolith on Postgres. Introduce microservices, event sourcing, sharding, edge compute, and service meshes only when you have evidence you need them. Premature architecture will sink a startup faster than premature optimization.
