STACK MAP
Frontend UI Library
React
A JS library for building UIs as a tree of components. You describe what the UI should look like given current data — React figures out how to update the DOM efficiently.
Made by Meta Problem it solves: Manual DOM manipulation hell Core idea: UI = f(state)
Problem it solves
Without React, you'd manually do document.querySelector(), update DOM nodes, track state in global variables, and sync the UI yourself every time data changes. With 10+ interactive elements it becomes spaghetti fast. React makes you just describe the UI and it handles the DOM updates.
Alternatives & how they differ
Vue
Same component idea, but uses HTML templates instead of JSX. More beginner friendly syntax.
Similar concept
Svelte
No virtual DOM. Compiles your components to vanilla JS at build time. Faster runtime, smaller bundle.
No runtime
Angular
Full framework (routing, forms, HTTP all built in). Much more opinionated, uses TypeScript by default.
Full framework
Solid.js
React-like syntax but no virtual DOM, fine-grained reactivity. Very fast.
Fine-grained
It's just… (Framework → JS/HTML primitive)
JSX / Component → JS function returning an object
JSX is not HTML. Babel compiles it to React.createElement() which returns a plain JS object — a "virtual DOM node".
// What you write
const Button = ({ label }) => (
  <button className="btn">{label}</button>
)

// What Babel compiles it to
const Button = ({ label }) =>
  React.createElement('button', { className: 'btn' }, label)

// What createElement actually returns — just a JS object
{ type: 'button', props: { className: 'btn', children: label } }
Props → Function arguments (plain JS object)
Props are literally just the first argument of your component function. When you write <Button label="Click" />, React calls Button({ label: "Click" }).
// Calling a component
<Button label="Click" disabled={true} />

// React literally does this under the hood
Button({ label: "Click", disabled: true })
useState → JS variable + setter that triggers re-render
useState is just a way to have a variable that, when changed, tells React "re-run this function (component)". The value persists between renders via React's internal storage (not a regular variable).
const [count, setCount] = useState(0)
// count  → current value (read-only snapshot)
// setCount → calling this re-renders the component
// useState(0) → returns [currentValue, setterFn]

// Equivalent mental model in vanilla JS (simplified):
let count = 0
const setCount = (val) => { count = val; rerenderComponent() }
useEffect → "Run this JS code after render"
useEffect is just a way to run side effects (fetching, subscriptions, timers) after the component renders. The dependency array controls when it re-runs.
useEffect(() => {
  // runs after every render
})

useEffect(() => {
  // runs only once — on mount (like componentDidMount)
}, [])

useEffect(() => {
  // runs when 'userId' changes
}, [userId])

useEffect(() => {
  const timer = setInterval(fn, 1000)
  return () => clearInterval(timer) // cleanup on unmount
}, [])
useRef → { current: value } — a box that persists
A ref is literally just an object { current: something } that persists across renders without triggering re-renders. Use it to hold DOM nodes or mutable values.
const inputRef = useRef(null)
// inputRef is just: { current: null }
// After <input ref={inputRef} />, React sets:
// inputRef.current = <the actual DOM input element>

inputRef.current.focus() // direct DOM access, plain JS
Context / Provider → Global JS object passed down the tree
Context is just a way to pass a JS value to any descendant component without manually passing it as props at every level.
const ThemeCtx = React.createContext('light')
// Creates: { Provider, Consumer, _currentValue }

// Provider just puts the value in scope
<ThemeCtx.Provider value="dark">
  <App />
</ThemeCtx.Provider>

// Any child can grab it
const theme = useContext(ThemeCtx) // → "dark"
Custom Hook → Plain JS function that uses other hooks
A custom hook is literally just a function that starts with "use". That's the only rule. It can call useState, useEffect, etc. and returns whatever you want.
// Custom hook = plain JS function
function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth)
  useEffect(() => {
    const handler = () => setWidth(window.innerWidth)
    window.addEventListener('resize', handler)
    return () => window.removeEventListener('resize', handler)
  }, [])
  return width // returns plain JS value
}
Essential Concepts A–Z
Conditional Renderingif/else inside JSX▾
JS Ternary / &&
JSX is JS, so you use && for "show if true" and ? : for if/else. {isLoggedIn && <Dashboard />} is just true && value in JS.
Lists & Keysarray.map() in JSX▾
Array.map()
Rendering a list is just array.map(item => <Li>{item}</Li>). The key prop is just a string/number React uses internally to track which item is which — it's not in the DOM.
Event HandlersJS functions on props▾
JS Function Reference
onClick={handleClick} is just passing a JS function as a prop. React attaches it as an event listener on the DOM node. It's element.addEventListener('click', handleClick).
Lifting State UpMove state to common parent▾
JS Closure / Props
When two sibling components need to share state, move the useState to their parent and pass it down as props. The children communicate upward via callback props (plain JS functions passed down).
useMemo / useCallbackMemoization — avoid re-computing▾
JS Memoization pattern
useMemo caches a computed value. useCallback caches a function reference. Both only recompute when dependencies change. It's the same as manually caching: if deps haven't changed, return cached result.
useReducerComplex state logic▾
JS Switch/Case + Object
Like useState but for complex state. You dispatch actions (plain objects like {type: 'increment'}) and a reducer function (switch/case) returns the new state. It's the Redux pattern in miniature.
PortalsRender outside parent DOM▾
DOM appendChild to different node
ReactDOM.createPortal lets you render a component into a different DOM node (like document.body) regardless of where it is in the React tree. Used for modals, tooltips.
Error Boundariestry/catch for components▾
try/catch equivalent
A class component that wraps children and catches JS errors in the render tree. If a child throws, the error boundary shows a fallback UI instead of crashing the whole app.
React Framework
Next.js
A framework built on top of React that adds routing, server-side rendering, static generation, API routes, and project conventions. React is the engine — Next.js is the car.
Made by Vercel Problem: React has no router, no SSR, no conventions Core idea: Folder = Route
Problem it solves
React alone gives you a blank canvas — no routing, no server rendering, no structure. You'd have to wire up React Router, configure Webpack/Vite, set up SSR yourself. Next.js gives you all of that out of the box with zero config. Also solves SEO — search engines struggle with client-rendered React. Next.js renders HTML on the server so crawlers see real content.
Alternatives & how they differ
Remix
More web-standards focused. Leans on native browser APIs (form, fetch). Better error handling model.
Web standards first
Gatsby
Static-site focused. Pulls data from any source at build time. Not great for dynamic apps.
Static-first
Astro
Ships zero JS by default. Works with React, Vue, Svelte. Best for content sites.
Island architecture
Vite + React
Just a fast build tool + React. No routing, no SSR. SPA only. You wire everything yourself.
Manual setup
It's just… (Framework → JS/HTML primitive)
File-based Routing (App Router) → Folder structure → URL mapping convention
Next.js looks at your folder structure and builds routes from it. No routing config file needed. The folder name IS the URL segment.
app/
  page.tsx          → /
  dashboard/
    page.tsx        → /dashboard
  users/
    [id]/
      page.tsx      → /users/123  (id = "123")
  (auth)/           → group folder, doesn't affect URL
    login/page.tsx  → /login
Server Component (default in App Router) → Async JS function that runs on server, returns HTML
By default, every component in the App Router is a Server Component — it runs on the server, can await data directly, and ships only HTML to the client. No JS bundle sent for these.
// This runs on the SERVER — no useEffect needed
async function UserProfile({ id }) {
  const user = await db.user.findById(id) // direct DB call!
  return <div>{user.name}</div> // renders to HTML string
}
// Client gets: <div>John Doe</div> — no JS, just HTML
"use client" → Boundary marker: "send this JS to the browser"
'use client' at the top of a file tells Next.js this component and its children run in the browser. Required whenever you use useState, useEffect, event handlers, or browser APIs.
'use client' // ← this file runs in browser

import { useState } from 'react'

export function Counter() {
  const [n, setN] = useState(0)
  return <button onClick={() => setN(n+1)}>{n}</button>
}
layout.tsx → React component wrapping {children}
A layout is just a component that wraps child pages. It persists across route changes — the layout doesn't re-mount when you navigate between pages in the same segment.
export default function DashboardLayout({ children }) {
  return (
    <div>
      <Sidebar />       // always rendered
      <main>{children}</main> // page content goes here
    </div>
  )
}
API Route (route.ts) → Exported async functions named GET, POST, etc.
API routes are just exported functions named after HTTP methods. Next.js maps incoming requests to the right function.
// app/api/users/route.ts
export async function GET(request) {
  const users = await db.getUsers()
  return Response.json(users) // native Web API
}

export async function POST(request) {
  const body = await request.json() // plain JS object
  return Response.json({ created: true }, { status: 201 })
}
Server Actions → Async server functions called from client like normal JS
'use server' marks a function to run on the server. You call it from a client component like any async function — Next.js handles the HTTP request behind the scenes.
'use server'
export async function createUser(formData) {
  await db.user.create({ name: formData.get('name') })
}

// Client component calls it like plain JS:
<button onClick={() => createUser(data)}>Save</button>
// Next.js auto-creates a POST endpoint for this
Essential Concepts A–Z
Dynamic Routes [id]URL params as folder name▾
URL Pattern Matching
A folder named [id] catches any URL segment. The value is passed as params.id to your page component — it's just a JS string from the URL.
MiddlewareRuns before every request▾
JS Function (req) => response
A function in middleware.ts at the root. Gets every request before it hits a route. Used for auth checks, redirects, A/B testing. Returns NextResponse to redirect/rewrite/continue.
Static vs Dynamic RenderingBuild time vs request time▾
HTML generation timing
Static = HTML built once at deploy time (fast, cacheable). Dynamic = HTML built on every request. Next.js decides automatically: if you use cookies(), headers(), or searchParams, it becomes dynamic.
Image ComponentOptimized <img> tag▾
HTML <img> + lazy load + WebP
Next's <Image> is just an <img> tag with automatic WebP conversion, lazy loading, and size optimization baked in. Always specify width/height or fill.
Link ComponentClient-side navigation▾
HTML <a> + pushState
Next's <Link> renders as a regular <a> tag but intercepts the click and does a client-side navigation using the History API — no full page reload.
loading.tsx / error.tsxAutomatic suspense/error UI▾
React Suspense + Error Boundary
Drop a loading.tsx next to a page and Next.js wraps that route in a React Suspense boundary automatically. error.tsx becomes an Error Boundary. Just convention-over-configuration wiring.
Server State Management
TanStack Query
A library that manages the lifecycle of async data — fetching, caching, background refetching, loading states, error states — so you don't have to manually handle all that with useState + useEffect.
Formerly React Query Problem: Async data lifecycle is repetitive and error-prone Core idea: Cache keyed by queryKey
Problem it solves
Every time you fetch data, you manually write: useState(null), useState(false), useState(null) for data/loading/error, then a useEffect to fetch, then cleanup. And you get no caching, no background refetching, no deduplication. TanStack Query gives you all that with one hook call.
Alternatives & how they differ
SWR
Vercel's answer. Simpler API, same core idea. Smaller feature set than TanStack Query.
Simpler
RTK Query
Redux Toolkit's built-in fetching. Best if you're already using Redux for everything.
Redux-coupled
Apollo Client
Same concept but built for GraphQL specifically. Normalised cache.
GraphQL-specific
Zustand + fetch
Manual — you manage async state yourself in a Zustand store. More control, more boilerplate.
Manual
It's just… (Framework → JS/HTML primitive)
useQuery → { data, isLoading, error } from a cached Promise
useQuery runs your async function, caches the result by queryKey, and gives you a reactive object with the current state. If another component requests the same key, it gets the cached value — no duplicate fetch.
const { data, isLoading, error } = useQuery({
  queryKey: ['users', userId],   // cache key — JS array
  queryFn: () => fetchUser(userId) // async fn returning data
})

// Is essentially this, done automatically:
const [data, setData] = useState(null)
const [isLoading, setLoading] = useState(true)
useEffect(() => { fetchUser(userId).then(setData) }, [userId])
useMutation → { mutate fn, isLoading, error } for write operations
useMutation wraps any async function that changes data (POST/PUT/DELETE). You call mutate() and get loading/error/success states back.
const { mutate, isPending } = useMutation({
  mutationFn: (newUser) => postUser(newUser),
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ['users'] })
    // ↑ tells the 'users' query: your cache is stale, refetch
  }
})

// Call it like a plain JS function:
mutate({ name: 'Alice', email: 'a@b.com' })
QueryClient & QueryClientProvider → Global JS Map (cache) passed via React Context
QueryClient is the cache store — a JS object holding all fetched data keyed by queryKey. The Provider passes it to all components via Context.
// QueryClient is essentially:
class QueryClient {
  cache = new Map() // queryKey → { data, status, updatedAt }
  invalidateQueries({ queryKey }) { /* mark stale → refetch */ }
  setQueryData(key, data) { /* manually set cache */ }
  getQueryData(key) { /* read cache */ }
}
queryKey → JS array used as a cache key (like Map key)
The queryKey is how TanStack Query identifies and deduplicates cache entries. Arrays are compared by value — same contents = same cache entry.
['users']               → all users
['users', 42]          → user with id 42
['users', 42, 'posts'] → posts of user 42

// invalidateQueries(['users']) invalidates ALL of the above
// because it matches the prefix — like a namespace
staleTime / gcTime → Numbers (milliseconds) controlling cache freshness
staleTime = how long (ms) data is considered fresh. During this window, no refetch happens. gcTime = how long unused cache data is kept in memory before garbage collected.
useQuery({
  queryKey: ['config'],
  queryFn: fetchConfig,
  staleTime: 5 * 60 * 1000,  // 5 min — don't refetch for 5 min
  gcTime: 10 * 60 * 1000,   // 10 min — keep in memory 10 min
})
Essential Concepts A–Z
Optimistic UpdatesUpdate UI before server confirms▾
Cache manipulation
Before the mutation resolves, you manually update the cache (setQueryData) to show the expected result immediately. If the server fails, you roll back using the previous cache value saved in onMutate.
Infinite QueriesPagination / infinite scroll▾
Array of pages (JS arrays)
useInfiniteQuery manages paginated data. Each page is a JS array. fetchNextPage() fetches the next page and appends it. data.pages is just an array of page results.
Query InvalidationMark cache as stale → refetch▾
Cache key lookup + refetch trigger
invalidateQueries tells TanStack Query "this cached data is outdated". Any component currently displaying that data will automatically refetch in the background.
enabled flagConditionally run the query▾
Boolean condition
enabled: !!userId — the query only runs when userId is truthy. It's just a boolean that controls whether the queryFn gets called. Used for dependent queries (query B depends on result of query A).
selectTransform/filter returned data▾
Array.map / Array.filter on cache data
select is a function applied to the cached data before returning it to your component. Like doing .map() or .filter() — the raw cache stays unchanged, you just get a transformed view.
Utility-First CSS Framework
Tailwind CSS
A CSS framework where instead of writing custom CSS, you compose pre-defined utility classes directly in your HTML/JSX. Each class maps to one (or a few) CSS properties.
Problem: Custom CSS is slow, inconsistent & hard to maintain Core idea: 1 class = 1 CSS rule
Problem it solves
Without Tailwind, you write class names, then jump to a CSS file to write the styles, then come back. You end up with hundreds of one-off class names that are hard to maintain. Tailwind keeps everything in one place — the class IS the style.
Alternatives & how they differ
Bootstrap
Pre-built components. Less flexible — everything looks "Bootstrap-y". Opinionated design.
Component-based
CSS Modules
Scoped CSS files per component. You still write CSS, but names are locally scoped.
Scoped CSS
styled-components
Write CSS in JS template literals. Dynamic styles tied to props. Runtime overhead.
CSS-in-JS
UnoCSS
Same utility concept but on-demand — generates only the CSS you use, faster build.
Same concept

Essential Concepts A–Z
cn() / clsx / cvaConditional class merging▾
JS String Concatenation
cn('base', isActive && 'active', variant === 'primary' && 'bg-blue') — just builds a class string conditionally. clsx/cn are tiny JS utilities. cva (Class Variance Authority) organises component variants cleanly.
tailwind.config.jsDesign system customization▾
Plain JS/TS export default object
The config file is just a JS object where you extend Tailwind's defaults — custom colors, fonts, spacing, breakpoints. Whatever you add becomes a new utility class.
@apply directiveUse Tailwind classes in CSS files▾
CSS macro expansion
In a .css file, @apply flex items-center p-4 expands to the full CSS those classes represent. Useful for extracting repeated class combinations into a named class.
JIT (Just-in-Time)Only generates CSS you actually use▾
Build-time CSS pruning
Tailwind scans your code at build time, finds every class name you use, and generates only those CSS rules. The final CSS bundle is tiny. This is why you can't build class names dynamically with string concatenation.
Component Development Environment
Storybook
A tool that lets you build and test UI components in isolation — outside of your actual app. Each "story" is just your component rendered with specific props.
Problem: Testing components requires running the whole app Core idea: Story = component + props snapshot
Problem it solves
Without Storybook, to test a button's disabled state you'd navigate to the right page, trigger the right condition, wait for the app to load. With Storybook, you just open the Button story in "disabled" state directly — instant. Great for design systems and component libraries.
It's just… (Framework → JS/HTML primitive)
Story (CSF format) → Exported JS object/function with props
A story is literally just an exported function or object that renders your component with specific props. Storybook renders it in an iframe. That's it.
// Button.stories.tsx
import { Button } from './Button'

// Default export: metadata about the component
export default {
  title: 'Components/Button',
  component: Button,
}

// Named exports: each one is a story (a state)
export const Primary = { args: { label: 'Click me', variant: 'primary' }}
export const Disabled = { args: { label: 'Click me', disabled: true }}
export const Loading  = { args: { label: 'Loading', loading: true }}

// Storybook just does: <Button {...story.args} />
Args → Plain JS object of props passed to component
Args are just the props you want to pass to your component in this story. Storybook auto-generates interactive controls (sliders, checkboxes, text inputs) from the arg types.
export const Story = {
  args: {
    label: 'Submit',     // → text input in Controls panel
    disabled: false,   // → checkbox in Controls panel
    size: 'md',         // → dropdown in Controls panel
  }
}
// Storybook reads the types and builds the UI controls automatically
Decorators → React wrapper components (Providers, padding, etc.)
Decorators wrap your stories with additional React components — like adding a ThemeProvider, or centering the component. Just React wrapper components in an array.
export default {
  decorators: [
    (Story) => (
      <ThemeProvider theme="dark">
        <div style={{ padding: '2rem' }}>
          <Story />  // your component renders here
        </div>
      </ThemeProvider>
    )
  ]
}
Essential Concepts A–Z
ControlsAuto-generated prop UI▾
HTML form inputs bound to args
Storybook reads your TypeScript prop types and auto-generates a panel of form controls (text, checkbox, select, number) to tweak args in real time. You don't write the controls — it infers them.
ActionsLogs event handler calls▾
Console.log for event props
When a story has an onClick/onChange etc., Storybook replaces it with a spy function that logs to the Actions panel. You can see "button was clicked with args: …" without writing any test code.
play functionInteraction testing▾
Async JS function simulating user interactions
A play() function on a story runs after render — you use Testing Library utilities to click, type, and assert. Basically an automated integration test that runs inside Storybook itself.
AddonsPlugins for extra panels/features▾
JS plugins registered in .storybook/main.ts
Addons add extra panels to the Storybook UI — accessibility checker (a11y), viewport switcher, design tokens viewer, etc. Registered as an array of npm package names in the config.
Backend Node.js Framework
NestJS
An opinionated Node.js framework that brings structure (modules, controllers, services) to Express/Fastify using TypeScript decorators. Think Angular, but for the backend.
Runs on Express or Fastify underneath Problem: Express has no structure conventions Core idea: Decorator-based dependency injection
It's just… (Framework → JS primitive)
Decorator (@Controller, @Get...) → Function that wraps a class/method and attaches metadata 
@Controller('users')
class UserController {}

// Is exactly equivalent to:
Controller('users')(UserController)
// Which just does: Reflect.metadata('path', 'users', UserController)
// Attaches metadata that Nest reads at bootstrap to map routes
Service / Injectable → Plain JS class with business logic 
@Injectable()
class UserService {
  async findAll() {
    return db.user.findMany() // just a plain async method
  }
}
// @Injectable() just marks it for Nest's DI container
// The class itself is plain JS — no magic inside it
Module → JS object grouping controllers + services 
@Module({
  controllers: [UserController],  // route handlers
  providers:   [UserService],     // injectable services
  imports:     [DatabaseModule],  // other modules
  exports:     [UserService],     // share with other modules
})
class UserModule {}
// It's just a manifest — tells Nest what belongs together
Guard → Function that returns true/false (allow or block) 
@Injectable()
class AuthGuard implements CanActivate {
  canActivate(context): boolean {
    const req = context.switchToHttp().getRequest()
    return validateToken(req.headers.authorization)
    // true → request proceeds, false → 403
  }
}
ORM / Database Toolkit
Prisma
A type-safe ORM that generates a JS client from your schema. Instead of writing SQL strings, you call JS methods on objects and get fully-typed results back.
Problem: Raw SQL has no type safety and is verbose Core idea: Schema → generated JS client
It's just… (Framework → JS primitive)
Prisma Client → JS object with async methods for each DB table 
const prisma = new PrismaClient()

// prisma.user → JS object for the "user" table
// Every method is just an async function returning plain JS objects

const user = await prisma.user.findUnique({ where: { id: 1 } })
// Returns: { id: 1, name: "Alice", email: "..." } — plain JS object

const users = await prisma.user.findMany({ where: { active: true } })
// Returns: plain JS array of objects
Schema (schema.prisma) → Type definitions → generates TS types + SQL migrations 
// schema.prisma
model User {
  id    Int    @id @default(autoincrement())
  name  String
  posts Post[]  // relation → nested JS object when queried
}

// Prisma generates from this:
// 1. SQL CREATE TABLE migration
// 2. TypeScript type: type User = { id: number, name: string }
// 3. prisma.user.findMany() with full type safety
Relations / include → Nested JS objects (JOIN under the hood) 
const user = await prisma.user.findUnique({
  where: { id: 1 },
  include: { posts: true }  // SQL JOIN
})

// Returns nested plain JS object:
{
  id: 1, name: "Alice",
  posts: [                   // just a JS array
    { id: 10, title: "Hello" }
  ]
}
Essential Concepts A–Z
MigrationsSchema change → SQL file▾
Generated SQL files
When you change schema.prisma and run prisma migrate dev, Prisma generates a SQL migration file and runs it. The migrations folder is just .sql files tracking your DB history.
select vs includeChoose fields vs join relations▾
SQL SELECT columns vs JOIN
select lets you pick specific columns (like SQL SELECT name, email). include does a JOIN to bring in related table data as nested objects. They can't be used together at the same level.
TransactionsMultiple operations atomically▾
SQL TRANSACTION
prisma.$transaction([op1, op2]) — if any operation fails, all are rolled back. You pass an array of Prisma operations and they all succeed or all fail together.
