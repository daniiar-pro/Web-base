01

Developer writes code

You open your editor and write in languages the browser cannot understand natively — TypeScript, JSX, SCSS. These are higher-level abstractions that make development faster and safer.

1.1 The source files you write ▼

A modern web project has these source files. None can run directly in a browser.

App.tsx — React component in TypeScript + JSX 
// TypeScript + JSX — browsers don't understand this!
import React, { useState, useEffect } from 'react';
import { fetchUsers } from './api/users';
import styles from './App.module.scss';  // SCSS import
import logo from './assets/logo.svg';     // Asset import

interface User {
  id: number;
  name: string;
  email: string;
}

const App: React.FC = () => {
  const [users, setUsers] = useState<User[]>([]);
  const [loading, setLoading] = useState<boolean>(true);

  useEffect(() => {
    fetchUsers()
      .then((data: User[]) => setUsers(data))
      .finally(() => setLoading(false));
  }, []);

  return (
    <div className={styles.container}>
      <img src={logo} alt="Logo" className={styles.logo} />
      <h1 className={styles.title}>User Dashboard</h1>
      {loading ? (
        <div className={styles.spinner}>Loading...</div>
      ) : (
        <ul className={styles.userList}>
          {users.map(user => (
            <li key={user.id} className={styles.userCard}>
              <h3>{user.name}</h3>
              <p>{user.email}</p>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
};

export default App;

Why can't browsers just run this directly?

Browsers only understand 3 languages: HTML, CSS, and JavaScript. TypeScript adds types JS doesn't have. JSX mixes HTML into JS. SCSS adds variables and nesting. Module imports like import React from 'react' don't resolve to real file paths. All must be transformed.

02

Build tools transform it

When you run npm run build, a pipeline of transformations kicks off. Each step takes the code closer to something the browser can understand — like a factory assembly line.

Source (TSX) → Transpile → Bundle → Optimize → dist/ output
2.1 Step 1: Transpile — convert syntax the browser doesn't know ▼

Transpilation converts one language into another at roughly the same level. TypeScript → JavaScript. JSX → function calls. SCSS → CSS. Tools: Babel, SWC (Rust, faster), tsc, Sass.

Before — TSX

);
→
After — Plain JS 
// Types stripped, JSX converted

const UserCard = ({ user }) => (
  React.createElement("div",

What happens to types?

TypeScript types (interface, type, generics like User[]) are completely erased. They exist only at dev time for error checking. Zero bytes of type info reach the browser. That's why TS is a "compile-time" type system.

2.2 Step 2: Bundle — merge hundreds of files into a few ▼

Your project has 50 source files and 500 node_modules packages. The bundler (Webpack, Vite/Rollup, esbuild) traces every import from your entry point and merges everything into a small number of output files.

What the bundler does — import resolution 
// Your code says:

Code splitting — smart chunking 
// Instead of ONE giant file, smart bundlers create chunks:

dist/
├── index.html

// The hash in filenames (Da3x8kQ1) = content hash
// If content doesn't change, hash stays the same
// → Browser cache still valid! Only changed files re-download

Why bundle at all?

HTTP/1.1: browsers could only make ~6 parallel requests per domain. 500 files = 83 round trips. One bundle = 1 request. Even with HTTP/2 (multiplexed), fewer files means less overhead. Bundling also enables tree-shaking and code-splitting.

Three optimizations: Minification (shrink code), Tree-shaking (remove unused code), Compression (gzip/brotli). Tools: Terser, cssnano, SWC minifier.

Deploy to the cloud

The cloud (AWS, Azure, GCP) is just someone else's powerful computer in a data center. You upload your dist/ folder so millions of users worldwide can request those files 24/7.

3.1 What "the cloud" actually is ▼ 
  "The Cloud" = Data centers around the world

  ┌─────────────────────────────────────────────────────┐
  │  AWS Data Center (us-east-1, Virginia)              │
  │                                                     │
  │  ┌──────────┐ ┌──────────┐ ┌──────────┐           │
  │  │ Server   │ │ Server   │ │ Server   │  ...×1000s│
  │  │ Rack #1  │ │ Rack #2  │ │ Rack #3  │           │
  │  │          │ │          │ │          │           │
  │  │ Your     │ │ Someone  │ │ Netflix  │           │
  │  │ dist/    │ │ else's   │ │ dist/    │           │
  │  │ folder   │ │ app      │ │ folder   │           │
  │  └──────────┘ └──────────┘ └──────────┘           │
  │                                                     │
  │  + Load balancers, networking, storage, cooling...  │
  └─────────────────────────────────────────────────────┘

  Your dist/ folder sits on one (or many) of these servers.
  Each is a real physical computer with CPU, RAM, SSD.
  The cloud provider manages hardware, networking, scaling.
AWS S3Store static files (dist/), serve directly. Cheapest. No server logic.
CloudFront CDNCache copies at 400+ edge locations worldwide. User hits nearest one.
EC2 / Azure VMFull server — run Node.js, SSR, APIs. You control everything.
Vercel / NetlifyAbstraction on top: git push → auto-build → deploy → CDN. Zero config.

Without CDN:
  User in Tokyo  → request to Virginia → 200ms latency

  User in Tokyo  → Tokyo edge server   → 15ms ⚡
  User in London → London edge server  → 10ms ⚡

  CDN edge servers ask origin ONCE, then cache.
  Every subsequent user gets served from nearest edge.

04

User makes a request

A user types https://yourdashboard.com and presses Enter. Here's every network step before a single byte of your code arrives.

4.1 DNS resolution — domain name to IP address ▼ 
User types: https://yourdashboard.com

Browser: "I need an IP address to connect to."

Step 1: Check browser DNS cache     → miss
Step 2: Check OS DNS cache          → miss
Step 3: Ask router's DNS cache      → miss
Step 4: Ask ISP's DNS resolver      → miss
Step 5: ISP asks Root DNS server    → "Try .com TLD server"
Step 6: Ask .com TLD server         → "Try ns1.cloudflare.com"
Step 7: Ask Cloudflare nameserver   → "93.184.216.34" ✅

Total: ~20-80ms (first visit), ~0ms (cached)
4.2 TCP 3-way handshake — reliable connection ▼ 
Browser (Client)                    Server (93.184.216.34)
     │                                     │
     │─── SYN (seq=100) ─────────────────▶│  "Want to talk?"
     │                                     │
     │◀── SYN-ACK (seq=300, ack=101) ─────│  "Sure, ready"
     │                                     │
     │─── ACK (ack=301) ─────────────────▶│  "Connected!"
     │                                     │

This "3-way handshake" takes 1 round-trip (~30ms).
Both sides agree on sequence numbers so packets
can be reassembled in order and retransmitted if lost.
4.3 TLS handshake — making it secure (HTTPS) ▼ 
Browser                                Server
   │                                      │
   │── ClientHello ─────────────────────▶│
   │   [TLS version, supported ciphers,  │
   │    random number]                   │
   │                                      │
   │◀─ ServerHello + Certificate ────────│
   │   [Chosen cipher, server's public   │
   │    key signed by a CA]              │
   │                                      │
   │   Browser verifies certificate:      │
   │   ✅ Signed by trusted CA?           │
   │   ✅ Domain name correct?            │
   │   ✅ Not expired?                    │
   │                                      │
   │── Key Exchange ───────────────────▶│
   │   [Diffie-Hellman → shared secret]  │
   │                                      │
   │══ Encrypted tunnel established ═════│
   │   All data encrypted with AES-256   │

TLS 1.3: 1 round-trip (~30ms)
TLS 1.2: 2 round-trips (~60ms)
4.4 HTTP request — asking for the page ▼ 
GET / HTTP/2
Host: yourdashboard.com
User-Agent: Mozilla/5.0 Chrome/120
Accept: text/html,application/xhtml+xml,*/*;q=0.8
Accept-Language: pl-PL,pl;q=0.9,en;q=0.8
Accept-Encoding: gzip, deflate, br     ← "I can handle compressed!"
Cookie: session=abc123; theme=dark      ← Previous session data
Cache-Control: no-cache
Sec-Fetch-Dest: document                ← "I want an HTML document"
4.5 Complete timeline — keystroke to first byte ▼
0msUser presses Enter
~5msURL parsed — protocol, domain, path
~25msDNS resolved → IP address
~55msTCP connected (SYN → SYN-ACK → ACK)
~85msTLS negotiated → encrypted tunnel
~90msHTTP request sent
~150msFirst byte received (TTFB)

05

Server responds

The server receives your HTTP request and decides what to send back. For a static site: find the file, compress it, send it.

5.1 Server processing & HTTP response ▼ 
Incoming: GET / HTTP/2

Server decision tree:
├── Cached at CDN edge?
│   ├── YES → Return cached copy (<5ms) ⚡
│   └── NO  → Forward to origin
│
├── Static site (our case):
│   1. Match "/" → serve index.html
│   2. Read file from disk/memory
│   3. Compress with Brotli
│   4. Set response headers
│   5. Send
│
└── Dynamic site (SSR):
    1. Run route handler
    2. Query database
    3. Render HTML on server
    4. Compress and send
HTTP response 
HTTP/2 200 OK
content-type: text/html; charset=UTF-8
content-encoding: br                       ← Brotli compressed
cache-control: no-cache                    ← HTML always re-validated
etag: "a3f8c2e1"                           ← Fingerprint for caching
cf-cache-status: HIT                       ← Served from CDN cache
strict-transport-security: max-age=31536000

[compressed HTML body follows]
5.2 Cascade of requests — HTML triggers more fetches ▼ 
Browser receives index.html, starts parsing.
As it finds resource references, it fires MORE requests:

Request #1: GET /                        → index.html (1KB)
  └─ Parser finds <link> and <script>:

Request #2: GET /assets/index-pQ8s.css   → styles (8KB)
Request #3: GET /assets/index-Da3x.js    → app bundle (45KB)
Request #4: GET /assets/vendor-Bk9m.js   → vendor chunk (140KB)
  └─ JS executes and triggers:

Request #5: GET /assets/logo-kM2x.svg    → logo (3KB)
Request #6: GET /api/users               → JSON data (12KB)

With HTTP/2, requests 2-4 happen IN PARALLEL (multiplexed)
over the SAME TCP connection. No extra handshakes!

Total: ~209KB compressed → ~850KB decompressed in memory

Caching headers control repeat visits

Hashed assets (app-Da3x.js) get max-age=31536000 (1 year!) because the hash changes when content changes. index.html gets no-cache so the browser always checks for a new version. Second visits are much faster.

06

Browser rendering engine

The rendering engine (Blink in Chrome, Gecko in Firefox, WebKit in Safari) takes raw files and turns them into interactive pixels. This is the most complex part of the chain.

6.1 Parse HTML → Build the DOM tree ▼ 
HTML bytes arrive:  3C 21 44 4F 43 54 59 50 45 ...
                          ↓ decode
Characters:         <!DOCTYPE html><html><head>...
                          ↓ tokenize
Tokens:             [DOCTYPE] [StartTag:html] [StartTag:head] ...
                          ↓ tree construction
DOM Tree:
  Document
  ├── DOCTYPE: html
  └── html
      ├── head
      │   ├── meta[charset="UTF-8"]
      │   ├── title → "User Dashboard"
      │   ├── link[rel="stylesheet" href="index-pQ8s.css"]
      │   │    ↑ Triggers CSS download! (render-blocking)
      │   └── script[type="module" src="index-Da3x.js"]
      │        ↑ Triggers JS download! (deferred with type=module)
      └── body
          └── div#root
               ↑ Empty — React fills this later

Parse-blocking behavior

A <script> without async/defer STOPS HTML parsing until it downloads and executes (JS might use document.write). Modern frameworks use type="module" (deferred by default). CSS is "render-blocking" — browser won't paint until CSS is parsed, but HTML parsing continues.

6.2 Parse CSS → Build CSSOM tree ▼ 
CSSOM (CSS Object Model):
  StyleSheet
  ├── Rule: .container
  │   ├── max-width: 800px
  │   ├── margin: 0 auto
  │   └── padding: 40px 24px
  ├── Rule: .container .title
  │   ├── font-size: 2rem        → computed: 32px
  │   ├── color: #378ADD
  │   └── margin-bottom: 24px
  └── Rule: .userCard
      ├── background: white       → computed: rgb(255,255,255)
      ├── border-radius: 12px
      └── box-shadow: 0 2px 8px rgba(0,0,0,0.1)

During CSSOM construction:
  1. Parse all CSS rules
  2. Resolve cascade (specificity: inline > #id > .class > element)
  3. Resolve inheritance (color inherits; margin doesn't)
  4. Compute final values (rem → px, % → px, colors → rgba)
6.3 DOM + CSSOM → Render tree → Layout → Paint ▼ 
RENDER TREE (only visible nodes — <head>, display:none excluded):

  RenderView (viewport: 1440×900)
  └── RenderBlock (body) [font: 16px, color: #1a1a1a]
      └── RenderBlock (div.container) [max-width: 800px, margin: 0 auto]
          ├── RenderBlock (h1.title) [color: #378ADD, font-size: 32px]
          │   └── RenderText "User Dashboard"
          └── RenderBlock (ul.userList) [display: grid, gap: 16px]
              ├── RenderBlock (li.userCard) [bg: white, radius: 12px]
              └── RenderBlock (li.userCard) [bg: white, radius: 12px]
LAYOUT (reflow) — exact positions & sizes:

  body       → x:0, y:0, width:1440, height:auto
  .container → x:320, y:0, width:800, height:auto  (centered)
  h1.title   → x:320, y:40, width:800, height:38
  ul.userList→ x:320, y:102, width:800 (3 cols × 256px + gaps)
  li:first   → x:320, y:102, width:256, height:76
  li:second  → x:592, y:102, width:256, height:76
PAINT — convert to drawing commands:

  1. fillRect(0, 0, 1440, 900, #fafaf8)        ← body bg
  2. drawText("User Dashboard", 320, 78, blue)  ← h1
  3. fillRoundedRect(320, 102, 256, 76, 12px)   ← card bg
  4. drawBoxShadow(...)                          ← card shadow
  5. drawText("Jane Smith", 340, 130, #1a1a1a)  ← name

COMPOSITE — GPU layers → final frame → display
  transform/opacity animations = own GPU layer = 60fps smooth
  width/height animations = layout thrashing = janky
~16msBudget per frame (60fps)
~2msPaint (typical)
~4msLayout (typical)
~1msComposite (GPU)

JavaScript event loop

JavaScript is single-threaded — one thing at a time. The event loop lets async operations (network, timers, user input) work without blocking the UI.

7.1 Components of the JS runtime ▼ 
┌─────────────────────────────────────────────────────────┐
│  JavaScript Engine (V8 in Chrome)                       │
│                                                         │
│  ┌──────────────────┐    ┌──────────────────────────┐  │
│  │   CALL STACK      │    │   HEAP (Memory)           │  │
│  │   (LIFO)          │    │                            │  │
│  │  ┌──────────────┐ │    │   { users: [...] }        │  │
│  │  │ fetchUsers() │ │    │   function App() {...}    │  │
│  │  ├──────────────┤ │    │   [1, 2, 3, 4, 5]        │  │

7.2 Event loop in action — real code walkthrough ▼
What order do logs appear? 
console.log('1: Script start');

setTimeout(() => {
  console.log('2: setTimeout');
}, 0);

Promise.resolve()
  .then(() => console.log('3: Promise 1'))
  .then(() => console.log('4: Promise 2'));

fetch('/api/users')
  .then(res => res.json())
  .then(data => console.log('5: Data received'));

console.log('6: Script end');
Execution order — step by step 
TICK 1 — Synchronous execution:
  → console.log('1: Script start')     // ✅ OUTPUT
  → setTimeout(cb, 0)                  // cb → Timer API
  → Promise.resolve().then(cb1)        // cb1 → microtask queue
  → fetch('/api/users')                // → Network API
  → console.log('6: Script end')       // ✅ OUTPUT
  Call Stack: [] (empty)

TICK 2 — Drain microtask queue:
  → console.log('3: Promise 1')        // ✅ OUTPUT
  → .then chains → console.log('4: Promise 2')  // ✅ OUTPUT

TICK 3 — Process one macrotask:
  → console.log('2: setTimeout')       // ✅ OUTPUT

TICK 4+ — fetch completes eventually:
  → console.log('5: Data received')    // ✅ OUTPUT

FINAL ORDER:
  "1: Script start"    ← synchronous
  "6: Script end"      ← synchronous
  "3: Promise 1"       ← microtask
  "4: Promise 2"       ← microtask
  "2: setTimeout"      ← macrotask (even with 0ms delay!)
  "5: Data received"   ← microtask (when network resolves)

setTimeout(fn, 0) is NOT instant

Even with 0ms delay, setTimeout goes to the macrotask queue. It must wait for: (1) synchronous code to finish, (2) ALL microtasks to drain, (3) its turn. Promise callbacks always run before setTimeout, even if setTimeout was registered first.

User sees & interacts

The page is visible and interactive. Every action triggers: Event → JS handler → State change → Re-render → New pixels. This loop runs at up to 60fps.

8.1 Core Web Vitals — measuring user experience ▼
FCP — First Contentful Paint (~1.2s)First text or image rendered. No longer blank.
LCP — Largest Contentful Paint (~2.0s)Biggest element (hero image, main heading) finishes rendering.
INP — Interaction to Next Paint (~100ms)Time between user input and next visual update.
TTI — Time to Interactive (~3.5s)Fully interactive. All JS loaded, handlers attached.
CLS — Cumulative Layout Shift (~0.05)Unexpected layout movement. Images without dimensions cause this.
8.2 The re-render cycle — what happens on every click ▼ 
User clicks "Delete User". Every step:

1. BROWSER EVENT
   └── Click detected → Event object created
   └── Dispatched through DOM (capture → target → bubble)

2. JS HANDLER
   └── React's delegated listener catches it
   └── handleDelete(userId) runs
   └── setState({ users: users.filter(u => u.id !== userId) })

3. REACT RECONCILIATION (Virtual DOM)
   └── New virtual DOM built with updated state
   └── Diff vs previous tree
   └── Finds: "remove 3rd <li>"
   └── Batches DOM mutations

4. DOM UPDATE
   └── removeChild(liElement)  ← only this one operation!
   └── NOT re-rendering entire page

5. BROWSER RE-RENDER (partial)
   └── Style recalc → affected elements only
   └── Layout       → list items reposition
   └── Paint        → repaint affected area
   └── Composite    → GPU recomposite

6. PIXELS UPDATED (~8-16ms total)
8.3 Network round-trip during interaction ▼ 
User clicks "Delete"
      │
      ├── Optimistic update: remove from UI immediately
      │   (user sees instant feedback!)
      │
      ▼
fetch('/api/users/42', { method: 'DELETE' })
      │

      │
