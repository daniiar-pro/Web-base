01

Developer writes code

You open your editor (VS Code, WebStorm, Vim) and write in languages the browser cannot understand natively. These are higher-level abstractions that make development faster and safer.

1.1 The source files you write ▼

A modern web project typically has these source files. None of them can run directly in a browser — they all need transformation.

App.tsx — Your React component in TypeScript + JSX 
// TypeScript + JSX — browsers don't understand this!
import React, { useState, useEffect } from 'react';
import { fetchUsers } from './api/users';
import styles from './App.module.scss';  // SCSS import
import logo from './assets/logo.svg';     // Asset import

interface User {
  id: number;
  name: string;
  email: string;
}

const App: React.FC = () => {
  const [users, setUsers] = useState<User[]>([]);
  const [loading, setLoading] = useState<boolean>(true);

  useEffect(() => {
    fetchUsers()
      .then((data: User[]) => setUsers(data))
      .finally(() => setLoading(false));
  }, []);

  return (
    <div className={styles.container}>
      <img src={logo} alt="Logo" className={styles.logo} />
      <h1 className={styles.title}>User Dashboard</h1>
      {loading ? (
        <div className={styles.spinner}>Loading...</div>
      ) : (
        <ul className={styles.userList}>
          {users.map(user => (
            <li key={user.id} className={styles.userCard}>
              <h3>{user.name}</h3>
              <p>{user.email}</p>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
};

export default App;

Why can't browsers just run this directly?

Browsers only understand 3 languages: HTML, CSS, and JavaScript (ES modules). TypeScript adds types that JS doesn't have. JSX mixes HTML-like syntax into JS. SCSS adds variables, nesting, and mixins to CSS. Module imports with bare specifiers (like import React from 'react') don't resolve to real file paths. All of this must be transformed before the browser can use it.

02

Build tools transform it

When you run npm run build, a pipeline of transformations kicks off. Each step takes the code closer to something the browser can understand. Think of it as a factory assembly line.

Source (TSX) → Transpile → Bundle → Optimize → dist/ output
2.1 Step 1: Transpile — Convert syntax the browser doesn't know ▼

Transpilation converts one language into another at roughly the same level of abstraction. TypeScript → JavaScript. JSX → function calls. SCSS → CSS. The tools doing this: Babel, SWC (Rust-based, faster), tsc (TypeScript compiler), Sass.

Before — TSX (browser can't read)

  id: number;
  name: string;
}

const UserCard: React.FC<{user: User}> = 
  ({ user }) => (
  <div className="card">
    <h3>{user.name}</h3>
    <p>ID: {user.id}</p>
  </div>
);
→
After — Plain JavaScript 
// Pure JS — all types stripped,
// JSX converted to function calls
const UserCard = ({ user }) => (
  React.createElement("div", 
    { className: "card" },
    React.createElement("h3", null, 
      user.name),
    React.createElement("p", null, 
      "ID: ", user.id)
  )
);
↓ SCSS also gets transpiled ↓

What exactly happens to types?

TypeScript types (interface, type, generics like User[]) are completely erased. They exist only at development time for error checking. Zero bytes of type info reach the browser. This is why TypeScript is called a "compile-time" type system.

2.2 Step 2: Bundle — Merge hundreds of files into a few ▼

Your project might have 50 source files and 500 node_modules packages. The bundler (Webpack, Vite/Rollup, esbuild) traces every import starting from your entry point and merges everything into a small number of output files.

What the bundler does — import resolution 
// Your code says:
import React from 'react';
import { fetchUsers } from './api/users';

// Bundler resolves these to actual file paths:
// 'react' → node_modules/react/index.js
//         → node_modules/react/cjs/react.production.min.js
// './api/users' → src/api/users.ts (then transpiles it)

// Then it INLINES everything into one big file:
// ┌──────────────────────────────────┐
// │  // React runtime code (~6KB)    │
// │  var React = { createElement... } │
// │                                    │
// │  // Your API module               │
// │  function fetchUsers() { ... }    │
// │                                    │
// │  // Your App component            │
// │  function App() { ... }           │
// │                                    │
// │  // Your UserCard component       │
// │  function UserCard() { ... }      │
// │                                    │
// │  // Entry point                   │
// │  ReactDOM.render(App, root);      │
// └──────────────────────────────────┘

Why bundle at all?

HTTP/1.1: browsers could only make ~6 parallel requests per domain. 500 files = 83 round trips. One bundle = 1 request. Even with HTTP/2 (which multiplexes), fewer files means less overhead from headers, compression boundaries, and cache management. Bundling also enables tree-shaking and code-splitting.

Code splitting — smart chunking 
// Instead of ONE giant bundle, smart bundlers create chunks:

dist/

// If the file content doesn't change, the hash stays the same
// → Browser cache still valid! Only changed files re-download

Deploy to the cloud

04

User makes a request

A user in Warsaw types https://yourdashboard.com and presses Enter. Here's every network step that happens before a single byte of your code arrives.

4.1 DNS Resolution — translating domain to IP address ▼ 
User types: https://yourdashboard.com

Browser thinks: "I need an IP address to connect to."

Step 1: Check browser DNS cache     → miss
Step 2: Check OS DNS cache          → miss  
Step 3: Ask router's DNS cache      → miss
Step 4: Ask ISP's DNS resolver      → miss
Step 5: ISP asks Root DNS server    → "Try .com TLD server"
Step 6: Ask .com TLD server         → "Try ns1.cloudflare.com"
Step 7: Ask Cloudflare nameserver   → "93.184.216.34" ✅

Total time: ~20-80ms (first visit)
           ~0ms (cached from previous visit)

Result: yourdashboard.com = 93.184.216.34

DNS is like a phone book

You know the name (yourdashboard.com), DNS gives you the number (93.184.216.34). The hierarchy goes: your machine → router → ISP → root servers → TLD servers → authoritative nameserver. Each level can cache the answer to speed up future lookups.

4.2 TCP Handshake — establishing a reliable connection ▼ 
Now the browser knows the IP. It needs a reliable connection.
TCP = Transmission Control Protocol (guarantees delivery & order)

  Browser (Client)                    Server (93.184.216.34)
       │                                     │
       │─── SYN (seq=100) ─────────────────▶│  "Hey, want to talk?"
       │    [I want to connect]              │
       │                                     │
       │◀── SYN-ACK (seq=300, ack=101) ─────│  "Sure, I'm ready too"
       │    [Got it, here's my sequence #]   │
       │                                     │
       │─── ACK (ack=301) ─────────────────▶│  "Great, we're connected!"
       │    [Connection established]         │
       │                                     │
  
  This "3-way handshake" takes 1 round-trip (~30ms).
  
  Why? TCP needs both sides to agree on sequence numbers
  so packets can be reassembled in order and retransmitted
  if lost. This is what makes TCP "reliable" vs UDP.
4.3 TLS Handshake — making the connection secure (HTTPS) ▼ 
TCP connection is open, but it's not encrypted yet.
TLS adds encryption on top (the S in HTTPS).

  Browser                                Server
     │                                      │
     │── ClientHello ─────────────────────▶│
     │   [TLS version, supported ciphers,  │
     │    random number]                   │
     │                                      │
     │◀─ ServerHello + Certificate ────────│
     │   [Chosen cipher, server's public   │
     │    key in a certificate signed by   │
     │    a Certificate Authority (CA)]    │
     │                                      │
     │   Browser verifies certificate:      │
     │   ✅ Is it signed by trusted CA?     │
     │   ✅ Is the domain name correct?     │
     │   ✅ Is it not expired?              │
     │                                      │
     │── Key Exchange ───────────────────▶│
     │   [Both sides compute shared secret │
     │    using Diffie-Hellman exchange]   │
     │                                      │
     │◀─▶ Both derive session keys ◀──────▶│
     │                                      │
     │══ Encrypted tunnel established ═════│
     │   All further data is encrypted     │
     │   with AES-256-GCM or similar       │

  TLS 1.3 does this in 1 round-trip (~30ms)
  TLS 1.2 needed 2 round-trips (~60ms)
4.4 HTTP Request — asking for the page ▼ 
Connection is secure. Browser sends the actual request:

GET / HTTP/2
Host: yourdashboard.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/120
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Accept-Language: pl-PL,pl;q=0.9,en;q=0.8
Accept-Encoding: gzip, deflate, br        ← "I can handle compressed!"
Connection: keep-alive
Cookie: session=abc123; theme=dark         ← Previous session data
Cache-Control: no-cache                    ← First visit, no cache
Sec-Fetch-Dest: document                   ← "I want an HTML document"
Sec-Fetch-Mode: navigate                   ← "User navigation"

Key headers explained:
• GET /         → "Give me the page at root path"
• HTTP/2        → Using HTTP version 2 (multiplexed, binary)
• Accept-Encoding → Browser can decompress gzip/brotli
• Cookie        → Sends stored cookies for this domain
• Cache-Control → Tells server what cached versions we have
4.5 Complete timeline — from keystroke to first byte ▼
0ms — User presses Enter Browser starts processing the URL
~5ms — URL parsed Protocol (https), domain (yourdashboard.com), path (/)
~25ms — DNS resolved Domain → IP address (cached or full lookup)
~55ms — TCP connected 3-way handshake complete (SYN → SYN-ACK → ACK)
~85ms — TLS negotiated Encrypted tunnel established (TLS 1.3)
~90ms — HTTP request sent GET / HTTP/2 flies through the encrypted tunnel
~150ms — First byte received (TTFB) Server processed request, first byte of HTML arrives

05

Server responds

The server (Nginx, Node.js, or a CDN edge) receives your HTTP request and decides what to send back. For a static site, it's straightforward: find the file, compress it, send it.

5.1 Server processing — what happens inside ▼ 
Incoming request: GET / HTTP/2

Server decision tree:
│
├── Is it cached at CDN edge? 
│   ├── YES → Return cached copy (< 5ms) ⚡
│   └── NO → Forward to origin server
│
├── Origin server receives request
│   │
│   ├── Static site (our case):
│   │   1. Match "/" → serve index.html
│   │   2. Read file from disk/memory
│   │   3. Compress with Brotli (br) or gzip
│   │   4. Set response headers
│   │   5. Send response
│   │
│   └── Dynamic site (SSR):
│       1. Match "/" → run route handler
│       2. Execute server-side code
│       3. Query database if needed
│       4. Render HTML on server
│       5. Compress and send

Processing time:
  Static file: ~1-5ms
  SSR page:    ~20-200ms
  Dynamic API: ~10-500ms (depends on DB, logic)
The HTTP response — what the server sends back 
HTTP/2 200 OK
content-type: text/html; charset=UTF-8
content-encoding: br                          ← Brotli compressed!
content-length: 892
cache-control: no-cache                       ← HTML always re-validated
etag: "a3f8c2e1"                              ← Fingerprint for caching
x-powered-by: Vercel
server: Cloudflare
cf-cache-status: HIT                          ← Served from CDN cache!
age: 3600                                     ← Cached for 1 hour
strict-transport-security: max-age=31536000   ← Force HTTPS
x-content-type-options: nosniff               ← Security header

<!DOCTYPE html>
<html lang="en">
  <head>...</head>
  <body><div id="root"></div></body>
</html>
5.2 Cascade of requests — HTML triggers more fetches ▼ 
The browser receives index.html and starts parsing it.
As it encounters resource references, it fires MORE requests:

Request #1: GET /                    → index.html (1KB)
  └─ Parser finds <link> and <script> tags:

Request #2: GET /assets/index-pQ8s.css    → styles (8KB)
Request #3: GET /assets/index-Da3x.js     → app bundle (45KB)
Request #4: GET /assets/vendor-Bk9m.js    → vendor chunk (140KB)
  └─ JS executes and triggers:

Request #5: GET /assets/logo-kM2x.svg     → logo image (3KB)
Request #6: GET /api/users                 → JSON data (12KB)
  └─ API response triggers more rendering

With HTTP/2, requests 2-4 happen IN PARALLEL (multiplexed)
over the SAME TCP connection. No extra handshakes needed!

Total download: ~209KB compressed
Decompressed:   ~850KB in browser memory

Caching headers control repeat visits

Hashed assets (app-Da3x.js) get Cache-Control: max-age=31536000 (1 year!) because the hash changes when content changes. index.html gets no-cache so the browser always checks for a new version. This is why second visits are much faster — only the HTML is re-fetched, everything else comes from disk cache.

06

Browser rendering engine

The browser's rendering engine (Blink in Chrome, Gecko in Firefox, WebKit in Safari) takes those raw files and turns them into interactive pixels. This is the most complex part of the entire chain.

6.1 Step 1: Parse HTML → Build the DOM tree ▼

The HTML parser reads bytes → converts to characters → identifies tokens (tags) → builds a tree of nodes. This tree is the DOM (Document Object Model) — your JavaScript interacts with this tree.

HTML bytes arrive:  3C 21 44 4F 43 54 59 50 45 ...
                          ↓ decode
Characters:         <!DOCTYPE html><html><head>...
                          ↓ tokenize
Tokens:             [DOCTYPE] [StartTag:html] [StartTag:head] ...
                          ↓ tree construction
DOM Tree:
  Document
  ├── DOCTYPE: html
  └── html
      ├── head
      │   ├── meta[charset="UTF-8"]
      │   ├── title → "User Dashboard"
      │   ├── link[rel="stylesheet" href="index-pQ8s.css"]
      │   │    ↑ Triggers CSS download! (render-blocking)
      │   └── script[type="module" src="index-Da3x.js"]
      │        ↑ Triggers JS download! (deferred with type=module)
      └── body
          └── div#root
               ↑ Empty for now — React will fill this later

Critical: Parse-blocking behavior

When the parser hits a <script> without async or defer, it STOPS parsing HTML until the script downloads and executes. That's because JS might write to the document via document.write(). Modern frameworks use type="module" (deferred by default) or defer to avoid this blocking. CSS is "render-blocking" — the browser won't paint until CSS is downloaded and parsed, but HTML parsing continues.

6.2 Step 2: Parse CSS → Build the CSSOM tree ▼ 
CSS bytes arrive and get the same treatment:
Bytes → Characters → Tokens → CSSOM Tree

CSSOM (CSS Object Model):
  StyleSheet
  ├── Rule: .container
  │   ├── max-width: 800px
  │   ├── margin: 0 auto
  │   └── padding: 40px 24px
  ├── Rule: .container .title
  │   ├── font-size: 2rem        (computed: 32px)
  │   ├── color: #378ADD
  │   └── margin-bottom: 24px
  ├── Rule: .userCard
  │   ├── background: white       (computed: rgb(255,255,255))
  │   ├── border-radius: 12px
  │   ├── padding: 20px
  │   └── box-shadow: 0 2px 8px rgba(0,0,0,0.1)
  └── Rule: .userCard:hover
      └── transform: translateY(-2px)

What happens during CSSOM construction:
  1. Parse all CSS rules
  2. Resolve cascade (which rule wins when they conflict?)
     → Specificity: inline > #id > .class > element
     → Order: later rules override earlier ones
     → !important overrides everything
  3. Resolve inheritance (color, font-size inherit; margin doesn't)
  4. Compute final values (rem → px, % → px, colors → rgba)
6.3 Step 3: DOM + CSSOM → Render tree ▼ 
The browser merges the DOM and CSSOM into the Render Tree.
Only VISIBLE elements are included.

DOM Tree:                    CSSOM:
  html                         html { font-size: 16px }
  ├── head (not visible)       .container { max-width: 800px }
  │   ├── meta                 .title { color: blue }
  │   ├── link                 .hidden { display: none }
  │   └── script              
  └── body                   
      └── div.container      
          ├── h1.title        
          ├── div.hidden ← display:none!       
          └── ul.userList     
              ├── li.userCard 
              └── li.userCard 

          ↓ MERGE (only visible nodes) ↓

Render Tree:
  RenderView (viewport)
  └── RenderBlock (body) [font: 16px, color: #1a1a1a]
      └── RenderBlock (div.container) [max-width: 800px, margin: 0 auto]
          ├── RenderBlock (h1.title) [color: #378ADD, font-size: 32px]
          │   └── RenderText "User Dashboard"
          └── RenderBlock (ul.userList) [display: grid, gap: 16px]
              ├── RenderBlock (li.userCard) [bg: white, radius: 12px]
              │   ├── RenderText "Jane Smith"
              │   └── RenderText "jane@example.com"
              └── RenderBlock (li.userCard) [bg: white, radius: 12px]
                  ├── RenderText "John Doe"
                  └── RenderText "john@example.com"

  ❌ <head>, <meta>, <script>, <link> → NOT in render tree
  ❌ div.hidden (display:none) → NOT in render tree
  ✅ Only elements that produce visual output make it in
6.4 Step 4: Layout (Reflow) — calculate positions & sizes ▼ 
The render tree knows WHAT to show and HOW it looks.
Layout calculates WHERE and HOW BIG each element is.

Input:  Render tree + viewport size (e.g., 1440×900)

Processing:
  body       → x:0, y:0, width:1440, height:auto
  .container → x:320, y:0, width:800, height:auto
               (max-width:800, margin:0 auto → centered)
  h1.title   → x:320, y:40, width:800, height:38
               (font-size:32px → line-height ~38px)
  ul.userList→ x:320, y:102, width:800, height:auto
               (display:grid, grid-template-columns: 
                repeat(auto-fill, minmax(250px, 1fr))
                → 3 columns of 256px each with 16px gaps)
  li:first   → x:320, y:102, width:256, height:76
  li:second  → x:592, y:102, width:256, height:76

Output: "Box model" — every element has exact:
  • Content box (width × height)
  • Padding
  • Border  
  • Margin
  • Absolute position (x, y) on the page

This is expensive! Changing one element's size can cascade
to reflow of every sibling and descendant. That's why
layout thrashing (reading then writing DOM in a loop) is slow.
6.5 Step 5: Paint → Composite → Pixels on screen ▼ 
PAINT phase — convert layout boxes to drawing commands:
  
  1. Paint background of body     → fillRect(0, 0, 1440, 900, #fafaf8)
  2. Paint .container background  → (transparent, skip)
  3. Paint h1 text                → drawText("User Dashboard", 320, 78, 
                                              #378ADD, 32px, bold)
  4. Paint li background          → fillRoundedRect(320, 102, 256, 76, 
                                                     12px, #ffffff)
  5. Paint li shadow              → drawBoxShadow(320, 102, 256, 76, 
                                                   0 2px 8px rgba(0,0,0,0.1))
  6. Paint li text "Jane Smith"   → drawText("Jane Smith", 340, 130, 
                                              #1a1a1a, 16px, bold)
  ... and so on for every visible element

COMPOSITE phase — layering for GPU acceleration:
  
  Some elements get their own GPU layer:
  • Elements with transform/opacity animations
  • Elements with will-change: transform
  • Position: fixed elements
  • Video/canvas elements
  
  Layer 0 (main):   body, container, static elements
  Layer 1 (promoted): .userCard (has transform on hover)
  Layer 2 (promoted): fixed navbar (if any)
  
  GPU composites these layers into final frame → sent to display
  This is why transform/opacity animations are smooth (60fps)
  while width/height animations cause layout thrashing (janky)
~16ms Budget per frame (60fps)
~2ms Paint (typical)
~4ms Layout (typical)
~1ms Composite (GPU)

JavaScript event loop

JavaScript is single-threaded — it can only do one thing at a time. The event loop is the mechanism that allows async operations (network requests, timers, user input) to work without blocking the UI. This is the brain of interactivity.

7.2 Event loop in action — real code walkthrough ▼
This code — what order do the logs appear? 
console.log('1: Script start');

setTimeout(() => {
  console.log('2: setTimeout');
}, 0);

Promise.resolve()
  .then(() => console.log('3: Promise 1'))
  .then(() => console.log('4: Promise 2'));

fetch('/api/users')
  .then(res => res.json())
  .then(data => console.log('5: Data received'));

console.log('6: Script end');
Execution order — step by step 
TICK 1 — Synchronous execution:
  Call Stack: [main]
  → console.log('1: Script start')     // OUTPUT: "1: Script start"
  → setTimeout(cb, 0)                  // cb → sent to Timer API  
  → Promise.resolve().then(cb1)        // cb1 → microtask queue
  → fetch('/api/users').then(cb2)      // fetch → sent to Network API
  → console.log('6: Script end')       // OUTPUT: "6: Script end"
  Call Stack: [] (empty)

TICK 2 — Drain microtask queue:
  Microtask queue: [cb1: Promise 1]
  → console.log('3: Promise 1')        // OUTPUT: "3: Promise 1"
  → .then(cb3) chains → cb3 added to microtask queue
  Microtask queue: [cb3: Promise 2]
  → console.log('4: Promise 2')        // OUTPUT: "4: Promise 2"
  Microtask queue: [] (empty)

TICK 3 — Process one macrotask:
  Macrotask queue: [setTimeout cb]
  → console.log('2: setTimeout')       // OUTPUT: "2: setTimeout"

TICK 4+ — Eventually, fetch completes:
  Network API → Promise resolves → microtask queue
  → console.log('5: Data received')    // OUTPUT: "5: Data received"

FINAL ORDER:
  "1: Script start"    ← synchronous
  "6: Script end"      ← synchronous
  "3: Promise 1"       ← microtask
  "4: Promise 2"       ← microtask
  "2: setTimeout"      ← macrotask (even with 0ms delay!)
  "5: Data received"   ← microtask (after network resolves)

Key insight: setTimeout(fn, 0) is NOT instant

Even with 0ms delay, setTimeout goes to the macrotask queue. It must wait for: (1) the current synchronous code to finish, (2) ALL microtasks to drain, and (3) its turn in the macrotask queue. That's why Promise callbacks always run before setTimeout callbacks, even if setTimeout was registered first.

User sees & interacts

The page is now visible and interactive. Every user action triggers a cycle: Event → JS handler → State change → Re-render → New pixels. This loop runs continuously at up to 60fps.

8.1 Core Web Vitals — measuring user experience ▼
FCP — First Contentful Paint (~1.2s)First text or image rendered. User sees "something." Background is no longer blank.
LCP — Largest Contentful Paint (~2.0s)The biggest element (hero image, main heading) finishes rendering. Page "looks" loaded.
FID / INP — Interaction Delay (~100ms)First time user clicks and the browser responds. Measures input responsiveness.
TTI — Time to Interactive (~3.5s)Page is fully interactive. All JS loaded, event handlers attached, no long tasks blocking.
CLS — Cumulative Layout Shift (~0.05)Measures unexpected layout movement. Images without dimensions, late-loading fonts cause CLS.
8.2 The re-render cycle — what happens on every click ▼ 
User clicks a "Delete User" button. Here's every step:

1. BROWSER EVENT
   └── Click detected by browser's event system
   └── Event object created: { type: 'click', target: button, ... }
   └── Dispatched through DOM (capture → target → bubble phase)

2. JAVASCRIPT HANDLER
   └── React's delegated listener catches the event
   └── Your handler runs: handleDelete(userId)
   └── setState({ users: users.filter(u => u.id !== userId) })

3. REACT RECONCILIATION (Virtual DOM)
   └── React builds NEW virtual DOM tree with updated state
   └── Diffs new tree vs previous tree ("reconciliation")
   └── Finds minimal changes: "remove 3rd <li> element"
   └── Batches all DOM mutations

4. DOM UPDATE
   └── React applies minimal changes to REAL DOM
   └── removeChild(liElement)    ← only this one operation!
   └── NOT re-rendering the entire page

5. BROWSER RE-RENDER PIPELINE (same as initial render but partial)
   └── Style recalculation  → only affected elements
   └── Layout (reflow)      → list items reposition
   └── Paint                → repaint affected area only
   └── Composite            → GPU layers recomposite

6. NEW PIXELS ON SCREEN
   └── User sees the card disappear
   └── Total time: ~8-16ms (within single frame budget)

   ┌──────────────────────────────────────────┐
   │  This entire cycle runs within 16ms      │
   │  to maintain 60fps. If it takes longer,  │
   │  the user sees "jank" — a dropped frame. │
   └──────────────────────────────────────────┘

8.4 Network requests during interaction (the data round-trip) ▼

When the user's action requires data from the server (like deleting a user from the database), the cycle extends to include a network round-trip. The same networking layer from Phase 4 runs again.

User clicks "Delete"
      │
      ▼
JS Handler: fetch('/api/users/42', { method: 'DELETE' })
      │
      ├── Optimistic update: remove from UI immediately
      │   (user sees instant feedback!)
      │
      ▼
Network layer kicks in again:
  → TCP connection already open (keep-alive/HTTP2) ✅
  → TLS session resumed (no handshake needed) ✅  
  → Just sends: DELETE /api/users/42 HTTP/2
      │
      ▼
Server:
  → Route: DELETE /api/users/:id
  → Middleware: check auth token
  → Controller: Users.destroy(42)
  → Database: DELETE FROM users WHERE id = 42
  → Response: 204 No Content
      │
      ▼
Response arrives (~50-200ms later):
  → Success → state already updated (optimistic)
  → Failure → ROLLBACK: put the user back in the list
              show error toast to user

This is why modern apps feel fast:
  OPTIMISTIC UPDATES = update UI before server confirms
  If server says "actually no" → undo the change
