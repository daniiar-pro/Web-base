01

Developer writes code

You open your editor and write in languages the browser cannot understand natively. TypeScript, JSX, SCSS — these are higher-level abstractions that make development faster and safer, but they all need transformation before the browser can use them.

1.1 The source files you write ▼

A modern web project has these source files. None of them run directly in a browser — they all need transformation.

App.tsx — React component in TypeScript + JSX 
// TypeScript + JSX — browsers don't understand this!
import React, { useState, useEffect } from 'react';
import { fetchUsers } from './api/users';
import styles from './App.module.scss';  // SCSS import
import logo from './assets/logo.svg';     // Asset import

interface User {
  id: number;
  name: string;
  email: string;
}

const App: React.FC = () => {
  const [users, setUsers] = useState<User[]>([]);
  const [loading, setLoading] = useState<boolean>(true);

  useEffect(() => {
    fetchUsers()
      .then((data: User[]) => setUsers(data))
      .finally(() => setLoading(false));
  }, []);

  return (
    <div className={styles.container}>
      <img src={logo} alt="Logo" className={styles.logo} />
      <h1 className={styles.title}>User Dashboard</h1>
      {loading ? (
        <div className={styles.spinner}>Loading...</div>
      ) : (
        <ul className={styles.userList}>
          {users.map(user => (
            <li key={user.id} className={styles.userCard}>
              <h3>{user.name}</h3>
              <p>{user.email}</p>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
};

Why can't browsers just run this directly?

Browsers only understand 3 languages: HTML, CSS, and JavaScript. TypeScript adds types JS doesn't have. JSX mixes HTML-like syntax into JS. SCSS adds variables, nesting, mixins. Module imports like import React from 'react' don't resolve to real file paths. All of this must be transformed.

02

Build tools transform it

When you run npm run build, a pipeline of transformations kicks off. Each step takes the code closer to something the browser can understand — like a factory assembly line.

Source (TSX)→ Transpile→ Bundle→ Optimize→ dist/ output
2.1 Transpile — convert syntax the browser doesn't know ▼

Transpilation converts one language into another at the same abstraction level. TypeScript → JavaScript. JSX → function calls. SCSS → CSS. Tools: Babel, SWC (Rust-based, faster), tsc, Sass.

    <p>ID: {user.id}</p>
  </div>
);
→
After — Plain JS

const UserCard = ({ user }) => (
  React.createElement(
    "div",

      "h3", null,

    ),

What happens to types?

TypeScript types (interface, type, generics) are completely erased. They exist only at dev time for error checking. Zero bytes of type info reach the browser.

2.2 Bundle — merge hundreds of files into a few ▼

The bundler (Webpack, Vite/Rollup, esbuild) traces every import from your entry point and merges everything into a small number of output files.

// Your code says:
import React from 'react';
import { fetchUsers } from './api/users';

// Bundler resolves these to actual file paths:
// 'react'        → node_modules/react/cjs/react.production.min.js
// './api/users'  → src/api/users.ts (transpiles it first)

// Then INLINES everything into one file:
// ┌────────────────────────────────────┐
// │  React runtime code (~6KB)         │
// │  Your API module                   │
// │  Your App component                │
// │  Your UserCard component           │
// │  Entry point: ReactDOM.render()    │
// └────────────────────────────────────┘
Code splitting — smart chunking 
dist/
├── index.html
├── assets/
│   ├── index-Da3x8kQ1.js        ← Main app (~45KB)
│   ├── vendor-Bk9mWe7R.js       ← React + libs (~140KB)
│   ├── UserDashboard-x8Km2.js   ← Lazy-loaded route (~12KB)
│   ├── index-pQ8sW2e1.css       ← All styles merged (~8KB)
│   └── logo-kM2xW9.svg          ← Hashed asset

// Hash in filenames = content hash
// Same content → same hash → browser cache still valid!
// Only changed files re-download
Why bundle at all?

HTTP/1.1: browsers made ~6 parallel requests per domain. 500 files = 83 round trips. Even with HTTP/2 multiplexing, fewer files means less overhead. Bundling also enables tree-shaking and code-splitting.

2.3 Optimize — shrink everything to minimum size ▼

Three optimizations: Minification (shrink code), Tree-shaking (remove unused code), Compression (gzip/brotli). Tools: Terser, cssnano, SWC minifier.

No TypeScript. No SCSS. No node_modules. No JSX. Just plain HTML, CSS, JavaScript, and assets. The entire build pipeline exists to transform developer-friendly source into browser-friendly output.

Deploy to the cloud

The cloud (AWS, Azure, GCP) is just someone else's powerful computer in a data center. You upload your dist/ folder so millions of users worldwide can request those files 24/7.

3.3 CDN — copies of your files everywhere ▼ 
Without CDN:
  User in Tokyo     → Virginia → 200ms
  User in London    → Virginia → 100ms
  User in São Paulo → Virginia → 180ms

With CDN (CloudFront, Cloudflare):
  User in Tokyo     → Tokyo edge   → 15ms ⚡
  User in London    → London edge  → 10ms ⚡
  User in São Paulo → SP edge      → 12ms ⚡

  Edge servers ask origin ONCE, then cache.
  Every user after that gets the cached copy.

04

User makes a request

A user types https://yourdashboard.com and presses Enter. Here's every network step that happens before a single byte of your code arrives.

4.1 DNS — domain name to IP address ▼ 
User types: https://yourdashboard.com

Browser: "I need an IP address."

1. Check browser DNS cache     → miss
2. Check OS DNS cache          → miss
3. Ask router DNS cache        → miss
4. Ask ISP DNS resolver        → miss
5. ISP asks Root DNS           → "Try .com TLD"
6. Ask .com TLD server         → "Try ns1.cloudflare.com"
7. Ask Cloudflare nameserver   → "93.184.216.34" ✅

Time: ~20-80ms (first visit), ~0ms (cached)
4.2 TCP handshake — reliable connection ▼ 
  Browser (Client)                Server
       │                              │
       │── SYN (seq=100) ───────────▶│  "Want to talk?"
       │                              │
       │◀─ SYN-ACK (seq=300,ack=101)─│  "Sure, ready"
       │                              │
       │── ACK (ack=301) ───────────▶│  "Connected!"
       │                              │

  3-way handshake = 1 round-trip (~30ms)
  Both sides agree on sequence numbers
  so packets reassemble in order.
4.3 TLS handshake — encryption (HTTPS) ▼ 
  Browser                           Server
     │                                 │
     │── ClientHello ────────────────▶│
     │   [TLS version, ciphers]       │
     │                                 │
     │◀─ ServerHello + Certificate ──│
     │   [Chosen cipher, public key]  │
     │                                 │
     │   Browser verifies cert:        │
     │   ✅ Trusted CA?                │
     │   ✅ Correct domain?            │
     │   ✅ Not expired?               │
     │                                 │
     │── Key Exchange ──────────────▶│
     │   [Diffie-Hellman shared secret]│
     │                                 │
     │══ Encrypted tunnel ready ══════│
     │   AES-256-GCM from here on     │

  TLS 1.3 = 1 round-trip (~30ms)
  TLS 1.2 = 2 round-trips (~60ms)
4.4 HTTP request — asking for the page ▼ 
GET / HTTP/2
Host: yourdashboard.com
User-Agent: Mozilla/5.0 Chrome/120
Accept: text/html,application/xhtml+xml
Accept-Encoding: gzip, deflate, br    ← "I can decompress!"
Cookie: session=abc123; theme=dark     ← Previous session
Cache-Control: no-cache
Sec-Fetch-Dest: document
Sec-Fetch-Mode: navigate
4.5 Timeline — keystroke to first byte ▼
0msUser presses Enter
~5msURL parsed (protocol, domain, path)
~25msDNS resolved → IP address
~55msTCP connected (3-way handshake)
~85msTLS negotiated (encrypted tunnel)
~90msHTTP request sent through tunnel
~150msFirst byte received (TTFB)

05

Server responds

The server receives the HTTP request and decides what to send back. For a static site: find the file, compress it, send it.

5.1 Server processing & HTTP response ▼ 
Incoming: GET / HTTP/2

Decision tree:
├── Cached at CDN edge? → YES → return cached (<5ms) ⚡
│                        → NO  → forward to origin
│
└── Origin:
    ├── Static site: match "/" → serve index.html → compress → send
    └── Dynamic SSR: run route handler → query DB → render HTML → send

Processing: static ~1-5ms, SSR ~20-200ms
HTTP response headers 
HTTP/2 200 OK
content-type: text/html; charset=UTF-8
content-encoding: br                     ← Brotli compressed
cache-control: no-cache                  ← HTML always re-validated
etag: "a3f8c2e1"                         ← Fingerprint
cf-cache-status: HIT                     ← Served from CDN!
strict-transport-security: max-age=31536000

<!DOCTYPE html>
<html lang="en">...</html>
5.2 Cascade of requests — HTML triggers more fetches ▼ 
Request #1: GET /                        → index.html (1KB)
  └─ Parser finds <link> and <script>:

Request #2: GET /assets/index-pQ8s.css   → styles (8KB)
Request #3: GET /assets/index-Da3x.js    → app bundle (45KB)
Request #4: GET /assets/vendor-Bk9m.js   → vendor chunk (140KB)
  └─ JS executes and triggers:

Request #5: GET /assets/logo-kM2x.svg    → image (3KB)
Request #6: GET /api/users               → JSON data (12KB)

HTTP/2: requests 2-4 happen IN PARALLEL over same connection!

Total: ~209KB compressed → ~850KB decompressed in memory
Caching headers control repeat visits

Hashed assets (app-Da3x.js) get max-age=31536000 (1 year!) — the hash changes when content changes. index.html gets no-cache — always re-checked. Second visit: only HTML re-fetched, everything else from disk cache.

06

Browser rendering engine

The rendering engine (Blink in Chrome, Gecko in Firefox, WebKit in Safari) takes raw files and turns them into interactive pixels. This is the most complex part of the entire chain.

6.1 Parse HTML → build the DOM tree ▼

HTML bytes → characters → tokens → tree of nodes = the DOM.

Bytes:    3C 21 44 4F 43 54 59 50 45 ...
            ↓ decode
Chars:    <!DOCTYPE html><html><head>...
            ↓ tokenize
Tokens:   [DOCTYPE] [StartTag:html] [StartTag:head]...
            ↓ tree construction

DOM Tree:
  Document
  ├── DOCTYPE: html
  └── html
      ├── head
      │   ├── meta[charset="UTF-8"]
      │   ├── link[rel="stylesheet"]  ← Triggers CSS download!
      │   └── script[type="module"]   ← Triggers JS download!
      └── body
          └── div#root                ← Empty (React fills later)
Parse-blocking behavior

A <script> without async/defer STOPS HTML parsing until it downloads and executes. CSS is "render-blocking" — browser won't paint until CSS is parsed, but HTML parsing continues. Modern frameworks use type="module" (deferred by default).

6.2 Parse CSS → build the CSSOM ▼ 
CSS bytes → characters → tokens → CSSOM tree

CSSOM (CSS Object Model):
  StyleSheet
  ├── .container → { max-width:800px, margin:0 auto }
  ├── .title     → { font-size:32px, color:#378ADD }
  ├── .userCard  → { background:#fff, border-radius:12px }
  └── .userCard:hover → { transform:translateY(-2px) }

During CSSOM construction:
  1. Parse all rules
  2. Resolve cascade (specificity: inline > #id > .class > elem)
  3. Resolve inheritance (color inherits; margin doesn't)
  4. Compute values (rem→px, %→px, colors→rgba)
6.3 DOM + CSSOM → Render tree ▼ 
Merge DOM + CSSOM → Render Tree (only VISIBLE elements)

DOM:                   CSSOM:
  html                   .container { max-width:800px }
  └── body               .hidden { display:none }
      └── div.container
          ├── h1.title
          ├── div.hidden  ← display:none!
          └── ul.userList
              ├── li
              └── li

        ↓ MERGE ↓

Render Tree:
  RenderView (viewport)
  └── RenderBlock (body)
      └── RenderBlock (div.container) [max-width:800px]
          ├── RenderBlock (h1.title) [color:#378ADD, 32px]
          └── RenderBlock (ul.userList) [display:grid]
              ├── RenderBlock (li) [bg:white, radius:12px]
              └── RenderBlock (li) [bg:white, radius:12px]

  ❌ <head>, <script>, display:none → EXCLUDED
  ✅ Only visually-rendered elements make it in
6.4 Layout (reflow) — calculate positions & sizes ▼ 
Viewport: 1440×900

body        → x:0    y:0    w:1440  h:auto
.container  → x:320  y:0    w:800   h:auto  (centered)
h1.title    → x:320  y:40   w:800   h:38
ul.userList → x:320  y:102  w:800   h:auto  (grid: 3 cols)
li:first    → x:320  y:102  w:256   h:76
li:second   → x:592  y:102  w:256   h:76

Every element gets exact:
  • Content box (w × h)
  • Padding, Border, Margin
  • Absolute position (x, y)

⚠️ Expensive! Changing one element can cascade
   to reflow of every sibling and descendant.
6.5 Paint → Composite → Pixels on screen ▼ 
PAINT — convert boxes to draw commands:
  fillRect(0, 0, 1440, 900, #fafaf8)         ← body bg
  drawText("User Dashboard", 320, 78, blue)   ← h1
  fillRoundedRect(320, 102, 256, 76, 12, #fff)← card
  drawBoxShadow(...)                           ← shadow
  drawText("Jane Smith", 340, 130, #1a1a1a)   ← name

COMPOSITE — GPU layering:
  Layer 0 (main):  body, container, static elements
  Layer 1:         .userCard (has transform on hover)
  Layer 2:         fixed navbar (if any)

  GPU composites layers → final frame → display

  This is why transform/opacity = smooth (60fps)
  while width/height changes = janky (triggers layout)
~16msBudget/frame (60fps)
~2msPaint (typical)
~4msLayout (typical)
~1msComposite (GPU)

7.2 Event loop in action — real code walkthrough ▼
What order do these logs appear? 
console.log('1: Script start');

setTimeout(() => {
  console.log('2: setTimeout');
}, 0);

Promise.resolve()
  .then(() => console.log('3: Promise 1'))
  .then(() => console.log('4: Promise 2'));

console.log('5: Script end');
Execution order — step by step 
TICK 1 — Synchronous:
  → "1: Script start"             // sync
  → setTimeout(cb, 0)             // cb → Timer API
  → Promise.resolve().then(cb1)   // cb1 → microtask queue
  → "5: Script end"               // sync
  Stack: empty

TICK 2 — Drain microtasks:
  → "3: Promise 1"                // microtask
  → .then chains cb2 → microtask queue
  → "4: Promise 2"                // microtask
  Microtask queue: empty

TICK 3 — Process macrotask:
  → "2: setTimeout"               // macrotask

FINAL ORDER:
  "1: Script start"     ← sync
  "5: Script end"       ← sync
  "3: Promise 1"        ← microtask
  "4: Promise 2"        ← microtask
  "2: setTimeout"       ← macrotask (even with 0ms!)
setTimeout(fn, 0) is NOT instant

Even with 0ms delay, it goes to the macrotask queue. Must wait for: (1) sync code to finish, (2) ALL microtasks to drain, (3) its turn. Promises always run before setTimeout.

User sees & interacts

The page is visible and interactive. Every user action triggers: Event → JS handler → State change → Re-render → New pixels. This loop runs at up to 60fps.

8.1 Core Web Vitals — measuring UX ▼
FCP ~1.2sFirst Contentful Paint — user sees "something"
LCP ~2.0sLargest Contentful Paint — main content visible
INP ~100msInteraction to Next Paint — input responsiveness
TTI ~3.5sTime to Interactive — fully interactive
CLS ~0.05Cumulative Layout Shift — visual stability
8.2 The re-render cycle — every click step by step ▼ 
User clicks "Delete User":

1. BROWSER EVENT
   └── Click detected → Event object created
   └── Dispatched (capture → target → bubble)

2. JS HANDLER
   └── React catches event
   └── handleDelete(userId) runs
   └── setState({ users: users.filter(...) })

3. REACT RECONCILIATION
   └── New virtual DOM tree with updated state
   └── Diff vs previous tree
   └── Finds: "remove 3rd <li>"
   └── Batches DOM mutations

4. DOM UPDATE
   └── removeChild(liElement)  ← minimal change!
   └── NOT re-rendering entire page

5. BROWSER RE-RENDER (partial)
   └── Style recalc → affected elements only
   └── Layout → list items reposition
   └── Paint → repaint affected area
   └── Composite → GPU recomposite

6. PIXELS UPDATED
   └── User sees the card disappear
   └── Total: ~8-16ms (within frame budget)

8.4 Network round-trips during interaction ▼ 
User clicks "Delete":
      │
      ▼
JS: fetch('/api/users/42', { method: 'DELETE' })
      │
      ├── Optimistic update: remove from UI NOW
      │   (user sees instant feedback!)
      │
      ▼
Network (same TCP connection, TLS resumed):
  → DELETE /api/users/42 HTTP/2
      │
      ▼
Server:
  → Auth check → Controller → DB: DELETE WHERE id=42
  → Response: 204 No Content
      │
      ▼
Back in browser (~50-200ms):
  → Success → state already correct (optimistic)
  → Failure → ROLLBACK: put user back, show error

OPTIMISTIC UPDATES = update UI before server confirms
  If server says "no" → undo
