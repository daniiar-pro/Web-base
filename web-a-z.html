Frontend source code

You open your editor and write in languages the browser cannot understand natively — TypeScript, JSX, SCSS. These are higher-level abstractions that make development faster, safer, and more maintainable than raw HTML/CSS/JS.

1.1 The source files you write ▼

A modern web project has these source files. None can run directly in a browser — they all need to be transformed first.

App.tsx — React component in TypeScript + JSX 
// TypeScript + JSX — browsers don't understand this!
import React, { useState, useEffect } from 'react';
import { fetchUsers } from './api/users';
import styles from './App.module.scss';  // SCSS import
import logo from './assets/logo.svg';     // Asset import

interface User {
  id: number;
  name: string;
  email: string;
}

const App: React.FC = () => {
  const [users, setUsers] = useState<User[]>([]);
  const [loading, setLoading] = useState<boolean>(true);

  useEffect(() => {
    fetchUsers()
      .then((data: User[]) => setUsers(data))
      .finally(() => setLoading(false));
  }, []);

  return (
    <div className={styles.container}>
      <img src={logo} alt="Logo" className={styles.logo} />
      <h1 className={styles.title}>User Dashboard</h1>
      {loading ? (
        <div className={styles.spinner}>Loading...</div>
      ) : (
        <ul className={styles.userList}>
          {users.map(user => (
            <li key={user.id} className={styles.userCard}>
              <h3>{user.name}</h3>
              <p>{user.email}</p>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
};

export default App;

Why can't browsers just run this directly?

Browsers only understand 3 languages: HTML, CSS, and JavaScript. TypeScript adds types JS doesn't have. JSX mixes HTML into JS. SCSS adds variables and nesting. Module imports like import React from 'react' don't resolve to real file paths. All must be transformed by a build tool before the browser ever sees them.

1.3 The frontend talks to a backend ▼

Notice the fetchUsers() call in App.tsx. That function lives in src/api/users.ts and makes an HTTP request to a server you don't see in this folder — the backend. The frontend can only render data; the backend owns the data, the database, the auth, and the business rules.

src/api/users.ts — the bridge to the backend 
// This file is part of the FRONTEND but its only job is to
// talk to the BACKEND over HTTP.

const API_BASE = import.meta.env.VITE_API_URL; // e.g. https://api.example.com

export async function fetchUsers(): Promise<User[]> {
  const res = await fetch(`${API_BASE}/users`, {
    headers: { 'Authorization': `Bearer ${getToken()}` },
    credentials: 'include',  // send cookies
  });

  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

Frontend & backend are separate programs

They run on different machines, are written by potentially different people, and communicate only through a contract — typically a REST or GraphQL API. The frontend trusts nothing the user types; the backend trusts nothing the frontend sends. Both validate. That contract is what the next section is about.

Backend source code

The backend is a separate program that runs on a server. It owns the database, enforces business rules, handles authentication, and exposes an API the frontend calls. In the JS/TS world it usually runs on the Node.js runtime (or Bun/Deno) — Java, Go, Python, C#, Rust, PHP and Ruby are all common alternatives.

2.1 What is Node.js? (the runtime, not a language) ▼

JavaScript is a language. Node.js is a program that runs JavaScript outside the browser. It bundles Google's V8 engine (the same one Chrome uses) with a set of system APIs the browser doesn't have — file system, networking, child processes, OS info.

Why use Node.js for the backend?

One language end-to-end, shared types between client and server (huge productivity win with TypeScript), an enormous package ecosystem, and a non-blocking I/O model that handles thousands of concurrent connections cheaply. The downside: CPU-heavy work (image processing, ML) blocks the single thread — for that, reach for Go, Rust, or worker threads.

2.2 Raw Node.js — the absolute minimum HTTP server ▼

Before frameworks, this is what a server looks like. Useful to understand what frameworks are abstracting away.

server.js — pure Node, zero dependencies 
import http from 'node:http';

const server = http.createServer((req, res) => {
  // req  = IncomingMessage  (URL, headers, method, body stream)
  // res  = ServerResponse   (write status, headers, body)

  if (req.url === '/users' && req.method === 'GET') {
    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify([{ id: 1, name: 'Jane' }]));
    return;
  }

  res.writeHead(404);
  res.end('Not found');
});

server.listen(3000, () => console.log('http://localhost:3000'));

This works — but you'd have to hand-roll routing, body parsing, validation, error handling, CORS, auth, logging, etc. for every endpoint. That's why frameworks exist.

2.3 Two philosophies: Express vs NestJS ▼

Node has dozens of frameworks, but they fall into two camps: minimal & flexible (Express, Fastify, Koa, Hono) where you wire everything yourself, and opinionated & structured (NestJS, AdonisJS) where the framework dictates the architecture. Click the tabs to compare the same endpoint in each.

Express — explicit, procedural, you compose everything 
import express from 'express';
import { z } from 'zod';
import { db } from './db';
import { requireAuth } from './middleware/auth';

const app = express();

// Global middleware — runs on every request, in order
app.use(express.json());           // parse JSON bodies
app.use(cors({ origin: '...' }));  // allow frontend domain
app.use(requestLogger);            // your custom logger

// Validation schema (runtime + types from one source)
const CreateUser = z.object({
  name:  z.string().min(1).max(100),
  email: z.string().email(),
});

// A route = method + path + handler chain
app.post('/users',
  requireAuth,                          // middleware
  async (req, res, next) => {           // handler
    try {
      const data = CreateUser.parse(req.body);   // throws if invalid
      const user = await db.user.create({ data });
      res.status(201).json(user);
    } catch (err) { next(err); }        // forward to error handler
  }
);

// Centralised error handler (4 args = error middleware)
app.use((err, req, res, next) => {
  console.error(err);
  res.status(err.status ?? 500).json({ error: err.message });
});

app.listen(3000);

Pros: tiny API surface, easy to learn, total control. Cons: on a large team, every project ends up structured differently, and forgetting next(err) silently breaks things.

NestJS — decorators, DI, modules. Angular-flavoured. 
// users.controller.ts — declares the HTTP surface
import { Body, Controller, Post, UseGuards } from '@nestjs/common';
import { AuthGuard } from '../auth/auth.guard';
import { UsersService } from './users.service';
import { CreateUserDto } from './dto/create-user.dto';

@Controller('users')               // → mounts at /users
export class UsersController {
  constructor(private users: UsersService) {} // ← injected, not imported

  @Post()                          // → POST /users
  @UseGuards(AuthGuard)            // → middleware-equivalent
  create(@Body() dto: CreateUserDto) {
    return this.users.create(dto); // validation handled by ValidationPipe
  }
}

// users.service.ts — business logic, no HTTP awareness
@Injectable()
export class UsersService {
  constructor(private prisma: PrismaService) {}

  create(data: CreateUserDto) {
    return this.prisma.user.create({ data });
  }
}

// create-user.dto.ts — DTO doubles as validation schema
import { IsEmail, IsString, Length } from 'class-validator';
export class CreateUserDto {
  @IsString() @Length(1, 100) name!: string;
  @IsEmail()                   email!: string;
}

// users.module.ts — wiring
@Module({
  controllers: [UsersController],
  providers:   [UsersService],
})
export class UsersModule {}

Pros: consistent structure across teams, dependency injection makes testing easy, batteries-included (validation, OpenAPI, WebSockets, microservices). Cons: heavy, lots of magic, steep learning curve if you've never seen Angular/Spring patterns.

Fastify — Express-style speed-focused, schema-first 
import Fastify from 'fastify';
const app = Fastify({ logger: true });

// JSON Schema → runtime validation + auto-OpenAPI docs + speed
app.post('/users', {
  schema: {
    body: {
      type: 'object',
      required: ['name', 'email'],
      properties: {
        name:  { type: 'string', minLength: 1, maxLength: 100 },
        email: { type: 'string', format: 'email' },
      },
    },
  },
  preHandler: [requireAuth],
  handler: async (req, reply) => {
    const user = await db.user.create({ data: req.body });
    reply.code(201).send(user);
  },
});

await app.listen({ port: 3000 });

Fastify compiles JSON schemas into hyper-optimised validation functions and routes. Often 2–3× faster than Express. Same minimal philosophy, modern API.

Which to pick?

Solo / startup / prototype: Express or Fastify. Ship fast, refactor later. Large team / long-lived enterprise app: NestJS — the rigid structure is the feature; new hires find their way faster. Performance-critical: Fastify, or move that one hot endpoint to Go/Rust.

2.4 The middleware chain — every request's journey ▼

A request never goes straight from "incoming" to "your handler". It walks through a pipeline of middleware — small functions that can read the request, modify it, short-circuit it, or pass it along.

Order matters!

If you put rate-limiting after auth, an attacker can hammer your auth endpoint forever. If you put body-parsing after the route, req.body is undefined. The chain is cumulative — each middleware adds context the next ones rely on.

2.5 Layered architecture — controller, service, repository ▼

A real backend isn't one big handler function. It's split into layers, each with one responsibility. The same pattern appears in NestJS, Spring (Java), .NET, and Rails — only the syntax differs.

ControllerHTTP-aware. Parses request, validates input, calls service, formats response. Knows nothing about DB.
ServiceBusiness logic. "Charge the card, send confirmation email, update inventory." Doesn't know about HTTP or SQL.
Repository / ORMData access. Translates "find users with active subscription" into SQL. Knows the database, nothing else.
DatabasePostgreSQL, MySQL, MongoDB, Redis. The persistent source of truth.
A real flow — POST /orders walked through every layer 
// 1. CONTROLLER (HTTP layer)
@Post('/orders')
async create(@Body() dto: CreateOrderDto, @CurrentUser() user: User) {
  const order = await this.orders.placeOrder(user.id, dto.items);
  return { id: order.id, total: order.total };  // ← shape for API
}

// 2. SERVICE (business logic — the brains)
async placeOrder(userId: string, items: OrderItem[]) {
  return this.db.$transaction(async (tx) => {
    const stock  = await this.inventory.reserve(items, tx);  // may throw
    const total  = this.pricing.calculate(items);
    const order  = await this.orderRepo.create(
                     { userId, items, total }, tx);
    await this.payments.charge(userId, total);               // may throw
    await this.email.queueConfirmation(order);               // async job
    return order;
  });
}

// 3. REPOSITORY (data access — Prisma here)
async create(data: NewOrder, tx?: PrismaTx) {
  const client = tx ?? this.prisma;
  return client.order.create({
    data,
    include: { items: true, user: true },
  });
}

// 4. DATABASE — Prisma generates SQL:
// INSERT INTO orders (user_id, total, ...) VALUES (...);
// INSERT INTO order_items (order_id, ...) VALUES (...);
// (all inside a single transaction so it's atomic)

Why bother with layers?

You can swap the database without touching business logic. You can call placeOrder from a CLI, a cron job, or a GraphQL resolver — not just HTTP. You can unit-test the service with a mocked repository. The layers also force you to think clearly about what each chunk of code is actually for.

2.6 The database layer — raw SQL, query builders, ORMs ▼

Three ways to talk to your database, from lowest- to highest-level abstraction. They're not mutually exclusive — most teams use an ORM for 95% of queries and drop to SQL for the gnarly 5%.

Raw SQL (pg driver) 
const { rows } = await pool.query(
  `SELECT u.id, u.name,
          COUNT(o.id) AS orders
   FROM users u
   LEFT JOIN orders o
     ON o.user_id = u.id
   WHERE u.active = $1
   GROUP BY u.id
   LIMIT $2`,
  [true, 50]
);

Pro: max performance & flexibility. Con: no type safety, easy to introduce SQL injection if you forget parameters.

Prisma ORM (typed)

2.7 API styles — REST vs GraphQL vs tRPC vs WebSockets ▼

The API is the contract between frontend and backend. The shape of that contract is one of the most consequential decisions in a project.

RESTResources as URLs (GET /users/42, POST /orders). Simple, cacheable, universal. The default for 90% of public APIs.
GraphQLSingle endpoint. Client picks fields, joins relations in one query. Solves over/under-fetching. Heavier infra, harder to cache.
tRPCTS-only. Call your server functions from the client like local functions, with full types. Zero schema duplication. Only when both ends are TS.
WebSockets / SSEPersistent connection, server can push to client. Chat, live dashboards, multiplayer, notifications.
gRPCBinary protocol over HTTP/2. Schema in .proto files. Common for service-to-service inside a backend, not browser-facing.

Run TS in production: compile, or runtime?

Compile first (recommended for prod): tsc outputs dist/, you run node dist/server.js. Fast startup, no surprises. Run TS directly (dev): tsx, ts-node, or Bun/Deno transpile on-the-fly. Slower start, more memory, but no build step. Most teams use tsx in dev and tsc in CI.

3.1 Step 1: Transpile — convert syntax the runtime doesn't know ▼

What happens to types?

TypeScript types (interface, type, generics like User[]) are completely erased. They exist only at dev time for error checking. Zero bytes of type info reach the runtime. That's why TS is a "compile-time" type system — you get no runtime guarantees from it. For runtime checks at API boundaries, pair TS with Zod, Valibot, or class-validator.

4.5 CDN deep dive — why latency drops 10× ▼ 
Without CDN:
  User in Tokyo  → request to Virginia → 200ms latency
  User in London → request to Virginia → 100ms latency

With CDN (CloudFront, Cloudflare, Akamai):
  User in Tokyo  → Tokyo edge server   → 15ms ⚡
  User in London → London edge server  → 10ms ⚡

  CDN edge servers ask origin ONCE, then cache.
  Every subsequent user gets served from nearest edge.
What's cacheable?
  ✅ Static assets (JS, CSS, images, fonts) — yes, for a year
  ✅ Public API responses — yes, with short TTL
  ⚠️  HTML — usually short TTL or no-cache
  ❌ Authenticated/personal pages — vary by user, hard to cache
  ❌ Mutating requests (POST, PUT, DELETE) — never cached

Cache hit rate is the metric that matters.
  90% hit rate = 10× less load on your origin servers.

Cache invalidation:
  • Hashed filenames → never invalidate, just deploy new ones
  • TTL expiry      → set max-age, let it expire naturally
  • Manual purge    → API call to CDN ("flush /api/users")

    async placeOrder(userId, items) {
      return db.$transaction(async (tx) => {
        const stock = await inventory.reserve(items, tx);  // may throw
        const total = pricing.calculate(items);
        const order = await orderRepo.create(
                        { userId, items, total, status: 'PENDING' }, tx);
        const charge = await payments.charge(userId, total);
        await orderRepo.update(order.id,
                        { status: 'PAID', chargeId: charge.id }, tx);
        await emailQueue.enqueue('order.confirmation', { order });
        return order;
      });
    }

Common HTTP status codes you'll deal with 
1xx  Informational    (rare)
2xx  Success
  200 OK              standard success
  201 Created         resource created (POST)
  204 No Content      success, body intentionally empty (DELETE)

3xx  Redirection
  301 Moved Permanently   permanent redirect (cached!)
  302 Found               temporary redirect
  304 Not Modified        client's cache is still good (saves bandwidth)

4xx  CLIENT errors (the request is wrong)
  400 Bad Request         malformed body / validation failed
  401 Unauthorized        no/invalid credentials
  403 Forbidden           authenticated but not allowed
  404 Not Found           resource doesn't exist
  409 Conflict            e.g. email already registered
  422 Unprocessable       semantic validation failure
  429 Too Many Requests   rate limited

5xx  SERVER errors (the server is wrong)
  500 Internal Server Error   unexpected exception
  502 Bad Gateway             load balancer can't reach backend
  503 Service Unavailable     down for maintenance / overloaded
  504 Gateway Timeout         backend took too long

4xx is the client's fault, 5xx is yours

A 4xx means "this request was invalid, fix it and try again" — log it but don't page anyone. A 5xx means "the server is broken" — alerts, dashboards, on-call. Returning a 500 when you should return a 400 (e.g. validation errors) is a common bug that blows up your alerting.

6.5 Cascade of requests — HTML triggers more fetches ▼ 
Browser receives index.html, starts parsing.
As it finds resource references, it fires MORE requests:

Request #1: GET /                        → index.html (1KB)
  └─ Parser finds <link> and <script>:

Request #2: GET /assets/index-pQ8s.css   → styles (8KB)
Request #3: GET /assets/index-Da3x.js    → app bundle (45KB)
Request #4: GET /assets/vendor-Bk9m.js   → vendor chunk (140KB)
  └─ JS executes and triggers:

Request #5: GET /assets/logo-kM2x.svg    → logo (3KB)
Request #6: GET /api/users               → JSON data (12KB) ← from backend!

With HTTP/2, requests 2-4 happen IN PARALLEL (multiplexed)
over the SAME TCP connection. No extra handshakes!

Total: ~209KB compressed → ~850KB decompressed in memory

Caching headers control repeat visits

Hashed assets (app-Da3x.js) get max-age=31536000, immutable (1 year!) because the hash changes when content changes. index.html gets no-cache so the browser always checks for a new version. API responses usually get no-store or short TTLs. Second visits are much faster.

Browser rendering engine

The rendering engine (Blink in Chrome, Gecko in Firefox, WebKit in Safari) takes raw bytes and turns them into interactive pixels. This is the most complex part of the chain.

7.1 Parse HTML → Build the DOM tree ▼ 
HTML bytes arrive:  3C 21 44 4F 43 54 59 50 45 ...
                          ↓ decode (UTF-8)
Characters:         <!DOCTYPE html><html><head>...
                          ↓ tokenize
Tokens:             [DOCTYPE] [StartTag:html] [StartTag:head] ...
                          ↓ tree construction
DOM Tree:
  Document
  ├── DOCTYPE: html
  └── html
      ├── head
      │   ├── meta[charset="UTF-8"]
      │   ├── title → "User Dashboard"
      │   ├── link[rel="stylesheet" href="index-pQ8s.css"]
      │   │    ↑ Triggers CSS download! (render-blocking)
      │   └── script[type="module" src="index-Da3x.js"]
      │        ↑ Triggers JS download! (deferred with type=module)
      └── body
          └── div#root
               ↑ Empty — React fills this later

Parse-blocking behavior

A <script> without async/defer STOPS HTML parsing until it downloads and executes (because old JS could call document.write). Modern frameworks use type="module" (deferred by default). CSS is "render-blocking" — browser won't paint until CSS is parsed, but HTML parsing continues underneath.

7.2 Parse CSS → Build CSSOM tree ▼ 
CSSOM (CSS Object Model):
  StyleSheet
  ├── Rule: .container
  │   ├── max-width: 800px
  │   ├── margin: 0 auto
  │   └── padding: 40px 24px
  ├── Rule: .container .title
  │   ├── font-size: 2rem        → computed: 32px
  │   ├── color: #378ADD
  │   └── margin-bottom: 24px
  └── Rule: .userCard
      ├── background: white       → computed: rgb(255,255,255)
      ├── border-radius: 12px
      └── box-shadow: 0 2px 8px rgba(0,0,0,0.1)

During CSSOM construction:
  1. Parse all CSS rules
  2. Resolve cascade (specificity: inline > #id > .class > element)
  3. Resolve inheritance (color inherits; margin doesn't)
  4. Compute final values (rem → px, % → px, colors → rgba)
7.3 DOM + CSSOM → Render tree → Layout → Paint ▼ 
RENDER TREE (only visible nodes — <head>, display:none excluded):

  RenderView (viewport: 1440×900)
  └── RenderBlock (body) [font: 16px, color: #1a1a1a]
      └── RenderBlock (div.container) [max-width: 800px, margin: 0 auto]
          ├── RenderBlock (h1.title) [color: #378ADD, font-size: 32px]
          │   └── RenderText "User Dashboard"

8.2 Event loop in action — real code walkthrough ▼
What order do logs appear? 
console.log('1: Script start');

setTimeout(() => {
  console.log('2: setTimeout');
}, 0);

Promise.resolve()
  .then(() => console.log('3: Promise 1'))
  .then(() => console.log('4: Promise 2'));

fetch('/api/users')
  .then(res => res.json())
  .then(data => console.log('5: Data received'));

console.log('6: Script end');
Execution order — step by step 
TICK 1 — Synchronous execution:
  → console.log('1: Script start')     // ✅ OUTPUT
  → setTimeout(cb, 0)                  // cb → Timer API
  → Promise.resolve().then(cb1)        // cb1 → microtask queue
  → fetch('/api/users')                // → Network API
  → console.log('6: Script end')       // ✅ OUTPUT
  Call Stack: [] (empty)

TICK 2 — Drain microtask queue:
  → console.log('3: Promise 1')        // ✅ OUTPUT
  → .then chains → console.log('4: Promise 2')  // ✅ OUTPUT

TICK 3 — Process one macrotask:
  → console.log('2: setTimeout')       // ✅ OUTPUT

TICK 4+ — fetch completes eventually:
  → console.log('5: Data received')    // ✅ OUTPUT

FINAL ORDER:
  "1: Script start"    ← synchronous
  "6: Script end"      ← synchronous
  "3: Promise 1"       ← microtask
  "4: Promise 2"       ← microtask
  "2: setTimeout"      ← macrotask (even with 0ms delay!)
  "5: Data received"   ← microtask (when network resolves)

setTimeout(fn, 0) is NOT instant

Even with 0ms delay, setTimeout goes to the macrotask queue. It must wait for: (1) synchronous code to finish, (2) ALL microtasks to drain, (3) its turn. Promise callbacks always run before setTimeout, even if setTimeout was registered first.

8.3 Same loop, different shape — Node vs browser ▼

Node's event loop has more "phases" than the browser's, because servers care about I/O patterns the browser doesn't. The principle is the same; the order of certain callbacks differs.

Node.js event loop phases (simplified, runs each tick):

  ┌─────────────┐
  │ timers      │  setTimeout/setInterval callbacks due
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ pending I/O │  TCP errors etc.
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ poll        │  Most I/O completes here (file, net).
  │             │  Will block here if no other work.
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ check       │  setImmediate() callbacks
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ close       │  socket.on('close'), etc.
  └──────┬──────┘
         │
         └─────── back to top
                   ↑
        Microtasks drain BETWEEN every phase.
        process.nextTick() callbacks drain even before microtasks.

CPU-bound work blocks everything

A 200ms image-resize loop on the main thread freezes every in-flight HTTP request on a Node server, and every animation in a browser. For heavy CPU: in the browser use a Web Worker; in Node use worker_threads or a separate microservice in Go/Rust.

User sees & interacts

The page is visible and interactive. Every action triggers: Event → JS handler → State change → Re-render → New pixels. This loop runs at up to 60fps.

9.1 Core Web Vitals — measuring user experience ▼
FCP — First Contentful Paint (~1.2s)First text or image rendered. No longer blank.
LCP — Largest Contentful Paint (~2.0s)Biggest element (hero image, main heading) finishes rendering. Google's main UX metric.
INP — Interaction to Next Paint (~100ms)Time between user input and next visual update. Replaced FID in 2024.
TTI — Time to Interactive (~3.5s)Fully interactive. All JS loaded, handlers attached.
CLS — Cumulative Layout Shift (~0.05)Unexpected layout movement. Images without dimensions cause this.

These metrics affect SEO

Google uses Core Web Vitals as a ranking signal. Two sites with equal content quality — the faster one wins. Measure with Lighthouse, PageSpeed Insights, or real-user monitoring (RUM) via the web-vitals npm package shipping data to your analytics backend.

9.2 The re-render cycle — what happens on every click ▼ 
User clicks "Delete User". Every step:

1. BROWSER EVENT
   └── Click detected → Event object created
   └── Dispatched through DOM (capture → target → bubble)

2. JS HANDLER
   └── React's delegated listener catches it
   └── handleDelete(userId) runs
   └── setState({ users: users.filter(u => u.id !== userId) })

3. REACT RECONCILIATION (Virtual DOM)
   └── New virtual DOM built with updated state
   └── Diff vs previous tree
   └── Finds: "remove 3rd <li>"
   └── Batches DOM mutations

4. DOM UPDATE
   └── removeChild(liElement)  ← only this one operation!
   └── NOT re-rendering entire page

5. BROWSER RE-RENDER (partial)
   └── Style recalc → affected elements only
   └── Layout       → list items reposition
   └── Paint        → repaint affected area
   └── Composite    → GPU recomposite

6. PIXELS UPDATED (~8-16ms total)
