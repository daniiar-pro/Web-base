1. Rendering Strategies (Complete)

The rendering strategy determines where, when, and how HTML is generated. This single decision has cascading effects on SEO, time-to-interactive, infrastructure cost, and developer experience. Below is every major rendering strategy used in modern frontend development.

1.1 Client-Side Rendering (CSR)

The server sends a minimal HTML shell and a JavaScript bundle. The browser downloads, parses, and executes JS to build the entire DOM.

  Browser                        Server / CDN
  ───────                        ────────────
     │  GET /app  ──────────────────►│
     │◄──── index.html (empty shell) │
     │                               │
     │  GET /bundle.js ─────────────►│
     │◄──── JavaScript bundle (~MB)  │
     │                               │
     │  [Parse + Execute JS]         │
     │  [Build Virtual DOM]          │
     │  [Mount to DOM → Paint]       │
     │                               │
     │  GET /api/data ──────────────►│  API Server
     │◄──── JSON response            │
     │                               │
     │  [Re-render with data]        │
     ▼  ✅ Interactive               │
CSR Flow — Browser does all rendering work after downloading JS
Tech Stacks: React + Vite, Vue + Vite, Angular CLI, Svelte (SPA mode), SolidJS + Vite. Typically deployed to static CDNs (Netlify, Vercel static, S3 + CloudFront).
Real-World Usage: Single-page apps behind authentication — dashboards (Grafana), admin panels, internal tools (Retool), Figma, Notion. SEO is irrelevant because pages aren't publicly indexed.
Advantages
  • Rich, app-like interactions
  • Cheap hosting (static CDN)
  • No server runtime needed
  • Smooth page transitions (no full reload)
  • Simple deployment pipeline
Disadvantages
  • Blank page until JS loads (poor FCP)
  • Large bundle = slow TTI on mobile
  • Bad SEO (crawlers see empty HTML)
  • Loading spinners everywhere
  • All logic exposed to client

1.2 Server-Side Rendering (SSR)

The server executes the component tree on every request, producing fully-formed HTML. The browser can display content immediately, then "hydrates" it with interactivity once JS loads.

  Browser                       Server (Node.js / Edge)
  ───────                       ───────────────────────
     │  GET /products ──────────────►│
     │                               │ [Fetch data from DB/API]
     │                               │ [Render components → HTML]
     │◄──── Full HTML + inlined data │
     │                               │
     │  [Paint immediately ✅ FCP]   │
     │                               │
     │  GET /bundle.js ─────────────►│
     │◄──── JS bundle (smaller)      │
     │                               │
     │  [Hydrate: attach events]     │
     ▼  ✅ Interactive (TTI)         │
SSR Flow — Server renders HTML per-request; browser hydrates
Tech Stacks: Next.js (React), Nuxt 3 (Vue), SvelteKit (Svelte), Remix (React), Angular Universal (Angular), Solid Start (SolidJS). Deployed on Node.js servers, serverless functions, or edge runtimes.
Real-World Usage: E-commerce product pages (Shopify Hydrogen), news sites, marketing pages — anywhere SEO and fast first-paint matter.

Hydration Explained

Hydration is the process where client-side JS walks the server-rendered DOM, attaches event listeners, and makes it interactive — without rebuilding it from scratch.

  Server HTML (static)              Hydrated DOM (interactive)
  ┌───────────────────┐             ┌───────────────────┐
  │ <button>Buy Now   │  ──hydrate──►  │ <button onClick={…}>Buy Now │
  │ <input value="1"> │             │ <input onChange={…} value="1"> │
  │ <div>$29.99</div> │             │ <div>$29.99</div> (reactive)  │
  └───────────────────┘             └───────────────────┘
  
  ⚠️ "Hydration Mismatch" = server HTML ≠ client render → React throws warning
  ⚠️ "Uncanny Valley" = page looks ready but isn't clickable yet (JS still loading)
Hydration attaches JS behavior to existing server-rendered markup

1.3 Static Site Generation (SSG)

Pages are pre-rendered at build time into static HTML files. Ideal when content doesn't change per-request.

  Build Time                          Runtime (CDN)
  ──────────                          ──────────────
  [Read Markdown / CMS data]          Browser
  [Render all pages → .html]              │  GET /blog/my-post ───►│
  [Output: /out/blog/my-post.html]        │◄──── pre-built HTML     │
  [Deploy to CDN edge]                    │  ✅ Instant paint       │
                                          │  [Hydrate if needed]    │
                                          ▼                         │
  ⏱ Build: 30s–10min (depends on pages)
  ⚡ Runtime: ~20ms TTFB from edge
SSG — Everything rendered ahead of time, served from CDN edge
Tech Stacks: Astro, Hugo (Go), Gatsby (React), Eleventy (JS), Jekyll (Ruby), VitePress (Vue), Docusaurus (React), Next.js (static export), Nuxt (generate mode).
Real-World Usage: Documentation sites (Docusaurus, VitePress), blogs (Astro, Hugo), marketing landing pages, portfolios.

1.4 Incremental Static Regeneration (ISR)

A hybrid of SSG and SSR. Pages are statically generated but can be regenerated in the background after a configurable time window — giving you static speed with dynamic freshness.

  First visit (cache cold)              Subsequent visit (within revalidate window)
  ────────────────────────              ────────────────────────────────────────────
  Browser──►CDN: MISS                   Browser──►CDN: HIT
            │                                      │
            ├──►Server: render page                 ├──►Return cached HTML (fast!)
            │   [fetch data, build HTML]            │
            │◄──HTML                                │  (if stale, trigger background rebuild)
            │                                       │
            ├──►Cache HTML at edge                  │  Background:
            │                                       │  Server re-renders → updates cache
  Browser◄──HTML                        Browser◄──HTML
  
  Next.js: revalidate: 60   // re-generate every 60 seconds
  Nuxt:    isr: true         // similar ISR support
ISR — Stale-while-revalidate for entire pages
Tech Stacks: Next.js (primary), Nuxt 3 (routeRules with ISR), Gatsby (DSG — Deferred Static Generation). Requires a runtime for background regeneration (serverless or edge).

1.5 Streaming SSR

Instead of waiting for the entire page to render, the server streams HTML chunks progressively. The browser paints content as it arrives — eliminating the "wait for the slowest query" bottleneck.

  Traditional SSR                        Streaming SSR
  ───────────────                        ─────────────
  Server: fetch ALL data                 Server: stream as ready
          ↓                                      ↓
  Render ALL components                  Stream <head> + shell immediately
          ↓                                      ↓
  Send complete HTML                     Stream ProductList when DB responds
  (user waits for slowest query)                 ↓
                                         Stream Reviews when API responds
                                         (user sees content progressively)
  
  Timeline comparison:
  Traditional: [────wait 800ms────][──send all──] Total: ~1s
  Streaming:   [send shell 50ms][...data 200ms...][...more 400ms...] Progressive!
  
  User perceives faster load because they see shell + nav + above-fold content first.
Streaming SSR — progressive HTML delivery, no waiting for slowest data source
Tech Stacks: Next.js 13+ (app router), Remix (built-in streaming), SvelteKit, Solid Start, React 18+ (renderToPipeableStream). Requires HTTP chunked transfer encoding support.

1.6 React Server Components (RSC)

A paradigm where components are split into server-only and client-only categories. Server Components run exclusively on the server, never ship JS to the client, and can directly access databases, file systems, and private APIs. Client Components handle interactivity.

  ┌──────────────────────────────────────────────────────┐
  │  React Server Components (RSC) Architecture          │
  │                                                      │
  │  Server Components          Client Components        │
  │  ┌─────────────────┐        ┌─────────────────┐      │
  │  │ async function   │        │ 'use client'    │      │
  │  │ ProductPage() {  │        │                 │      │
  │  │   const data =   │   ──►  │ function AddTo  │      │
  │  │   await db.query │        │ Cart() {        │      │
  │  │   return <div>.. │        │   const [n, set]│      │
  │  │ }                │        │   = useState(0) │      │
  │  └─────────────────┘        └─────────────────┘      │
  │  • Run on server only        • Ship JS to browser    │
  │  • Direct DB/fs access       • useState, useEffect   │
  │  • Zero client bundle size   • Event handlers        │
  │  • Can't use state/effects   • Interactivity         │
  │  • Can use async/await       • Cannot be async       │
  │  • Serialized as payload     • Hydrated normally     │
  └──────────────────────────────────────────────────────┘
  
  Component Boundary:
  ServerComponent → can render → ClientComponent (pass serializable props)
  ClientComponent → CANNOT import → ServerComponent (but can accept as children)
RSC — server/client component split, zero JS for server-only components
Tech Stacks: Next.js 13+ (app router default), Remix (experimental RSC), Gatsby 5 (partial hydration via RSC). The React Compiler (React 19+) auto-optimizes memoization for RSC-based apps.

1.7 Islands Architecture (Partial Hydration)

The page is rendered as mostly static HTML. Only specific interactive components ("islands") are hydrated with JavaScript. The rest of the page requires zero client-side JS. This is the core idea behind Astro.

  Traditional SPA Hydration              Islands Architecture
  ─────────────────────────              ──────────────────────
  ┌──────────────────────────┐           ┌──────────────────────────┐
  │ ████████████████████████ │           │ ░░░░░░░░░░░░░░░░░░░░░░░ │  ← static HTML (0 JS)
  │ ████████████████████████ │           │ ░░░░┌──────────┐░░░░░░░ │
  │ ████████████████████████ │           │ ░░░░│ Island 1 │░░░░░░░ │  ← hydrated (JS loaded)
  │ ████████████████████████ │           │ ░░░░│ (React)  │░░░░░░░ │
  │ ████████████████████████ │           │ ░░░░└──────────┘░░░░░░░ │
  │ ████████████████████████ │           │ ░░░░░░░░░░░░░░░░░░░░░░░ │  ← static HTML (0 JS)
  │ ████████████████████████ │           │ ░░░░░░┌────────┐░░░░░░░ │
  └──────────────────────────┘           │ ░░░░░░│Island 2│░░░░░░░ │  ← hydrated (Svelte!)
  100% of page hydrated                 │ ░░░░░░│(Svelte)│░░░░░░░ │
  Even static content gets JS            │ ░░░░░░└────────┘░░░░░░░ │
                                         │ ░░░░░░░░░░░░░░░░░░░░░░░ │  ← static HTML (0 JS)
                                         └──────────────────────────┘
                                         Only interactive parts get JS
                                         Even mix frameworks per island!
  
  Astro hydration directives:
    <LikeButton client:idle />      ← hydrate when browser is idle
    <Comments client:visible />     ← hydrate when scrolled into view
    <Carousel client:load />        ← hydrate immediately on load
    <Search client:media="(min-width:768px)" />  ← hydrate on media query
Islands Architecture — static sea + interactive islands, multi-framework support
Tech Stacks: Astro (primary), Fresh (Deno + Preact), Marko (auto-detected islands), Elder.js (Svelte). Astro supports React, Vue, Svelte, Solid, Preact, Lit, and Alpine as island frameworks.
Advantages
  • Near-zero JS by default
  • Excellent Core Web Vitals
  • Mix and match frameworks
  • Simple mental model
  • Progressive enhancement built-in
Disadvantages
  • No shared state between islands
  • Communication between islands is hard
  • Less suitable for highly interactive SPAs
  • Limited client-side routing

1.8 Resumability (Zero-Hydration)

Pioneered by Qwik, resumability eliminates hydration entirely. Instead of re-executing the entire component tree on the client, the framework serializes its state (event listeners, component boundaries, reactivity graph) into HTML. The client "resumes" from where the server left off — loading JS only when a user interacts with a specific component.

  Hydration (React, Vue, Svelte)          Resumability (Qwik)
  ──────────────────────────────          ──────────────────────
  Server renders HTML                     Server renders HTML
           ↓                                       ↓
  Client downloads ALL component JS       Client downloads ZERO component JS
           ↓                                       ↓
  Execute components (reconcile)          Page is INSTANTLY interactive
           ↓                                       ↓
  Attach event listeners                  User clicks a button
           ↓                                       ↓
  ✅ Now interactive                      Download ONLY that handler (~2KB)
                                                   ↓
  Cost: O(n) — scales with page size      Resume execution from serialized state
                                                   ↓
                                          ✅ Handler executes
                                          
                                          Cost: O(1) — constant regardless of page size
  
  Serialized in HTML:
  ┌────────────────────────────────────────────────┐
  │ <button on:click="./chunk-abc.js#handler_42">  │
  │   Buy Now                                      │
  │ </button>                                      │
  │ <script type="qwik/json">                      │
  │   { "refs": {...}, "objs": [...], "subs": [..]}│
  │ </script>                                      │
  └────────────────────────────────────────────────┘
  Event listeners are strings (not JS), resolved lazily on interaction.
Resumability — no hydration, O(1) startup, lazy-load JS on interaction
Tech Stacks: Qwik + Qwik City (primary). Marko 6 (upcoming, similar resumable approach). Both use fine-grained reactivity + serialization.

1.9 Edge-Side Rendering (ESR)

Instead of rendering on a centralized server or in the browser, HTML is generated at CDN edge locations physically close to the user. This combines SSR's dynamic freshness with CDN's low-latency delivery, achieving TTFB of 10–50ms globally.

  Traditional SSR                        Edge-Side Rendering
  ───────────────                        ──────────────────────
  User (Tokyo) ──────────────►           User (Tokyo) ───►
       │                                      │
       │  1500ms round trip                   │  ~15ms to nearest edge
       │  to US-East server                   │
       ▼                                      ▼
  ┌──────────────┐                       ┌─────────────────┐
  │  Origin       │                       │  Edge (Tokyo)    │
  │  US-East      │                       │  V8 Isolate      │
  │  Node.js      │                       │  Runs SSR logic  │
  │  Full runtime │                       │  ~5ms cold start │
  └──────────────┘                       └─────────────────┘
  
  Edge network topology:
  ┌── Edge (Tokyo) ────► User in Japan        (~15ms TTFB)
  ├── Edge (Mumbai) ───► User in India        (~20ms TTFB)
  ├── Edge (Frankfurt) ► User in Europe       (~12ms TTFB)
  ├── Edge (São Paulo) ► User in Brazil       (~18ms TTFB)
  └── Edge (Virginia) ─► User in US East      (~8ms TTFB)
  
  Each edge node runs lightweight JS runtime (V8 isolates, not full Node.js).
  Limitations: No native Node.js APIs (fs, child_process), limited execution time.
Edge-Side Rendering — SSR at 330+ global locations, sub-50ms TTFB
Tech Stacks: Cloudflare Workers/Pages (Hono, Remix, SvelteKit), Vercel Edge Functions (Next.js), Deno Deploy (Fresh), Netlify Edge Functions, AWS Lambda@Edge. Runtime: V8 Isolates (not full Node.js).

1.10 Progressive Hydration / Selective Hydration

Instead of hydrating the entire page at once, components are hydrated progressively — critical above-the-fold content first, then less important components during idle time or when they become visible. React 18's Suspense boundaries enable selective hydration automatically.

  Full Hydration                         Progressive / Selective Hydration
  ────────────────                       ────────────────────────────────────
  [Load page] → hydrate EVERYTHING       [Load page] → hydrate Header (critical)
  at once, blocking main thread                      → hydrate Hero (above fold)
                                                     → idle... hydrate Sidebar
  ┌─────────────────────┐                            → user scrolls...
  │ ████ All at once ███ │                            → hydrate Comments (visible)
  │ █████████████████████ │                            → hydrate Footer (visible)
  │ (blocks for 500ms+)  │
  └─────────────────────┘               ┌────────┐ ┌──┐ ┌───┐    ┌────┐
                                         │ Header │ │..│ │...│    │Cmts│
                                         └────────┘ └──┘ └───┘    └────┘
                                         50ms       idle  idle     on-scroll
  
  React 18 Selective Hydration:
  ┌─────────────────────────────────────────────────────┐
  │ <Suspense fallback={<Skeleton/>}>                   │
  │   <Comments />   ← hydrated when React is free     │
  │ </Suspense>                                         │
  │                                                     │
  │ User clicks Comments before hydration?              │
  │ → React PRIORITIZES hydrating Comments immediately! │
  └─────────────────────────────────────────────────────┘
Progressive hydration — prioritize critical UI, defer the rest
Tech Stacks: React 18+ (Suspense + lazy), Next.js 13+ (automatic with app router), Nuxt 3, Google's Angular (defer blocks in Angular 17+).

1.11 Hybrid / Per-Route Rendering

Modern meta-frameworks allow choosing a different rendering strategy for each route. A single app can use SSG for the blog, SSR for product pages, CSR for the dashboard, and ISR for the catalog — all in one codebase.

  Same application, different strategies per route:
  
  ┌──────────────────────────────────────────────────────────┐
  │  Next.js App                                             │
  │                                                          │
  │  /                    → SSG    (static marketing page)   │
  │  /blog/[slug]         → SSG    (markdown at build time)  │
  │  /products            → ISR    (revalidate: 3600)        │
  │  /products/[id]       → SSR    (per-request, real-time)  │
  │  /dashboard           → CSR    (behind auth, no SEO)     │
  │  /api/checkout        → Edge   (edge function, low TTFB) │
  │  /search              → Streaming SSR (progressive)      │
  └──────────────────────────────────────────────────────────┘
  
  Nuxt 3 routeRules equivalent:
  ┌─────────────────────────────────────────┐
  │ routeRules: {                           │
  │   '/': { prerender: true },             │
  │   '/blog/**': { prerender: true },      │
  │   '/products/**': { isr: 3600 },        │
  │   '/dashboard/**': { ssr: false },      │
  │   '/api/**': { cors: true }             │
  │ }                                       │
  └─────────────────────────────────────────┘
Hybrid rendering — pick the best strategy per route within one app
Tech Stacks: Next.js (app router), Nuxt 3 (routeRules), SvelteKit (+page.ts options), Remix (loader-based), Astro (hybrid mode), Solid Start.

2. Structural / Architectural Patterns

These patterns define how you organize code at the macro level — folder structure, module boundaries, dependency direction, and how features compose together.

2.1 Feature-Based Architecture (Feature Slices)

Instead of grouping by technical role (components/, services/, hooks/), group by business feature. Each feature folder is a self-contained vertical slice.

  ❌ Technical Grouping                ✅ Feature-Based Grouping
  ─────────────────────                ───────────────────────────
  src/                                 src/
  ├── components/                      ├── features/
  │   ├── ProductCard.tsx              │   ├── products/
  │   ├── CartItem.tsx                 │   │   ├── components/
  │   ├── UserAvatar.tsx               │   │   │   ├── ProductCard.tsx
  │   └── CheckoutForm.tsx             │   │   │   └── ProductGrid.tsx
  ├── hooks/                           │   │   ├── hooks/
  │   ├── useProducts.ts               │   │   │   └── useProducts.ts
  │   ├── useCart.ts                    │   │   ├── api/
  │   └── useAuth.ts                   │   │   │   └── productApi.ts
  ├── services/                        │   │   ├── store/
  │   ├── productApi.ts                │   │   │   └── productSlice.ts
  │   ├── cartApi.ts                   │   │   ├── types.ts
  │   └── authApi.ts                   │   │   └── index.ts  ← public API
  └── store/                           │   ├── cart/
      ├── productSlice.ts              │   │   ├── components/
      ├── cartSlice.ts                 │   │   ├── hooks/
      └── authSlice.ts                 │   │   └── ...
                                       │   └── auth/
  Problem: Adding a feature                  │   └── ...
  touches 5+ folders.                  │
  Files with no relation               ├── shared/        ← cross-feature code
  sit next to each other.              │   ├── ui/         (Button, Modal, etc.)
                                       │   ├── lib/        (utils, helpers)
                                       │   └── hooks/      (useDebounce, etc.)
                                       └── app/           ← routing, providers
Feature-based architecture — each feature is a cohesive, self-contained module
Key Rule: Features can import from shared/ but NEVER from other features directly. Cross-feature communication goes through the global store, events, or the app layer.

2.2 Feature-Sliced Design (FSD)

A formalized methodology that extends feature-based architecture with strict layers and public API boundaries. Increasingly popular in 2025-2026 for large-scale applications.

  Feature-Sliced Design Layers (top → bottom, imports only go DOWN)
  ──────────────────────────────────────────────────────────────────
  
  ┌─────────┐  app/         ← Providers, global styles, routing
  │  app    │               One per project. Composes everything.
  ├─────────┤
  │  pages  │  pages/       ← Full page compositions
  ├─────────┤               Compose widgets + features into pages.
  │ widgets │  widgets/     ← Composite UI blocks (Header, Sidebar)
  ├─────────┤               Compose features + entities. Reusable.
  │features │  features/    ← Business scenarios (auth, like, comment)
  ├─────────┤               User-facing actions. Can use entities.
  │entities │  entities/    ← Business objects (User, Product, Order)
  ├─────────┤               Data models, CRUD components for them.
  │ shared  │  shared/      ← Reusable non-business code
  └─────────┘               UI kit, libs, API client, types.
  
  Each slice has:  ui/ | model/ | api/ | lib/ | config/ | index.ts (public API)
  
  ⚠️ Import rule: Each layer can ONLY import from layers BELOW it.
     features/ can import from entities/ and shared/ but NOT from widgets/ or pages/.
Feature-Sliced Design — strict layer hierarchy with public API boundaries

2.3 Layered Architecture

Organize code into horizontal layers with strict dependency direction: each layer can only depend on layers below it.

  ┌─────────────────────────────────────────────────┐
  │              Presentation Layer                  │
  │  (Pages, Components, Layouts, Styling)           │
  │  React components, CSS modules, view logic       │
  └────────────────────┬────────────────────────────┘
                       │ depends on ▼
  ┌─────────────────────────────────────────────────┐
  │              Application Layer                   │
  │  (Use Cases, Orchestration, State Management)    │
  │  Custom hooks, stores, context, business flows   │
  └────────────────────┬────────────────────────────┘
                       │ depends on ▼
  ┌─────────────────────────────────────────────────┐
  │                Domain Layer                      │
  │  (Entities, Business Rules, Validation)          │
  │  Pure TypeScript — NO framework dependencies     │
  └────────────────────┬────────────────────────────┘
                       │ depends on ▼

3.3 DRY, KISS, YAGNI

DRY — Don't Repeat Yourself

Extract shared logic into hooks, utilities, or components. But beware of premature abstraction — the "Rule of Three" says wait until you've duplicated something three times before extracting.

KISS — Keep It Simple

Choose the simplest solution that works. Don't reach for Redux when useState is enough. Don't add GraphQL when REST serves your needs. Complexity should be justified by requirements, not anticipated.

YAGNI — You Aren't Gonna Need It

Don't build abstractions for hypothetical future requirements. If you only have one theme, you don't need a theme engine. Build for now, refactor when the need is real.

11. Accessibility Architecture

Accessibility (a11y) is no longer optional — regulations like the European Accessibility Act (2025) require WCAG 2.1 AA compliance. Good architecture makes accessibility the default, not an afterthought.

  Accessibility Architecture Layers
  ──────────────────────────────────────────────────────────
  
  ┌─────────────────────────────────────────────────────┐
  │  1. SEMANTIC HTML (Foundation)                      │
  │     <nav>, <main>, <article>, <button>, <dialog>  │
  │     Use native elements before ARIA roles           │
  │     "No ARIA is better than bad ARIA"               │
  └──────────────────────┬──────────────────────────────┘
                         ▼
  ┌─────────────────────────────────────────────────────┐
  │  2. HEADLESS UI LIBRARIES (Interaction patterns)    │
  │     Radix UI, React Aria (Adobe), Headless UI       │
  │     Handle focus trapping, arrow keys, screen reader │
  │     announcements — so you don't have to.           │
  └──────────────────────┬──────────────────────────────┘
                         ▼
  ┌─────────────────────────────────────────────────────┐
  │  3. DESIGN SYSTEM TOKENS (Visual accessibility)     │
  │     Contrast ratios ≥ 4.5:1 (AA) baked into tokens │
  │     Focus-visible styles on all interactive elements│
  │     Reduced motion media queries in animations      │
  └──────────────────────┬──────────────────────────────┘
                         ▼
  ┌─────────────────────────────────────────────────────┐
  │  4. AUTOMATED TESTING (Continuous validation)       │
  │     axe-core in CI, Lighthouse a11y audits          │
  │     Testing Library queries (getByRole, getByLabel) │
  │     Playwright a11y assertions                      │
  └─────────────────────────────────────────────────────┘
  
  Key patterns:
  • Keyboard navigation: all interactive elements reachable via Tab
  • Focus management: trap focus in modals, restore on close
  • Live regions: aria-live for dynamic content updates
  • Skip links: bypass navigation for keyboard users
  • Responsive text: rem/em units, never px for font-size
Accessibility baked into architecture — not bolted on after launch

13. Frontend Security Patterns

Security is an architectural concern, not a last-minute checkbox. Modern frontend apps face unique attack vectors that must be addressed structurally.

  Frontend Attack Vectors & Defenses
  ──────────────────────────────────────────────────────────────────
  
  ┌──────────────────────┬──────────────────────────────────────────┐

  │  Scripting)          │  • Sanitize user input (DOMPurify)       │
  │                      │  • React auto-escapes JSX by default     │
  │                      │  • Avoid dangerouslySetInnerHTML         │
  ├──────────────────────┼──────────────────────────────────────────┤

  │                      │  • Use POST for state-changing actions   │
  ├──────────────────────┼──────────────────────────────────────────┤
  │  Supply chain attack │  • Lock dependencies (lockfile)          │
  │  (malicious npm pkg) │  • Audit: npm audit, Snyk, Socket.dev   │
